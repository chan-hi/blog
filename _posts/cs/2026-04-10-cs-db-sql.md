## **1\. 핵심 개념 정리**

### **① SQL의 분류와 특징 (DDL, DML, TCL)**

* **정의**: 데이터베이스를 조작하는 언어. 목적에 따라 구조를 정의하는 DDL, 데이터를 조작하는 DML, 트랜잭션을 제어하는 TCL 등으로 나뉨.  
* **왜 존재하는가**: 수많은 데이터를 파일 텍스트로 일일이 수정할 수 없으니, 표준화된 언어(SQL)를 통해 DBMS 엔진에 "무엇을 어떻게 해달라"고 요청하기 위함이야.  
* **어떻게 동작하는가**:  
  * **DDL (CREATE, ALTER, DROP, TRUNCATE)**: 실행 즉시 디스크에 구조적 변경을 일으키며 \*\*자동으로 커밋(Auto-commit)\*\*돼. 롤백이 불가능해.  
  * **DML (INSERT, UPDATE, DELETE, SELECT)**: 디스크에 바로 쓰기 전, 메모리(버퍼 풀) 단위에서 데이터를 조작해. TCL과 결합해 확정(Commit)하거나 취소(Rollback)할 수 있어.  
* **개념 간 관계**: DDL로 만든 뼈대(테이블) 위에 DML로 살(데이터)을 붙이고, TCL로 이 작업의 안정성(원자성)을 보장하는 유기적인 구조야.

### **② SELECT 문의 실제 실행 순서**

* **정의**: 개발자가 쿼리를 작성하는 순서(문법)와, DBMS 엔진이 실제로 데이터를 찾아오는 순서(실행 순서)는 완전히 달라.  
* **작성 순서**: SELECT ➔ FROM ➔ WHERE ➔ GROUP BY ➔ HAVING ➔ ORDER BY ➔ LIMIT  
* **실제 실행 순서**: **FROM** (어느 테이블에서) ➔ **WHERE** (어떤 조건의 레코드만 걸러서) ➔ **GROUP BY** (어떻게 묶은 다음) ➔ **HAVING** (그룹핑된 결과에서 다시 걸러내고) ➔ **SELECT** (최종적으로 어떤 컬럼만 뽑아서) ➔ **ORDER BY** (정렬한 뒤) ➔ **LIMIT** (개수를 제한할까?)

## **2\. 장애·트러블슈팅 연결**

### **🚨 트러블슈팅 시나리오 1: DDL에 의한 서비스 전체 장애 (Table Lock)**

* **장애 상황**: 평일 낮, "유저 테이블에 컬럼 하나만 급하게 추가할게요\!" 하고 백엔드 개발자가 ALTER TABLE users ADD COLUMN...을 실행한 순간, 사이트 전체 접속이 마비되고 DB CPU가 100%를 침.  
* **주니어의 함정**: "그냥 컬럼 하나 껍데기 추가하는 건데 왜 터지지? 쿼리가 무거운 것도 아닌데."  
* **원리를 알면 보이는 것**: DDL(특히 구버전 MySQL/MariaDB에서의 ALTER)은 작업이 끝날 때까지 테이블 전체에 **메타데이터 락(Metadata Lock)이나 테이블 락**을 걸어버려. 운영 중인 수천만 건의 테이블에 락이 걸리니, 다른 조회/삽입 요청들이 전부 Lock wait timeout으로 대기 상태에 빠지고 커넥션 풀이 꽉 차서 서비스가 터진 거야.  
  * (해결: 인프라 엔지니어는 무중단으로 스키마를 변경하는 Online DDL 옵션을 확인하거나, pt-online-schema-change 같은 도구 도입을 가이드해야 해.)

### **🚨 트러블슈팅 시나리오 2: WHERE 조건 없는 대참사 (Auto-commit의 공포)**

* **장애 상황**: 운영 DB에 직접 붙어서 특정 유저 데이터만 지우려다가 DELETE FROM users; (WHERE 절 누락)를 엔터 쳐버림. 데이터가 전부 날아감.  
* **주니어의 함정**: "빨리 롤백해야지\! ROLLBACK; 엔터\!" \-\> 하지만 복구되지 않음.  
* **원리를 알면 보이는 것**: MySQL 등 대다수의 RDBMS는 CLI로 접속 시 **기본적으로 autocommit=1 (자동 커밋)** 상태야. 엔터를 치는 순간 쿼리 1개가 1개의 독립된 트랜잭션으로 취급되어 즉시 디스크에 영구 반영(Commit)돼.  
  * (인프라 철칙: 운영 DB에서 DML 작업을 할 때는 무조건 START TRANSACTION; 또는 BEGIN;을 먼저 쳐서 자동 커밋을 끈 상태로 작업하고, SELECT로 검증한 뒤 수동으로 COMMIT; 해야 해.)

## **3\. 성능·튜닝 연결**

### **① WHERE vs HAVING 성능 튜닝**

* **설명**: 둘 다 조건을 걸러내는 필터링 역할이지만 성능 차이는 하늘과 땅 차이야.  
* **내부 동작**:  
  * WHERE는 디스크에서 메모리로 데이터를 퍼 올릴 때 일차적으로 필터링해. (가벼워짐)  
  * HAVING은 이미 올려서 다 묶어둔(GROUP BY) 결과물을 상대로 필터링해. (무거움)  
* **튜닝 포인트**: 그룹화할 필요가 없는 단순 조건(예: 나이 20세 이상)을 굳이 HAVING 절에 넣는 개발자가 있다면, 이를 WHERE 절로 올리도록 튜닝 가이드를 해야 해. 임시 테이블(Temp Table)에 올라가는 데이터 크기가 확 줄어서 쿼리가 수십 배 빨라져.

### **② 대용량 마이그레이션 시 autocommit 끄기**

* **설명**: 100만 건의 데이터를 INSERT 해야 하는 배치(Batch) 작업이 너무 느리다면?  
* **튜닝**: autocommit=1 상태에서는 1번 INSERT 할 때마다 트랜잭션 로그를 디스크에 쓰는(Flush) 엄청난 I/O 작업이 발생해. 스크립트 상단에 SET autocommit=0;을 넣고, 10,000건 INSERT 후 한 번씩 COMMIT;을 하도록 바꾸면 디스크 I/O가 획기적으로 줄어들어 처리 속도가 비약적으로 상승해.

## **4\. 이전/이후 강의 연결**

* **선행 지식 (34/35강)**  
  * 무결성 제약조건: 오늘 배운 INSERT나 UPDATE가 실패하는 이유는 이전 시간에 배운 외래키(FK)나 기본키(PK) 제약조건이라는 방패가 막아섰기 때문이야.  
* **후속 지식 (이후 강의)**  
  * **인덱스 (Index)**: 쿼리 실행 순서의 첫 관문인 WHERE 절이 데이터를 찾을 때, 테이블을 처음부터 끝까지 다 뒤질지(Full Table Scan), 아니면 목차만 보고 쏙 뽑아올지(Index Scan) 결정하는 원리를 배우게 될 거야. (성능 튜닝의 꽃\!)

## **5\. 바로 해볼 수 있는 실험**

K8s나 로컬 도커의 DB 컨테이너 안에서 직접 실험해 보자.

1. **Auto-commit의 무서움과 트랜잭션 방어망 체험하기**  
   * DB 콘솔 접속 후:  
     CREATE TABLE test\_rollback (id INT);  
     INSERT INTO test\_rollback VALUES (1), (2);  
     \-- 현재 세션의 오토커밋 확인  
     SELECT @@autocommit; 

     \-- 방어망 활성화 (트랜잭션 시작)  
     START TRANSACTION;   
     DELETE FROM test\_rollback; \-- 다 지움\!  
     SELECT \* FROM test\_rollback; \-- 데이터가 없는 것처럼 보임 (현재 트랜잭션 버퍼 내)

     \-- 아차\! 실수했네\! 복구\!  
     ROLLBACK;  
     SELECT \* FROM test\_rollback; \-- 데이터가 마법처럼 살아남

2. **TRUNCATE vs DELETE의 동작 차이 비교**  
   * 데이터가 10만 건 정도 있는 더미 테이블에서,  
   * DELETE FROM 테이블명; 을 쳐보고 걸리는 시간(초)을 확인해 봐. (한 땀 한 땀 삭제하며 트랜잭션 로그를 남겨서 느림)  
   * 데이터를 다시 채운 뒤, TRUNCATE TABLE 테이블명;을 쳐봐. (DDL이라서 트랜잭션 무시하고 테이블 자체를 밀어버린 뒤 껍데기만 새로 만들어 속도가 0.01초 만에 끝남. 대신 ROLLBACK 불가\!)

## **1\. 핵심 개념 정리 (클라우드 인프라 관점)**

메모리(RAM)의 '휘발성'과 '비싼 가격'이라는 단점을 극복하기 위해 데이터를 영구적으로 보관하는 보조기억장치(EBS, SSD)를 여러 개 묶어 쓰는 기술이야.

### **① RAID 0 (Striping \- 스트라이핑)**

* **정의**: 데이터를 여러 디스크에 조각조각 쪼개서(분산) 동시에 쓰고 읽는 방식.  
* **특징**: 디스크가 N개면 속도도 이론상 N배가 되지만(초고속), **디스크가 하나라도 고장 나면 전체 데이터가 파괴됨(안정성 0\)**.  
* **클라우드 실무**: 재부팅하면 날아가도 상관없는 '임시 캐시 데이터'나, 극단적인 I/O 성능이 필요한 경우 EC2 Instance Store 여러 개를 묶을 때 사용해.

### **② RAID 1 (Mirroring \- 미러링)**

* **정의**: 두 개의 디스크에 똑같은 데이터를 거울처럼 복사해서 쓰는 방식.  
* **특징**: 하나가 고장 나도 복제본이 있어서 매우 안전하지만, 용량 효율이 반토막(50%) 나고 쓰기 속도가 상대적으로 느림.

### **③ RAID 5 / 6 (Parity \- 패리티 분산)**

* **정의**: 데이터와 함께 오류 복구용 정보인 '패리티(Parity)'를 여러 디스크에 분산 저장하는 방식.  
* **특징**: 가성비와 안정성을 모두 챙긴 방식. RAID 5는 디스크 1개 고장까지, RAID 6은 2개 고장까지 버틸 수 있음. 단, 패리티를 계산하고 쓰는 데 '쓰기 페널티(Write Penalty)'가 발생해 쓰기 성능이 떨어짐.

### **④ RAID 10 (1+0, Nested RAID)**

* **정의**: RAID 1(거울 복사)로 안전하게 묶은 그룹들을 다시 RAID 0(스트라이핑)으로 묶어 속도까지 챙긴 하이브리드 방식.  
* **클라우드 실무**: I/O 성능과 안정성이 동시에 필요한 **대규모 RDBMS(관계형 데이터베이스) 서버 스토리지를 구성할 때 가장 많이 선택하는 아키텍처**야.

## **2\. 장애·트러블슈팅 연결**

### **🚨 이 원리가 드러나는 장애 상황**

**"자체 구축한 DB 서버(EC2)에서 dmesg에 I/O error가 뜨더니, 파티션 마운트가 해제되고 데이터베이스 데몬이 완전히 뻗어버린 현상"**

* **CS 지식의 발현**: 클라우드라고 디스크가 무적이 아니야. AWS EBS 역시 뒷단은 물리적인 하드웨어이기 때문에 드물지만 볼륨 장애가 발생해. 만약 주니어가 "속도가 제일 중요하니까\!"라며 DB 스토리지를 **RAID 0**으로 묶어버렸다면, 볼륨 하나가 불량 섹터(Bad block)를 뱉는 순간 시스템 전체의 데이터는 영구적으로 복구 불가능한 상태가 돼버려.

### **🕳️ 원리를 모르면 빠지는 함정**

* **주니어의 흔한 실수 (Write Penalty 간과)**: "스토리지 용량도 아끼면서 안정성도 챙겨야 하니까, 쓰기(Write)가 엄청나게 많이 일어나는 로깅 DB 시스템에 RAID 5를 적용하자\!"  
* **부작용**: RAID 5/6는 새로운 데이터를 쓸 때마다 전체 디스크의 패리티(오류 검출 코드)를 다시 계산해서 덮어써야 해. 이를 \*\*쓰기 페널티(Write Penalty)\*\*라고 부르는데, 쓰기 작업이 폭주하는 로깅 시스템에 RAID 5를 적용하면 디스크 병목(iowait 100%)이 발생해서 서버 성능이 처참하게 무너져.

### **💡 원리를 알면 보이는 것**

"아, 워크로드의 특성에 따라 디스크 레이아웃을 다르게 짜야 하는구나\! Read/Write가 빈번하고 절대 데이터가 날아가면 안 되는 **트랜잭션 DB는 비용이 들어도 RAID 10**으로 묶고, 단순히 읽기(Read) 위주의 거대한 아카이빙 데이터는 가성비 좋은 **RAID 5**로 구성해야겠다\!"

## **3\. 성능·튜닝 연결**

AWS 등 클라우드 벤더는 기본적으로 EBS 내부에서 데이터를 다중화(Mirroring)하여 보호하고 있어. 하지만 **클라우드 엔지니어가 직접 OS 레벨에서 소프트웨어 RAID를 튜닝**해야 하는 순간이 반드시 와.

* **단일 EBS 성능 한계 돌파 (RAID 0 스트라이핑 튜닝)**:  
  * AWS gp3 볼륨 하나의 최대 대역폭은 1,000 MB/s, 최대 IOPS는 16,000 수준이야. 만약 우리 서비스의 DB가 이보다 더 높은 30,000 IOPS를 요구한다면?  
  * 클라우드 엔지니어는 10,000 IOPS짜리 EBS 볼륨 3개를 EC2에 붙인 뒤, 리눅스 OS 레벨에서 mdadm 명령어를 이용해 \*\*RAID 0 (스트라이핑)\*\*으로 묶어 논리적인 1개의 볼륨으로 만들어버려. 이렇게 하면 단일 볼륨 한계를 우회해서 **30,000 IOPS를 달성**할 수 있어\! (이건 AWS 공식 문서에서도 추천하는 최고급 성능 튜닝 기법이야.)

## **4\. 이전/이후 강의 연결**

* **이전 지식 (10\~12강 메모리)**:  
  * "RAM은 엄청 빠르지만 전원을 끄면 날아간다(휘발성)"는 단점 때문에, 우리는 데이터를 잃지 않기 위해 보조기억장치(SSD/HDD)를 써야만 했어. 그 디스크들을 영리하게 조합해 성능과 안정성을 극대화하는 게 RAID야.  
* **이후 강의 (입출력장치 기법)**:  
  * 이렇게 빵빵하게 RAID로 묶은 디스크들이 대체 \*\*"어떤 방식(버스와 컨트롤러)"\*\*으로 CPU와 대화해서 데이터를 주고받는지, 입출력(I/O) 기법에 대해 배우게 될 거야.

## **5\. 바로 해볼 수 있는 실험**

내 클라우드 리눅스 서버에서 디스크 블록 장치들의 구조를 두 눈으로 확인해 보자.

1. **내 서버의 디스크(블록 장치) 구조 시각화하기**  
   * 터미널 명령어: lsblk  
   * 현재 내 EC2에 붙어있는 디스크(xvda, nvme0n1 등)와, 그 디스크가 어떻게 파티션으로 쪼개져서 리눅스의 어느 경로(/, /data 등)에 마운트(Mount)되어 있는지 트리 구조로 보여주는 가장 기본적인 인프라 명령어.  
2. **소프트웨어 RAID 설정 상태 확인하기**  
   * 터미널 명령어: cat /proc/mdstat  
   * 리눅스의 md(Multiple Device) 커널 모듈 상태를 보여줘. 만약 서버에 소프트웨어 RAID가 구성되어 있다면 md0 : active raid10 ... 처럼 어떤 디스크들이 묶여 있는지, 현재 동기화(Sync) 상태는 정상인지 확인할 수 있어.  
3. **실시간 디스크 I/O 병목(부하) 관찰하기**  
   * 터미널 명령어: iostat \-xd 1 (sysstat 패키지 필요)  
   * 1초마다 각 디스크(nvme1n1, nvme2n1 등)의 읽기/쓰기 속도(MB/s)와 \*\*%util (디스크 사용률)\*\*을 보여줘. RAID를 구성한 후 부하 테스트를 걸어보면, 여러 디스크에 I/O가 분산되어 %util이 골고루 오르는 걸 볼 수 있어\!

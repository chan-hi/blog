## **1\. 핵심 개념 정리**

### **① 프로세스의 메모리 구조 (Code, Data, Heap, Stack)**

* **정의**: 하나의 프로그램이 실행되어 메모리에 적재될 때, 역할에 따라 4개의 구역으로 나누어 저장되는 구조입니다.  
* **왜 존재하는가**: 명령어(코드), 정적 데이터, 동적 데이터, 임시 데이터를 한곳에 뒤섞어두면 관리가 불가능하고 메모리 충돌이 발생하기 때문입니다.  
* **어떻게 동작하는가**:  
  * **Code/Data (정적)**: 실행할 명령어와 전역 변수 등이 저장됩니다 (Read-only 성격).  
  * **Heap (동적)**: 개발자가 런타임에 동적으로 할당하는 공간(객체 생성 등)입니다. 가비지 컬렉터(GC)가 관리합니다.  
  * **Stack (동적)**: 함수 호출, 지역 변수 등 임시 데이터가 쌓였다가 사라지는 공간입니다.  
* **개념 간 관계**: 에러 로그에서 흔히 보는 Stack Trace는 이 Stack 영역의 함수 호출 기록을 역추적해서 보여주는 것입니다.

### **② PCB (Process Control Block)와 컨텍스트 스위칭**

* **정의**: PCB는 커널이 프로세스를 식별하고 관리하기 위한 '프로세스 신분증'이며, 컨텍스트 스위칭은 CPU가 A 프로세스에서 B 프로세스로 실행을 전환하는 과정입니다.  
* **왜 존재하는가**: 하나의 CPU 코어는 한 번에 하나만 실행할 수 있습니다. 수많은 프로세스가 동시에 도는 것처럼 보이게 하려면, 현재 상태(문맥)를 PCB에 저장하고 다른 PCB를 불러오는 작업이 필수입니다.  
* **어떻게 동작하는가**: 타이머 인터럽트 발생 ➔ CPU가 현재 프로세스 상태를 커널 영역의 PCB에 백업 ➔ 다음 실행할 프로세스의 PCB 데이터를 CPU 레지스터에 복구 ➔ 실행 재개.  
* **개념 간 관계**: 15강에서 배운 **시스템 콜과 이중 모드 전환**보다 이 컨텍스트 스위칭이 훨씬 무겁고 오버헤드가 큽니다.

### **③ 멀티 프로세스 vs 멀티 스레드**

* **정의**: 한 프로그램 내에서 여러 갈래의 작업을 동시에 처리하는 두 가지 접근법입니다.  
* **왜 존재하는가**: 무거운 작업을 병렬로 처리하여 응답성과 처리량을 높이기 위해 존재합니다.  
* **어떻게 동작하는가**:  
  * **멀티 프로세스**: 각자 독립된 메모리 구조(Code, Data, Heap, Stack)를 가집니다. 하나가 죽어도 다른 프로세스에 영향이 없습니다 (안전하지만 무겁고 자원 공유가 힘듦).  
  * **멀티 스레드**: 하나의 프로세스 안에서 Code, Data, Heap 영역과 열린 파일(자원)을 공유하고, 각 스레드는 자신만의 Stack, PC(Program Counter)만 따로 가집니다. (가볍고 빠르지만, 스레드 하나가 에러를 내면 프로세스 전체가 죽을 수 있음).

### **④ IPC (Inter-Process Communication)와 RPC**

* **정의**: 독립적인 프로세스 간에 데이터를 주고받거나(IPC), 네트워크 너머의 다른 서버의 함수를 내 것처럼 호출(RPC)하는 기술입니다.  
* **왜 존재하는가**: 멀티 프로세스 구조나 마이크로서비스(MSA) 환경에서는 메모리가 분리되어 있어, 서로 협력하려면 통신 창구가 필요합니다.  
* **어떻게 동작하는가**:  
  * **공유 메모리**: 커널의 개입을 최소화하고 메모리 일부를 공유해 초고속으로 통신.  
  * **메시지 패싱**: 커널(또는 네트워크)을 거쳐 파이프, 시그널, 소켓 등으로 안전하게 통신. (gRPC 등이 RPC의 대표적 예시)

## **2\. 장애·트러블슈팅 연결**

### **💡 컨테이너(Docker/K8s)의 좀비 프로세스 (Zombie Process) 함정**

* **이 원리가 드러나는 장애 상황**:  
  Kubernetes Worker Node의 CPU/Memory는 남는데 "cannot allocate memory"나 "fork: retry: Resource temporarily unavailable" 에러가 나면서 파드 생성이 실패함. ps aux | grep Z를 치면 \[defunct\] (좀비) 프로세스가 수천 개 쌓여 있음.  
* **원리를 모르면 빠지는 함정**:  
  메모리나 CPU가 부족한 줄 알고 노드 스케일아웃만 반복함.  
* **원리를 알면 보이는 것**:  
  "자식 프로세스가 종료되었는데, 부모가 PCB를 거두어주지(wait) 않아 커널의 프로세스 테이블(PID 공간)이 꽉 찼구나\!" K8s 컨테이너의 PID 1 (Entrypoint) 앱이 자식 프로세스를 제대로 수거하지 못하는 구조적 문제를 파악하고, tini나 dumb-init 같은 init 프로세스를 도입하여 해결.

### **💡 OOM (Out Of Memory)과 힙(Heap) 메모리 누수**

* **이 원리가 드러나는 장애 상황**:  
  K8s 파드가 자꾸 OOMKilled 상태로 죽고 재시작됨.  
* **원리를 모르면 빠지는 함정**:  
  무작정 Pod의 resources.limits.memory만 계속 늘려줌. 며칠 뒤 또 죽음.  
* **원리를 알면 보이는 것**:  
  "애플리케이션의 힙(Heap) 영역에 동적 할당된 객체들이 가비지 컬렉션(GC)으로 반환(Free)되지 않고 메모리 누수(Memory Leak)가 발생했구나\!" Heap Dump를 떠서 메모리를 쥐고 안 놔주는 범인 코드(컬렉션 등)를 찾아내야 함을 인지.

### **💡 스레드 풀 고갈과 블로킹 I/O (대기 상태)**

* **이 원리가 드러나는 장애 상황**:  
  Java Spring(Tomcat) 서버의 CPU 사용률은 5%밖에 안 되는데, 외부 트래픽 응답이 전부 Timeout 남.  
* **원리를 모르면 빠지는 함정**:  
  "CPU가 놀고 있는데 왜 처리를 안 해?"라며 ALB 로드밸런서나 네트워크 문제만 의심.  
* **원리를 알면 보이는 것**:  
  "모든 스레드가 DB 쿼리나 외부 API 응답을 기다리느라 \*\*블로킹(대기 상태, Waiting)\*\*에 빠졌구나\! 일할 수 있는(Ready/Running) 스레드가 하나도 없네." Thread Dump를 떠서 어떤 DB나 외부 서비스가 지연을 일으키고 있는지 확인.

## **3\. 성능·튜닝 연결**

* **관련 시스템 파라미터/설정값**:  
  * JVM의 힙 사이즈 설정 (-Xms, \-Xmx)  
  * Node.js의 힙 사이즈 설정 (--max-old-space-size)  
  * WAS(Tomcat, Nginx)의 스레드 풀(Thread pool) / 워커 프로세스 개수 설정  
* **내부 동작과 튜닝의 원리**:  
  CPU 코어 수보다 스레드를 너무 많이 만들면, 실제 연산(Running)보다 스레드 간 **컨텍스트 스위칭(Context Switching)** 오버헤드가 더 커져 오히려 전체 성능이 저하됩니다. I/O 블로킹이 잦은 서비스라면 스레드를 넉넉히(예: 200개), CPU 연산 위주라면 코어 수에 맞게(예: 4\~8개) 스레드 풀을 튜닝하는 것이 핵심입니다.  
* **부작용**: K8s에서 CPU limit을 너무 타이트하게 주면, CFS 스케줄러에 의해 스로틀링이 걸려 컨텍스트 스위칭 지연이 폭발적으로 증가해 레이턴시가 튀는 현상이 발생합니다.

## **4\. 이전/이후 강의 연결**

* **선행 지식 (15강)**:  
  운영체제가 하드웨어를 통제하는 큰 그림(커널, 시스템 콜, 이중 모드)을 배웠습니다.  
* **이 강의의 위치 (현재)**:  
  그 커널 위에서 애플리케이션이 실제로 어떻게 생명주기를 가지고(프로세스 상태), 메모리 공간을 차지하며, 어떻게 나뉘어(스레드) 돌아가는지를 배웠습니다.  
* **후행 지식 (이후 내용)**:  
  여러 스레드가 Heap 영역 같은 '공유 자원'에 동시에 접근하면 값이 꼬이게 됩니다. 이를 해결하기 위해 ➔ \*\*동기화(Mutex/Semaphore)와 교착 상태(Deadlock)\*\*에 대해 배울 차례입니다.

## **5\. 바로 해볼 수 있는 실험**

본인의 Linux 서버나 K8s 파드 셸에 접속해서 확인해 보세요\!

1. **top \-H로 내 앱의 실제 '스레드'들 확인하기**  
   \# 프로세스(PID) 하나만 보이던 top 명령어에 \-H를 주면,   
   \# 해당 프로세스를 구성하는 수많은 '스레드'들이 각각 CPU를 얼마나 쓰는지 보입니다.  
   top \-H \-p \<자바나 노드 프로세스의 PID\>

2. **좀비 프로세스 눈으로 직접 찾기**  
   \# 리눅스 시스템에서 상태(STAT)가 'Z' (Zombie/Defunct)인 프로세스를 찾습니다.  
   ps aux | awk '{ print $8 " " $2 }' | grep \-w Z

3. **IPC 중 하나인 '시그널(Signal)' 직접 보내보기**  
   \# 프로세스 간 메시지 전달(IPC)인 시그널을 직접 보내서 프로세스를 제어해 봅니다.  
   \# SIGTERM(15): 정상 종료 요청 (앱 내의 시그널 핸들러가 처리함)  
   kill \-15 \<PID\> 

   \# SIGKILL(9): 커널이 강제로 죽임 (핸들러가 무시할 수 없음)  
   kill \-9 \<PID\>

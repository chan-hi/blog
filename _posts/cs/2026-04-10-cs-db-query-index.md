## **1\. 핵심 개념 정리**

### **① 서브쿼리(Subquery)와 조인(Join)**

* **정의**:  
  * **서브쿼리**: 하나의 쿼리문 안에 포함된 또 다른 쿼리(Nested Query).  
  * **조인**: 여러 테이블을 특정 조건(주로 PK-FK)으로 결합하여 하나의 결과셋으로 만드는 연산.  
* **왜 존재하는가**: 데이터가 정규화(Normalization)되어 여러 테이블에 흩어져 있기 때문이야. 사용자 이름은 users에, 게시글 내용은 posts에 있을 때 이를 한 번에 가져오려면 반드시 필요해.  
* **어떻게 동작하는가**:  
  * **이너 조인(Inner Join)**: 양쪽 테이블에 모두 데이터가 있는 '교집합'만 반환.  
  * **아우터 조인(Outer Join)**: 한쪽 테이블의 데이터를 보존하면서 반대쪽에 데이터가 없으면 NULL로 채움.  
* **개념 간 관계**: 많은 서브쿼리는 조인으로 변환(Rewrite)이 가능해. 최신 DBMS 최적화기(Optimizer)는 서브쿼리를 내부적으로 조인으로 바꿔서 실행하기도 하지만, 명시적으로 조인을 쓰는 게 가독성과 성능 면에서 유리할 때가 많아.

### **② 뷰(View)**

* **정의**: SELECT 문의 결과를 마치 실제 테이블처럼 이름 붙여 사용하는 **'가상의 테이블'**.  
* **왜 존재하는가**:  
  1. **복잡성 은닉**: 조인이 5\~6개 걸린 복잡한 쿼리를 매번 쓰기 힘드니까 뷰로 감싸서 단순하게 조회하려고.  
  2. **보안(접근 제어)**: 전체 테이블 중 민감한 정보(급여, 주민번호 등)만 제외하고 뷰를 만들어 해당 뷰에만 사용자 권한을 줄 수 있어.  
* **어떻게 동작하는가**: 물리적으로 데이터를 저장하지 않고, 뷰를 조회할 때마다 정의된 SELECT 문이 내부적으로 다시 실행되는 구조야.

### **③ 인덱스(Index)**

* **정의**: 테이블의 검색 속도를 높이기 위해 별도로 생성하는 **'색인 자료구조'**. 주로 **B-Tree** 구조를 사용해.  
* **왜 존재하는가**: 수억 건의 데이터가 있을 때, 처음부터 끝까지 다 뒤지는 **Full Table Scan**은 디스크 I/O 폭발을 일으키기 때문이지.  
* **어떻게 동작하는가**: 특정 컬럼(예: 닉네임)을 기준으로 데이터를 정렬하여 트리 구조로 저장해. 검색 시 O(log N)의 속도로 원하는 위치를 바로 찾아가.

## **2\. 장애·트러블슈팅 연결**

### **🚨 트러블슈팅 시나리오 1: 인덱스 부재로 인한 CPU/디스크 I/O 폭증**

* **장애 상황**: 서비스 응답이 갑자기 느려짐. CloudWatch에서 RDS의 **Disk Read IOPS**가 평소보다 10배 높고, CPU 사용률이 90% 이상 유지됨. Slow Query Log를 확인하니 특정 SELECT 쿼리가 수십 초씩 걸리고 있음.  
* **주니어의 함정**: "사용자가 늘었나 봐요\! RDS 인스턴스 타입을 m5.large에서 m5.2xlarge로 업그레이드합시다\!" (비용 4배 상승)  
* **원리를 알면 보이는 것**: EXPLAIN 명령어로 해당 쿼리의 실행 계획을 확인했을 때 type: ALL (Full Table Scan)이 뜨고 있다면, 인덱스가 없어서 DB가 매번 수백만 개의 행을 디스크에서 읽어 메모리로 올리는 중인 거야. 인스턴스 업그레이드가 아니라 **인덱스 하나만 생성(CREATE INDEX)하면 IOPS와 CPU가 즉시 정상화**돼.

### **🚨 트러블슈팅 시나리오 2: 인덱스 남발로 인한 쓰기 장애**

* **장애 상황**: 배치(Batch) 작업으로 대량의 데이터를 INSERT 하는데 속도가 갈수록 느려지고 DB의 **Write Latency**가 치솟음.  
* **주니어의 함정**: "디스크 성능(EBS 타입)이 문젠가? gp2를 gp3나 io2로 바꿀까요?"  
* **원리를 알면 보이는 것**: 해당 테이블에 인덱스가 너무 많이(예: 10개 이상) 걸려 있는 건 아닌지 봐야 해. **데이터가 삽입될 때마다 모든 인덱스 트리 구조를 다시 계산해서 디스크에 써야 하거든.** 조회 속도를 위해 만든 인덱스가 쓰기 성능에는 '독'이 된 상황이야. 불필요한 인덱스를 삭제해야 해결돼.

## **3\. 성능·튜닝 연결**

### **① 커널/인프라 파라미터와 인덱스 (innodb\_buffer\_pool\_size)**

* **설명**: MySQL(InnoDB)에서 인덱스와 실제 데이터 페이지를 메모리에 캐싱하는 공간이야.  
* **튜닝**: 인덱스가 아무리 잘 걸려 있어도, 이 버퍼 풀 크기가 작으면 계속 디스크 I/O가 발생해. 보통 서버 \*\*전체 RAM의 70\~80%\*\*를 이 값으로 할당하는 게 정석이야.  
* **부작용**: 너무 크게 잡으면 OS 커널이 쓸 메모리가 부족해져서 **OOM Killer**가 DB 프로세스를 죽여버릴 수 있으니 주의해야 해.

### **② 카디널리티(Cardinality) 고려**

* **설명**: 특정 컬럼의 중복도가 낮을수록(값이 다양할수록) 카디널리티가 높다고 해.  
* **원리**: '성별' 컬럼(남/여)에 인덱스를 거는 건 의미가 없어. DB가 트리를 타는 것보다 그냥 다 읽는 게 빠르다고 판단하거든(Optimizer의 판단). 인덱스는 '유저 ID', '이메일'처럼 **값이 고유할수록 효과가 극대화**돼.

## **4\. 이전/이후 강의 연결**

* **선행 지식 (36강)**  
  * SQL 기본 문법: SELECT, WHERE 절을 알아야 조인과 서브쿼리를 구성할 수 있어.  
* **후속 지식 (이후 강의)**  
  * **DB 설계와 정규화**: 왜 테이블을 쪼개야 하는지(조인이 왜 필요한지)에 대한 설계론적인 근거를 배우게 돼.

## **5\. 바로 해볼 수 있는 실험**

1. **실행 계획(EXPLAIN) 확인하기 (가장 중요\!)**  
   * DB 콘솔에서 평소 자주 쓰는 쿼리 앞에 EXPLAIN을 붙여봐.  
   * EXPLAIN SELECT \* FROM users WHERE nickname \= 'test';  
   * 결과에서 type 컬럼이 ALL이면 위험(인덱스 안 탐), ref나 const면 안전(인덱스 탐)이야.  
2. **인덱스 생성 전후 속도 비교**  
   * 데이터가 수만 건 있는 테이블에서 WHERE 절에 인덱스 없는 컬럼으로 조회해보고 시간을 측정해봐.  
   * CREATE INDEX 후에 같은 쿼리를 날려봐. 0.x 초 단위에서 0.00x 초 단위로 바뀌는 마법을 볼 수 있어.  
3. **뷰(View) 생성해보기**  
   * CREATE VIEW v\_user\_posts AS SELECT u.name, p.title FROM users u JOIN posts p ON u.id \= p.user\_id;  
   * SELECT \* FROM v\_user\_posts; 로 복잡한 조인 결과를 한 번에 조회해봐. 인프라 운영 모니터링 쿼리를 짤 때 아주 유용해.

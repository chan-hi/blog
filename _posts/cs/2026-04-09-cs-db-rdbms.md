## **1\. 핵심 개념 정리**

### **① 키 (Key): PK와 FK**

* **정의**: 테이블 내에서 특정 레코드(행)를 고유하게 식별하기 위한 필드(PK)이거나, 다른 테이블의 레코드를 참조하기 위한 필드(FK)야.  
* **왜 존재하는가**: 수천만 건의 데이터 속에서 "정확히 그 데이터"를 찾아내고, 중복 저장을 막기 위해서. 또, 쪼개진 테이블들을 나중에 다시 연결(Join)하기 위한 연결고리가 필요하기 때문이지.  
* **어떻게 동작하는가**:  
  * **기본키(PK)**: DB는 이 필드에 중복된 값이나 NULL(빈 값)이 들어오는 것을 엔진 단에서 원천 차단해.  
  * **외래키(FK)**: 새 데이터가 들어오거나 삭제될 때, DB 엔진이 부모 테이블을 쳐다보며 "저쪽에 이 데이터가 진짜 있어?" 하고 검증(Lock)해.  
* **개념 간 관계**: N:M (다대다) 관계를 DB에 구현하려면, 두 테이블의 PK를 자신의 FK로 끌어와서 조합(복합키)하는 \*\*'중간 테이블(Mapping Table)'\*\*이 반드시 필요해.

### **② 무결성 제약 조건 (Integrity Constraints)**

* **정의**: DB에 쓰레기 데이터(Garbage Data)가 들어오지 못하게 막는 데이터베이스의 방어막(규칙). (도메인, 키, 엔티티, 참조 무결성)  
* **왜 존재하는가**: 프론트엔드나 백엔드 코드의 버그로 인해 나이가 \-1살로 들어오거나, 존재하지 않는 상품을 결제하는 등 비즈니스 로직이 꼬이는 것을 막는 **최후의 보루** 역할을 해.  
* **어떻게 동작하는가**: 쿼리(INSERT/UPDATE/DELETE)가 DB에 도착하면, DB는 디스크에 쓰기 전에 이 조건들을 검사해. 위배되면 쿼리 처리를 거부하고 즉시 Error를 반환(Rollback)해.

## **2\. 장애·트러블슈팅 연결**

### **🚨 트러블슈팅 시나리오 1: 참조 무결성이 드러나는 순간 (FK 제약)**

* **장애 상황**: 백엔드 파드에서 회원 탈퇴 API 호출 시 HTTP 500이 대량 발생. 로그를 까보니 Cannot delete or update a parent row: a foreign key constraint fails 에러가 찍혀 있음.  
* **주니어의 함정**: "어? 회원 탈퇴 코드(Delete)에 버그가 있나?" 하고 애플리케이션의 컨트롤러/서비스 로직만 밤새 뒤져봄.  
* **원리를 알면 보이는 것**: 회원을 지우려는데, '주문 내역'이나 '게시글' 테이블이 그 회원의 ID(PK)를 외래키(FK)로 쥐고 있는 상태야. 이때 부모(회원)를 지우면 고아 데이터가 생기니까 DB의 \*\*참조 무결성 제약 조건(Restrict)\*\*이 삭제를 막은 거지.  
  * 해결책: 개발자에게 "FK 제약조건에 걸리는데, 연쇄 삭제(Cascade) 처리할지 아니면 상태만 바꾸는 Soft Delete(is\_deleted 플래그)로 갈지 논의하시죠"라고 리드할 수 있어.

### **🚨 트러블슈팅 시나리오 2: 엔티티 무결성이 드러나는 순간 (PK 제약)**

* **장애 상황**: 테라폼이나 CI/CD 파이프라인에서 초기 DB 마이그레이션 스크립트를 돌리는데, Duplicate entry 'admin' for key 'PRIMARY' 에러가 뜨면서 파이프라인이 중단됨.  
* **주니어의 함정**: "인프라 파이프라인이 왜 터지지? 젠킨스 설정이 문젠가?"  
* **원리를 알면 보이는 것**: 인프라 문제가 아니라, 밀어 넣으려는 초기 데이터 스크립트(.sql) 안에 동일한 PK 값을 가진 INSERT 문이 두 번 이상 있거나, 이미 데이터가 있는데 덮어쓰려 한 거야. DB가 **엔티티 무결성(기본키 중복 불가)** 원칙에 따라 방어해 준 정상적인 상황임을 인지하게 됨.

## **3\. 성능·튜닝 연결**

### **① FK(외래키) 검사와 foreign\_key\_checks 파라미터 (MySQL/MariaDB)**

* **설명**: 외래키 제약조건은 데이터 정확성에 좋지만, **성능에는 꽤 큰 악영향**을 미쳐. FK가 걸려있으면 INSERT나 DELETE를 할 때마다 부모 테이블에 가서 해당 데이터가 있는지 조회해야 하고, 이때 다른 트랜잭션이 부모 테이블을 수정하지 못하게 \*\*공유 락(Shared Lock)\*\*을 걸어버려.  
* **튜닝**: 수십 기가의 DB 덤프 파일을 임포트(Import)하거나 대용량 배치 작업을 할 때 속도가 미친 듯이 느리다면 이 검사 때문일 수 있어.  
  * 복구할 때 스크립트 맨 위에 SET FOREIGN\_KEY\_CHECKS \= 0; (끄기)을 넣고, 복구가 끝난 뒤 SET FOREIGN\_KEY\_CHECKS \= 1; (켜기)로 되돌리면 **복구 속도가 몇 배는 빨라져.** (인프라 엔지니어의 핵심 팁\!)

### **② 무결성 제약조건 vs 애플리케이션 검증**

* **아키텍처 관점**: DB의 제약조건 검사(특히 복잡한 제약이나 FK 검사)가 많아질수록 DB CPU 사용량이 올라가. 대용량 트래픽을 받는 최신 MSA 아키텍처에서는 일부러 DB의 FK(외래키)를 물리적으로 끊어버리고, 데이터 정합성 체크를 백엔드 애플리케이션 레벨(코드)로 넘겨서 DB 부하를 줄이는 튜닝 기법도 많이 써.

## **4\. 이전/이후 강의 연결**

* **선행 지식 (34강)**  
  * 트랜잭션과 ACID: 트랜잭션의 'C(일관성)'를 물리적으로 지켜주는 수단이 바로 오늘 배운 '무결성 제약 조건'이야.  
* **후속 지식 (이후 강의)**  
  * **조인(Join)**: 분리된 테이블을 묶기 위해 FK라는 '연결고리'를 팠으니, 이제 쿼리 단에서 이 고리를 어떻게 묶어서(Join) 꺼내올지 배우게 돼.  
  * **인덱스(Index)**: 기본키(PK)는 제약조건이기도 하지만, DB 엔진 내부적으로 데이터를 가장 빠르게 찾게 해주는 \*\*'클러스터링 인덱스'\*\*를 생성하는 기준이 돼. (PK 설계가 인프라 디스크 I/O에 미치는 영향이 여기서 나와\!)

## **5\. 바로 해볼 수 있는 실험**

K8s나 도커에 띄워둔 MySQL/MariaDB 파드에 접속해서 바로 테스트해 봐\!

1. **테이블의 제약조건과 키(Key) 눈으로 확인하기**  
   * 아무 데이터베이스나 골라서 테이블 생성 DDL을 확인해 봐:  
   * SHOW CREATE TABLE {테이블명}\\G  
   * 출력 결과에 PRIMARY KEY, CONSTRAINT, FOREIGN KEY ... REFERENCES 같은 키워드가 어떻게 정의되어 있는지 인프라 레벨에서 직접 확인할 수 있어.  
2. **무결성 위배 고의로 발생시켜보기 (방패 테스트)**  
   * 간단한 테이블을 하나 만들고 기본키(PK)를 넣어봐.  
     CREATE TABLE test\_pk (id INT PRIMARY KEY, name VARCHAR(50));

   * 똑같은 ID를 두 번 넣어봐.  
     INSERT INTO test\_pk VALUES (1, 'Kube');  
     INSERT INTO test\_pk VALUES (1, 'Docker'); \-- 여기서 에러 발생\!

   * ERROR 1062 (23000): Duplicate entry '1' for key 'PRIMARY' 라는 에러를 직접 보면, DB가 방어막 역할을 어떻게 수행하는지 체감할 수 있을 거야.

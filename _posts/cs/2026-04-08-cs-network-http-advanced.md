## **1\. 핵심 개념 정리**

### **① HTTP의 진화 (1.1 → 2 → 3\)**

* **정의**: 웹의 성능 한계를 극복하기 위해 통신 방식(메커니즘)을 개선해 온 과정.  
* **왜 존재하는가**: HTTP/1.1의 **HOLB(Head-of-Line Blocking)** 문제 때문이야. 앞선 요청이 늦어지면 뒤의 요청들이 줄줄이 대기하는 병목 현상을 해결해야 했거든.  
* **어떻게 동작하는가**:  
  * **HTTP/1.1**: 요청당 연결 1개(또는 파이프라이닝), 텍스트 기반 헤더 중복 전송.  
  * **HTTP/2**: **멀티플렉싱(Multiplexing)** 도입. 하나의 TCP 연결에서 여러 요청/응답을 동시에 처리. 헤더 압축(HPACK).  
  * **HTTP/3**: TCP의 태생적 한계(패킷 손실 시 전체 대기)를 극복하기 위해 **UDP 기반의 QUIC** 프로토콜 사용.  
* **개념 간 관계**: L4(TCP/UDP)의 전송 방식 변화가 L7(HTTP)의 성능 향상을 이끄는 관계야.

### **② 쿠키(Cookie)와 캐시(Cache)**

* **정의**: 스테이트리스(Stateless)한 HTTP를 보완하고 성능을 높이는 도구들.  
* **왜 존재하는가**:  
  * **쿠키**: "로그인 상태"를 유지하기 위해(상태 기억).  
  * **캐시**: 동일한 데이터를 매번 서버에서 가져오는 낭비를 막기 위해(성능).  
* **어떻게 동작하는가**:  
  * **쿠키**: 서버가 Set-Cookie 헤더를 주면 클라이언트가 저장 후 매 요청마다 Cookie 헤더에 담아 보냄.  
  * **캐시**: 브라우저나 중간 서버(CDN)에 복사본을 저장하고 Cache-Control 헤더에 따라 재사용 여부 결정.  
* **개념 간 관계**: 쿠키는 '누구인지' 식별하고, 캐시는 '데이터를 얼마나 빨리 줄지' 결정해.

### **③ 프록시 서버 (Forward vs Reverse)**

* **정의**: 클라이언트와 서버 사이에서 대리인 역할을 수행하는 서버.  
* **왜 존재하는가**: 보안, 캐싱, 부하 분산(Load Balancing)을 위해.  
* **어떻게 동작하는가**:  
  * **포워드 프록시**: 클라이언트 쪽에서 서버로 나가는 요청을 대행(내부망 보안, 캐싱).  
  * **리버스 프록시**: 서버 쪽에서 클라이언트 요청을 받아 뒷단 서버로 전달(**ALB, Nginx**).  
* **개념 간 관계**: 클라우드 인프라의 핵심인 **로드밸런서**가 바로 대표적인 리버스 프록시야.

## **2\. 장애·트러블슈팅 연결**

### **🚨 시나리오 A: 로드밸런서의 "Sticky Session" 불일치**

* **이 원리가 드러나는 장애 상황**: 사용자가 로그인을 했는데, 새로고침을 할 때마다 간헐적으로 로그아웃 처리가 됨. 서버 로그에는 에러가 없는데 특정 서버 노드에서만 세션 정보를 못 찾음.  
* **원리를 모르면 빠지는 함정**: "서버 소스코드 버그인가?" 하고 로직만 뒤짐.  
* **원리를 알면 보이는 것**: "HTTP는 스테이트리스해서 서버가 클라이언트를 모르는데, 로드밸런서(리버스 프록시)가 **쿠키 기반의 세션 고정(Session Affinity)** 설정을 안 해줬구나\! 요청이 매번 다른 서버 노드로 튀고 있네." \-\> ALB 설정에서 Sticky Session을 활성화하거나 세션 저장소를 Redis로 중앙화해야 함을 파악.

### **🚨 시나리오 B: 캐시 오염(Cache Pollution) 및 미갱신**

* **이 원리가 드러나는 장애 상황**: 프론트엔드 코드를 배포했는데, 사용자들에게 이전 버전의 화면이 계속 보임. 서버에는 최신 파일이 있는데 브라우저나 CDN이 옛날 파일을 줌.  
* **원리를 모르면 빠지는 함정**: "배포가 꼬였나?" 하고 서버만 계속 재배포함.  
* **원리를 알면 보이는 것**: "Cache-Control 헤더 설정이 너무 길게 잡혀있거나(Immutable), CDN(리버스 프록시 캐시) 무효화(Invalidation)를 안 했구나." \-\> 응답 헤더의 Age나 X-Cache 로그를 보고 캐시 히트 여부 확인 후 Purge 수행.

## **3\. 성능·튜닝 연결**

* **Nginx의 proxy\_buffer\_size와 Large Headers**  
  * **파라미터**: proxy\_buffer\_size, proxy\_buffers  
  * **내부 동작**: 리버스 프록시(Nginx)가 백엔드 서버로부터 응답 헤더를 받을 때 사용하는 메모리 크기.  
  * **잘못 튜닝 시**: 쿠키나 JWT 토큰이 너무 커서 헤더가 버퍼 크기를 넘어가면 Nginx는 **502 Bad Gateway**를 뱉거나 디스크에 임시 파일을 써서 성능이 급락함. upstream sent too big header 에러 로그가 단서임.  
* **HTTP/2 활성화 (Multiplexing 이용)**  
  * **설정**: Nginx listen 443 ssl http2; 또는 ALB의 HTTP/2 설정.  
  * **효과**: 한 페이지에 수많은 이미지/스크립트가 있을 때, 1.1처럼 순차적으로 안 기다리고 한 번에 쏟아부어 로딩 속도를 2\~3배 높임.

## **4\. 이전/이후 강의 연결**

* **이전 강의 (\[31강. HTTP 기초\])**: 시작 라인, 헤더, 상태 코드의 기초가 잡혀있어야 이번 강의의 '진화' 과정이 이해됨.  
* **이후 강의 (\[33강. 보안과 HTTPS\])**: HTTP/2와 3는 사실상 암호화(SSL/TLS)가 필수야. 이 성능 개선이 보안 기술과 어떻게 결합되는지 다음 강의에서 다룰 거야.

## **5\. 바로 해볼 수 있는 실험**

### **실험 1: 내 브라우저는 지금 어떤 HTTP 버전을 쓸까?**

1. 크롬 개발자 도구(F12) \-\> **Network** 탭 클릭.  
2. 테이블 헤더 우클릭 \-\> **Protocol** 체크.  
3. h2 (HTTP/2), h3 (HTTP/3), http/1.1 중 어떤 게 찍히는지 확인.  
* **포인트**: 구글이나 유튜브는 이미 h3를 쓰고 있을 거야. 왜 빠른지 체감해 봐.

### **실험 2: 리버스 프록시 헤더 확인 (X-Forwarded-For)**

\# Nginx나 ALB 뒤에 있는 서버에서 로그 찍어보기  
\# 클라이언트의 실제 IP를 확인하려면 리버스 프록시가 넘겨주는 헤더를 봐야 함  
curl \-I \[https://www.google.com\](https://www.google.com)

* **포인트**: 응답 헤더에 Via나 Server 정보가 있는지, 혹은 내가 서버라면 X-Forwarded-For 헤더에 클라이언트의 진짜 IP가 담겨 오는지 확인해 봐. 리버스 프록시가 중간에서 정보를 어떻게 조작/전달하는지 알 수 있어.

### **실험 3: 쿠키의 동작 확인**

\# 특정 사이트에 접속하며 헤더에 찍히는 Set-Cookie 보기  
curl \-v \[https://www.naver.com\](https://www.naver.com) 2\>&1 | grep "set-cookie"

* **포인트**: 서버가 클라이언트에게 "이 값을 다음 요청부터 꼭 가져와\!"라고 명령하는 Set-Cookie 헤더의 실제 모습을 관찰해 봐.

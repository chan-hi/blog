## **1\. 핵심 개념 정리**

### **① DNS (Domain Name System)**

* **정의**: 호스트의 IP 주소를 사람이 읽기 쉬운 문자열(도메인 네임)로 변환해주는 분산 데이터베이스 시스템.  
* **왜 존재하는가**: 수만 개의 서버 IP를 외우는 건 불가능하고, 서버 IP가 바뀌더라도 사용자는 동일한 도메인으로 접속할 수 있게 하기 위해(유연성) 존재해.  
* **어떻게 동작하는가**: 로컬 네임 서버 \-\> 루트 \-\> TLD(.com) \-\> 권한 있는 네임 서버 순으로 계층적 질의(Recursive Query)를 수행해 IP를 알아냄.  
* **개념 간 관계**: L3(IP) 위에 존재하며, 브라우저가 TCP 연결(L4)을 맺기 위해 가장 먼저 수행하는 '선행 작업'이야.

### **② HTTP의 스테이트리스(Stateless) 특성**

* **정의**: 서버가 클라이언트의 이전 요청 상태를 보존하지 않고, 모든 요청을 독립적으로 처리하는 성질.  
* **왜 존재하는가**: 서버가 수만 명의 상태를 다 기억하면 메모리가 터져나가고, 서버를 여러 대(Scale-out)로 늘렸을 때 상태 공유가 너무 힘들기 때문이야. 즉, \*\*확장성(Scalability)\*\*을 위해 선택한 설계지.  
* **어떻게 동작하는가**: 매 요청마다 필요한 정보를 다 담아서 보내거나, 쿠키/세션/토큰 같은 보완 기술을 사용해 상태를 '흉내' 냄.  
* **개념 간 관계**: 이 특성 때문에 우리가 클라우드에서 '무상태(Stateless) 애플리케이션'을 설계하고 L4/L7 로드밸런싱을 자유롭게 할 수 있는 거야.

### **③ HTTP 메시지 구조 (Method & Status Code)**

* **정의**: 요청 라인(메서드, 경로), 상태 라인(코드), 헤더, 본문(Body)으로 구성된 규격화된 텍스트 메시지.  
* **왜 존재하는가**: 서로 다른 개발자가 만든 브라우저와 서버가 "이건 조회용이야(GET)", "이건 성공했어(200)"라고 약속된 언어로 대화하기 위해 존재해.  
* **어떻게 동작하는가**: 텍스트 기반으로 전송되며, 헤더에는 메타데이터(타입, 길이 등), 본문에는 실제 자원(HTML, JSON 등)이 담김.  
* **개념 간 관계**: REST API 설계의 근간이 되며, 클라우드 모니터링(CloudWatch 등)에서 에러율을 측정하는 기준이 돼.

## **2\. 장애·트러블슈팅 연결**

### **🚨 시나리오 A: "502 Bad Gateway" vs "504 Gateway Timeout"**

* **장애 상황**: AWS ALB 뒤에 웹 서버를 두었는데, 사용자가 접속하면 502나 504 에러가 발생함.  
* **원리를 모르면 빠지는 함정**: "인터넷이 느린가?" 하고 내 PC 네트워크를 점검하거나, 무작정 서버 사양(Instance Type)을 올림.  
* **원리를 알면 보이는 것**:  
  * **502(Bad Gateway)**: 중간 서버(ALB)가 백엔드 서버로부터 '잘못된 응답'을 받았거나, TCP 연결이 즉시 끊겼을 때 발생. (주로 서버 프로세스가 죽어있음)  
  * **504(Gateway Timeout)**: 백엔드 서버가 웅답은 하려고 노력 중이나, ALB의 타임아웃 설정 시간 내에 HTTP 응답을 못 보낼 때 발생. (주로 DB 쿼리 지연이나 서버 과부하)  
  * **추론**: "로그에 504가 찍히네? 이건 네트워크 연결 문제가 아니라 서버 내부 연산이 느려진 거구나\! DB Slow Query를 먼저 보자."

### **🚨 시나리오 B: "403 Forbidden" 에러와 IAM 권한**

* **장애 상황**: S3 버킷에 파일을 올리려는데 403 Forbidden 에러가 뜸.  
* **원리를 모르면 빠지는 함정**: "로그인을 안 했나?" 하고 401(Unauthorized)과 혼동하거나, 파일 경로(URL)가 틀린 줄 알고 경로만 계속 수정함.  
* **원리를 알면 보이는 것**:  
  * **403(Forbidden)**: 서버가 클라이언트가 누구인지는 알지만(인증 성공), 해당 자원에 접근할 \*\*권한(인가)\*\*이 없을 때 발생.  
  * **추론**: "403이니까 인증(ID/PW)은 된 거야. AWS IAM 정책(Policy)에서 S3 Write 권한이 빠져 있는지 확인해야겠네."

## **3\. 성능·튜닝 연결**

### **① DNS TTL (Time To Live) 설정**

* **연결되는 파라미터**: Route 53 레코드의 TTL 값.  
* **내부 동작**: 로컬 네임 서버가 DNS 결과를 캐시에 얼마나 들고 있을지 결정함.  
* **튜닝**:  
  * 서비스 이전(Migration) 직전에는 TTL을 짧게(예: 60초) 줄여서 IP 변경이 전 세계에 빨리 퍼지게 함.  
  * 평상시에는 300\~3600초 정도로 늘려 DNS 질의 부하를 줄이고 응답 속도를 높임.  
* **부작용**: TTL이 너무 길면 서버 장애 시 백업 서버로 IP를 바꿔도 사용자들이 한참 동안 죽은 IP로 접속하려 함.

### **② HTTP Keep-Alive (지속 연결)**

* **연결되는 파라미터**: Nginx의 keepalive\_timeout, AWS ALB의 Idle timeout.  
* **내부 동작**: 매번 3-Way Handshake를 하는 오버헤드를 줄이기 위해 한 번 맺은 TCP 연결을 유지함.  
* **튜닝**: API 요청이 잦은 서비스는 이 시간을 적절히 늘려 Latency를 줄임.  
* **부작용**: 너무 길게 잡으면 아무 일도 안 하는 소켓이 서버 자원(메모리)을 점유해 새로운 접속을 못 받을 수 있음.

## **4\. 이전/이후 강의 연결**

* **이전 강의 (\[30강. 전송 계층\])**: TCP 연결이 맺어져야 그 위에서 HTTP 메시지(텍스트)가 흐를 수 있다는 걸 배웠지. 포트 80, 443이 여기서 등장해.  
* **이후 강의 (\[32강. HTTP의 진화 및 프록시\])**: HTTP 1.1의 한계(HOL Blocking 등)를 극복하기 위한 HTTP 2/3의 특징과, 실무의 핵심인 리버스 프록시(Nginx, ALB)의 심화 동작을 배우게 될 거야.

## **5\. 바로 해볼 수 있는 실험**

### **실험 1: DNS 리졸빙 과정 추적하기 (dig 명령어)**

\# 특정 도메인의 IP를 찾기 위한 질의 과정 상세 보기  
dig google.com \+trace

* **포인트**: 루트(.)부터 TLD(.com)를 거쳐 구글 네임 서버까지 계층적으로 질의가 타고 내려가는 과정을 로그로 확인해 봐.

### **실험 2: HTTP 요청/응답 헤더 직접 확인하기 (curl)**

\# 구글에 요청을 보내고 헤더 정보만 출력  
curl \-vI \[https://www.google.com\](https://www.google.com)

* **포인트**: GET / HTTP/2, Host: www.google.com 같은 요청 라인과 HTTP/2 200, content-type, server 같은 응답 헤더를 직접 눈으로 확인해 봐. 강의에서 배운 '시작 라인'과 '필드 라인'이 보일 거야.

### **실험 3: 로컬 DNS 캐시 강제 무력화 (/etc/hosts)**

\# 리눅스나 맥에서 실행  
sudo vi /etc/hosts  
\# 맨 아래에 다음 추가: 127.0.0.1  naver.com  
\# 그 다음 브라우저에서 naver.com 접속 시도

* **포인트**: DNS 서버에 묻기 전에 OS가 로컬 파일을 먼저 본다는 '우선순위'를 이해하는 실험이야. 트러블슈팅 때 특정 도메인을 테스트 서버로 돌려보고 싶을 때 현업에서 가장 많이 쓰는 기술이야\! (실험 후 꼭 지워줘\!)

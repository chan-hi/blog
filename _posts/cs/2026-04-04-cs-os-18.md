## **1\. 핵심 개념 정리**

### **① 선점형(Preemptive) vs 비선점형(Non-preemptive) 스케줄링**

* **정의**: '선점형'은 OS가 프로세스의 CPU를 강제로 뺏을 수 있는 방식이고, '비선점형'은 프로세스가 스스로 CPU를 놓을 때까지(종료되거나 I/O 대기에 빠질 때까지) OS가 기다려주는 방식입니다.  
* **왜 존재하는가**: 하나의 무거운 프로그램이 시스템 전체를 먹통으로 만드는 것을 막고, 여러 프로그램이 동시에 응답성 있게 돌아가게 하려면 강제로 교대시키는 '선점형'이 필수적입니다.  
* **어떻게 동작하는가**: 타이머 인터럽트가 발생하여 정해진 '타임 슬라이스(Time Slice)'가 만료되면, 커널이 개입해 현재 프로세스의 문맥(Context)을 PCB에 백업하고, 준비 큐(Ready Queue)에 있던 다른 프로세스에게 CPU를 넘깁니다.  
* **개념 간 관계**: 16강에서 배운 \*\*컨텍스트 스위칭(Context Switching)\*\*이 바로 이 선점형 스케줄링이 작동할 때마다 폭발적으로 발생합니다.

### **② 입출력 집중(I/O Bound) vs CPU 집중(CPU Bound) 프로세스**

* **정의**: 파일 읽기/쓰기, 네트워크 통신 등 대기 시간이 긴 프로세스가 'I/O 집중'이고, 복잡한 연산(압축, 암호화, 렌더링)을 주로 하는 프로세스가 'CPU 집중'입니다.  
* **왜 존재하는가**: 두 프로세스의 특성이 완전히 다르기 때문에, 똑같은 시간(Time Slice)을 부여하면 비효율이 발생합니다.  
* **어떻게 동작하는가**: OS는 보통 **I/O 집중 프로세스의 우선순위를 더 높여줍니다.** 어차피 I/O 요청을 던지고 바로 대기 상태(Wait Queue)로 빠질 테니, 빨리 CPU를 주고 빨리 넘겨받아 CPU 활용률(Utilization)을 극대화하기 위함입니다. 실무의 Web Server나 DB Application은 대부분 I/O Bound 입니다.

### **③ 리눅스의 CFS (Completely Fair Scheduler)**

* **정의**: '완전히 공평하게' CPU를 나눠주는 것을 목표로 하는 현대 리눅스의 기본 스케줄러입니다. (강의 후반부에 나온 핵심 내용\!)  
* **어떻게 동작하는가**: 프로세스마다 \*\*가상 실행 시간(vruntime)\*\*을 계산합니다. CPU를 적게 쓴 프로세스는 vruntime이 낮아 다음 번에 우선적으로 CPU를 할당받습니다. 이를 빠르게 찾기 위해 '레드-블랙 트리(RB-Tree)'라는 자료구조를 사용합니다.  
* **개념 간 관계**: 우리가 K8s나 Docker에서 CPU를 제한할 때 사용하는 기술(cgroups)이 바로 이 CFS 스케줄러의 타임 슬라이스와 가중치를 제어하는 것입니다.

## **2\. 장애·트러블슈팅 연결**

### **💡 K8s CPU 스로틀링 (Throttling) 함정**

* **이 원리가 드러나는 장애 상황**:  
  Kubernetes 환경의 Spring Boot/Node.js 파드 응답이 간헐적으로 엄청나게 느려짐(레이턴시 튐). 그런데 Grafana에서 CPU 사용률 메트릭을 보면 30%밖에 안 됨.  
* **원리를 모르면 빠지는 함정**:  
  "CPU 사용률도 넉넉하고 메모리도 남는데 왜 느리지? 네트워크 문제인가?" 하며 엉뚱한 VPC나 로드밸런서 쪽만 파고듦.  
* **원리를 알면 보이는 것**:  
  "아\! 이 파드의 CPU limits를 너무 빡빡하게 줘서, 커널의 **CFS 스케줄러**가 강제로 CPU를 뺏었구나(선점형 스케줄링)\!"  
  ➔ K8s에서 limit을 설정하면 리눅스의 cfs\_quota\_us와 cfs\_period\_us가 설정됩니다. 파드가 100ms(period) 중 할당받은 타임 슬라이스(quota)를 초반 20ms 만에 다 써버리면, 남은 80ms 동안은 CFS에 의해 강제로 \*\*대기 상태(스로틀링)\*\*가 됩니다. 그래서 평균 사용률은 낮아 보여도 실제로는 CPU를 뺏겨 멈춰있던 시간이 존재했던 것입니다. (해결: Limit을 넉넉히 주거나 없애고 Request만 설정)

### **💡 아사 현상 (Starvation) 과 데몬 프로세스**

* **이 원리가 드러나는 장애 상황**:  
  리눅스 서버에 무거운 데이터 배치(Batch) 스크립트를 돌렸더니, SSH 접속조차 안 될 정도로 서버가 먹통이 됨.  
* **원리를 모르면 빠지는 함정**:  
  SSH 데몬(sshd)이 죽은 줄 알고 서버를 무작정 재부팅함.  
* **원리를 알면 보이는 것**:  
  "배치 프로세스(CPU 집중)가 자원을 독식하면서, 우선순위가 밀린 SSH나 모니터링 에이전트 프로세스가 CPU를 아예 할당받지 못하는 \*\*아사 현상(Starvation)\*\*에 빠졌구나." ➔ 배치 프로세스 실행 시 nice 값을 낮춰서(우선순위를 떨어뜨려) 실행하도록 조치.

## **3\. 성능·튜닝 연결**

강의에서 배운 우선순위와 스케줄링은 클라우드 자원 할당의 핵심 원리입니다.

* **관련 시스템 설정: 리눅스 nice / renice 명령어**  
  * **내부 동작**: 프로세스의 우선순위(NI 값, \-20 \~ 19)를 조절합니다. 값이 **낮을수록 우선순위가 높습니다** (가중치가 높아져 CFS에서 타임 슬라이스를 더 많이 줌).  
  * **튜닝**: Prometheus Node Exporter나 Fluentd 같은 로깅/모니터링 에이전트는 애플리케이션보다 중요도가 낮으므로 nice 값을 10\~15 정도로 높게 주어 백그라운드에서 조용히 돌게 만들 수 있습니다.  
* **관련 K8s 설정: requests 와 limits**  
  * **requests.cpu**: CFS 스케줄러의 \*\*'가중치(Weight/Shares)'\*\*를 결정합니다. 노드의 CPU가 부족할 때, 파드들이 CPU를 나눠 갖는 비율(우선순위)이 됩니다.  
  * **limits.cpu**: CFS 스케줄러의 \*\*'최대 타임 슬라이스 할당량(Quota)'\*\*을 결정합니다. 이를 넘으면 가차 없이 선점(Preemption) 당해 스로틀링이 걸립니다.

## **4\. 이전/이후 강의 연결**

* **선행 지식 (15\~17강)**:  
  컨텍스트 스위칭(Context Switching)의 원리와 오버헤드, 프로세스/스레드의 생명주기(대기 상태 등)를 배웠습니다.  
* **이 강의의 위치 (현재)**:  
  그 수많은 프로세스 중 "누구에게 CPU를 줄 것인가?"에 대한 운영체제의 공평하고도 효율적인 분배 법칙을 배웠습니다.  
* **후행 지식 (이후 내용)**:  
  CPU를 분배하는 문제는 끝났습니다. 다음은 "프로세스들을 띄울 **메모리(Memory) 공간**이 부족한데 이건 어떻게 분배할까?" 라는 주제로 넘어갑니다. (스와핑, 페이징, 가상 메모리 등)

## **5\. 바로 해볼 수 있는 실험**

본인의 리눅스 서버나 K8s 파드 셸에서 다음을 직접 눈으로 확인해 보세요\!

1. **내 프로세스의 우선순위(NI와 PR) 확인하기**  
   \# top 명령어 실행 후, PR(Priority)와 NI(Nice) 컬럼을 확인해보세요.  
   top

   * 시스템 프로세스는 PR이 낮고(높은 우선순위), 일반 프로세스는 보통 NI가 0입니다.  
2. **CFS 스케줄러의 vruntime 훔쳐보기**  
   \# 현재 실행 중인 내 쉘(bash)의 PID를 확인  
   echo $$

   \# 해당 프로세스가 CFS 스케줄러에서 관리되는 상세 정보(vruntime 등) 보기  
   cat /proc/$$/sched

   * vruntime, se.sum\_exec\_runtime 등 강의에서 배운 CFS 지표들이 실제 커널에 기록되고 있는 것을 볼 수 있습니다.  
3. **K8s CPU 스로틀링(선점형 스케줄링의 결과) 모니터링 쿼리 (Prometheus)**  
   \# 회사나 개인 클러스터의 Grafana/Prometheus가 있다면 이 쿼리를 날려보세요.  
   \# 내 파드가 타임슬라이스를 초과하여 CFS에 의해 강제로 '대기' 당한 시간을 의미합니다.  
   rate(container\_cpu\_cfs\_throttled\_seconds\_total{pod="내-파드-이름"}\[5m\])

   * 이 수치가 계속 올라간다면, 프로세스가 선점 당해 멈춰있는 시간이 길다는 뜻이며, 최우선적으로 튜닝(limits 완화)이 필요한 상태입니다.

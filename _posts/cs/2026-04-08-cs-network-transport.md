## **1\. 핵심 개념 정리**

### **① 포트 (Port)**

* **정의**: 한 호스트(IP) 내에서 실행 중인 특정 프로세스를 식별하기 위한 16비트 논리적 주소.  
* **왜 존재하는가**: IP 주소만으로는 목적지 "컴퓨터"까지는 갈 수 있지만, 그 안에서 실행되는 수많은 앱(웹 서버, DB, SSH 등) 중 누구에게 데이터를 줘야 할지 알 수 없기 때문이야.  
* **어떻게 동작하는가**: 0\~65535번까지의 번호를 사용하며, 패킷 헤더에 송/수신지 포트 번호를 기록해. OS는 이 번호를 보고 해당 소켓(Socket)을 점유한 프로세스에게 데이터를 전달해.  
* **개념 간 관계**: IP 주소 : 포트 번호의 조합을 \*\*소켓(Socket)\*\*이라고 불러. L3(IP)가 건물을 찾는 주소라면, 포트는 그 건물의 호실(Room Number)이야.

### **② NAPT (Network Address Port Translation)**

* **정의**: 하나의 공인 IP 주소를 여러 개의 사설 IP 주소가 공유할 수 있도록, 포트 번호를 이용해 변환하는 기술.  
* **왜 존재하는가**: 공인 IPv4 주소는 유한해. 사내 수백 대의 PC가 각각 공인 IP를 가질 수 없으니, 공유기(NAT 장비)가 포트 번호를 장부처럼 기록해서 트래픽을 중개하는 거야.  
* **어떻게 동작하는가**: 내부 IP 192.168.0.5:5000이 밖으로 나갈 때, NAT 장비가 공인 IP 1.2.3.4:12345로 바꿔서 보내고 그 매핑 정보를 테이블에 기록해둬. 응답이 오면 다시 역변환해주는 식이지.  
* **개념 간 관계**: 클라우드(AWS)의 **NAT Gateway**가 바로 이 동작을 수행하는 핵심 컴포넌트야.

### **③ TCP vs UDP**

* **TCP (Transmission Control Protocol)**: 연결 지향적이며, **신뢰성**을 보장해. 3-way handshake로 연결을 맺고, 데이터가 잘 갔는지 확인(ACK)하며, 흐름/혼잡 제어를 수행해. (예: 웹 브라우징, 파일 전송)  
* **UDP (User Datagram Protocol)**: 비연결형이며, 신뢰성보다는 **속도와 실시간성**에 집중해. "일단 던지고 본다"는 방식이라 오버헤드가 매우 적어. (예: 스트리밍, 게임, DNS 조회)

## **2\. 장애·트러블슈팅 연결**

### **🚨 시나리오 A: 애플리케이션 먹통과 CLOSE\_WAIT 누수**

* **장애 상황**: 서버의 CPU/메모리는 정상인데 새로운 접속이 안 됨. netstat \-ant | grep CLOSE\_WAIT 명령어를 쳐보니 수천 개의 소켓이 해당 상태로 멈춰 있음.  
* **원리를 모르면 빠지는 함정**: "네트워크 장비가 이상한가?" 하고 인프라 탓을 하거나 서버를 재시작함. 하지만 재시작해도 곧 다시 장애가 발생함.  
* **원리를 알면 보이는 것**: CLOSE\_WAIT은 상대방이 FIN을 보내서 연결 종료를 요청했는데, **우리 쪽 애플리케이션이 close() 함수를 호출하지 않아서** 생기는 상태야. 즉, 이건 100% 애플리케이션 코드 버그(소켓 자원 미반환)라는 걸 확신하고 개발팀에 수정을 요청할 수 있어.

### **🚨 시나리오 B: AWS NAT Gateway 포트 고갈 (Port Exhaustion)**

* **장애 상황**: EKS 클러스터에서 외부 API 서버로 요청을 보낼 때 Connection Timeout이 간헐적으로 발생함.  
* **원리를 모르면 빠지는 함정**: 외부 API 서버의 성능 문제라고 생각해서 그쪽 팀에 항의함.  
* **원리를 알면 보이는 것**: NAPT 원리에 따라 NAT Gateway가 가질 수 있는 포트(Ephemeral Port)는 약 6만 개로 제한되어 있어. 특정 시간에 아웃바운드 요청이 폭증하면 **포트가 고갈**되어 연결을 못 맺는 거지. CloudWatch의 ErrorPortAllocationError 메트릭을 보고 NAT Gateway를 증설하거나 소켓 재사용 설정을 검토하게 됨.

## **3\. 성능·튜닝 연결**

### **① TCP TIME\_WAIT 재사용 (tcp\_tw\_reuse)**

* **연결되는 파라미터**: sysctl \-w net.ipv4.tcp\_tw\_reuse=1  
* **내부 동작**: 연결 종료 후 TIME\_WAIT 상태로 남은 포트를 새로운 연결에 즉시 재사용할 수 있게 해줘.  
* **부작용**: 패킷이 꼬일 가능성이 미세하게 있지만, 수만 개의 짧은 연결이 발생하는 고부하 API 서버에서는 포트 고갈 방지를 위해 필수적인 튜닝이야.

### **② 흐름 제어(Flow Control)와 윈도우 사이즈**

* **연결되는 파라미터**: net.ipv4.tcp\_window\_scaling  
* **내부 동작**: TCP 헤더의 Window Size 필드를 확장해서, 한 번에 확인할(ACK) 필요 없이 보낼 수 있는 데이터 양을 늘려줘.  
* **튜닝 효과**: 네트워크 대역폭은 넓은데 지연 시간(Latency)이 긴 환경(예: 해외 리전 간 통신)에서 처리량을 극대화할 수 있어.

## **4\. 이전/이후 강의 연결**

* **이전 강의 (\[29강. IP 주소\])**: IP 주소가 "어느 서버"로 갈지 정했다면, 이제 그 서버 안의 "어느 방(프로세스)"으로 갈지 포트가 결정함.  
* **이후 강의 (\[31강. 응용 계층\])**: TCP/UDP 위에서 동작하는 실제 서비스 프로토콜(HTTP, DNS, gRPC 등)의 실체를 배우게 됨. 포트 번호 80(HTTP)이나 443(HTTPS)이 왜 그렇게 정해졌는지 이해하게 될 거야.

## **5\. 바로 해볼 수 있는 실험**

### **실험 1: 현재 서버의 포트 점유 현황 확인 (ss 명령어)**

\# 리눅스 서버에서 실행 (포트 80을 어느 프로세스가 쓰는지 확인)  
sudo ss \-lntp | grep :80

* **포인트**: LISTEN 상태인 포트와 이를 점유한 Process ID(PID)를 매핑해서 확인해봐. K8s 환경이라면 Ingress Controller 프로세스가 잡고 있을 거야.

### **실험 2: TCP 연결 상태 변화 관찰**

\# 실시간으로 소켓 상태 변화 모니터링  
watch \-n 1 "ss \-s"

* **포인트**: 새로운 HTTP 요청을 보낼 때 ESTAB 숫자가 늘어나는지, 종료 후 TIME\_WAIT이 발생하는지 실시간 메트릭으로 느껴봐.

### **실험 3: Ephemeral Port 범위 확인**

cat /proc/sys/net/ipv4/ip\_local\_port\_range

* **포인트**: 클라이언트로서 나갈 때 사용할 수 있는 포트의 시작과 끝 번호야. 이 범위가 좁으면 동시 접속이 많은 서버에서 장애가 날 수 있어.

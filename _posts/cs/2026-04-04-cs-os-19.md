## **1\. 핵심 개념 정리**

### **① 물리 주소와 논리 주소 (그리고 MMU)**

* **정의**: '물리 주소'는 실제 RAM 하드웨어의 주소이고, '논리 주소'는 각 프로세스(CPU)가 바라보는 0번지부터 시작하는 자기만의 주소입니다.  
* **왜 존재하는가**: 수많은 프로세스가 물리 메모리 하나를 공유하는데, 프로세스마다 자신이 RAM의 어느 위치에 적재될지 미리 알 수 없습니다.  
* **어떻게 동작하는가**: CPU는 '논리 주소'로 명령을 내리고, 중간에 있는 \*\*MMU(Memory Management Unit)\*\*라는 하드웨어가 이를 실제 '물리 주소'로 찰나의 순간에 변환해 줍니다.  
* **개념 간 관계**: Docker 컨테이너 안에서 내 앱이 메모리를 독점하는 것처럼 보이지만, 사실은 OS와 MMU가 만들어낸 '논리적 환상' 속에서 돌고 있는 것입니다.

### **② 가상 메모리와 페이징 (Paging)**

* **정의**: 프로세스를 \*\*페이지(Page)\*\*라는 일정한 크기(보통 4KB)로 잘게 쪼개고, 물리 메모리도 \*\*프레임(Frame)\*\*이라는 같은 크기로 쪼개어, 필요한 조각만 불연속적으로 메모리에 적재하는 기술입니다.  
* **왜 존재하는가**: RAM이 8GB인데 16GB짜리 프로그램을 돌리고 싶거나, 빈 공간은 많은데 연속된 큰 공간이 없어서(외부 단편화) 앱을 띄우지 못하는 문제를 해결하기 위해 등장했습니다.  
* **어떻게 동작하는가**: 프로세스의 전체 페이지 중 **당장 실행에 필요한 일부 페이지만 RAM에 올리고(요구 페이징)**, 나머지는 디스크(보조기억장치)에 둡니다. CPU는 \*\*페이지 테이블(Page Table)\*\*을 보고 조각난 페이지들을 찾아갑니다.

### **③ 페이지 폴트 (Page Fault)와 스와핑 (Swapping)**

* **정의**: CPU가 특정 페이지를 찾았는데 RAM에 없을 때 발생하는 예외가 '페이지 폴트'이며, 디스크의 **수압(Swap) 영역**과 RAM 사이에서 페이지를 교환하는 과정이 '스와핑'입니다.  
* **왜 존재하는가**: RAM 용량은 한정되어 있으므로, 안 쓰는 데이터는 디스크로 내쫓고(Swap Out), 필요한 데이터는 RAM으로 불러와야(Swap In) 하기 때문입니다.  
* **마이너 vs 메이저**:  
  * **마이너 폴트**: 디스크까지 갈 필요 없이 메모리 어딘가에 있는 걸 페이지 테이블에 연결만 하면 됨. (빠름)  
  * **메이저 폴트**: 진짜 RAM에 없어서 디스크(스왑 영역 등)에서 읽어와야 함. (매우 느림, I/O 대기 발생)

## **2\. 장애·트러블슈팅 연결**

### **💡 Kubernetes에서 swapoff \-a를 치는 이유 (Swap의 함정)**

* **이 원리가 드러나는 장애 상황**:  
  EC2나 온프레미스 리눅스 서버가 갑자기 응답을 안 하고 SSH 접속도 안 됩니다. 디스크 I/O(Wait)는 100%를 치고 있습니다.  
* **원리를 모르면 빠지는 함정**:  
  단순히 트래픽이 많아서 느린 줄 알고 무작정 기다리거나, 앱에 무한 루프 버그가 생긴 줄 압니다.  
* **원리를 알면 보이는 것 (Thrashing)**:  
  "메모리(RAM)가 꽉 차서 커널이 미친 듯이 \*\*페이지 교체(Swap In/Out)\*\*만 하고 있구나\! CPU가 연산은 못하고 디스크에서 페이지를 가져오는 **메이저 페이지 폴트(Major Page Fault)** 처리하느라 멈춰버렸네."  
  ➔ K8s는 파드의 메모리 제한(Limit)을 엄격히 통제해야 하는데, OS가 몰래 디스크(Swap)를 써버리면 성능(레이턴시) 예측이 완전히 박살 납니다. 그래서 K8s 환경에서는 성능 격하를 막고 차라리 깔끔하게 OOM으로 죽게 만들기 위해 전통적으로 Swap을 꺼버리는 것입니다.

### **💡 VIRT vs RES 메모리의 오해 (Java/Spring 트러블슈팅)**

* **이 원리가 드러나는 장애 상황**:  
  top 명령어를 쳤더니 Java 프로세스의 VIRT(가상 메모리)가 10GB로 나옵니다. "우리 서버 RAM이 8GB인데 메모리 누수인가요?\!"라며 주니어가 패닉에 빠짐.  
* **원리를 모르면 빠지는 함정**:  
  VIRT 수치만 보고 장애로 오인하여 JVM의 Heap 사이즈를 비정상적으로 줄여버림.  
* **원리를 알면 보이는 것**:  
  "VIRT는 프로세스가 커널에 '나 이만큼 쓸래\!' 하고 예약해 둔 **논리 주소(가상 메모리) 공간의 총합**일 뿐이구나. 실제로 물리 메모리(RAM)의 프레임에 매핑된 진짜 사용량은 \*\*RES(Resident Set Size)\*\*를 봐야 해\!" ➔ JVM은 시작할 때 최대 힙(-Xmx) 크기만큼 미리 가상 메모리 공간(VIRT)을 잡아두기 때문에 VIRT가 큰 것은 정상임을 인지.

## **3\. 성능·튜닝 연결**

가상 메모리와 스와핑의 원리는 리눅스 서버 성능 튜닝의 핵심 파라미터들과 이어집니다.

* **관련 시스템 설정: vm.swappiness**  
  * **설명**: 리눅스 커널이 물리 메모리가 부족해질 때, \*\*"캐시를 지울래? 아니면 안 쓰는 프로세스 메모리를 스왑(디스크)으로 내쫓을래(Page Out)?"\*\*를 결정하는 비율입니다. (0 \~ 100\)  
  * **내부 동작**: 기본값은 보통 60입니다. 즉, RAM이 꽤 남아있어도 안 쓰는 프로세스가 있으면 적극적으로 디스크로 쫓아냅니다.  
  * **튜닝**: 데이터베이스(MySQL, Redis) 서버나 K8s 워커 노드는 디스크로 Swap Out 되는 순간 성능이 수백 배 느려집니다(메이저 페이지 폴트 발생). 따라서 이런 서버들은 sysctl \-w vm.swappiness=1 (또는 0)으로 설정하여, \*\*"진짜 RAM이 터지기 직전까지는 절대 스왑하지 마라"\*\*라고 튜닝하는 것이 실무의 기본 정석입니다.  
* **관련 시스템 설정: LRU (Least Recently Used) 캐시**  
  * 강의에서 배운 '가장 오랫동안 사용되지 않은 페이지를 내쫓는' LRU 알고리즘은 가상 메모리뿐만 아니라 실무의 **Redis의 maxmemory-policy, CDN의 캐시 축출 원리**와 100% 동일하게 동작합니다.

## **4\. 이전/이후 강의 연결**

* **선행 지식 (16강)**:  
  프로세스의 구조(Code, Data, Heap, Stack)를 배웠습니다. 이제 이 4가지 영역이 사실 4KB짜리 '페이지' 단위로 조각조각 나뉘어 RAM 여기저기에 불연속적으로 흩뿌려져(페이징) 있다는 것을 알게 되었습니다.  
* **이 강의의 위치 (현재)**:  
  한정된 물리 메모리를 여러 프로세스가 안전하고 효율적으로 나눠 쓰는 '가상 메모리'의 마법을 배웠습니다.  
* **후행 지식 (이후 내용)**:  
  이제 메모리와 CPU의 이야기는 끝났습니다. 다음 강의부터는 보조기억장치(디스크)와 파일들이 어떻게 물리적으로, 논리적으로 관리되는지 **'파일 시스템(File System)'** 구조로 들어가게 됩니다.

## **5\. 바로 해볼 수 있는 실험**

본인의 리눅스 서버나 컨테이너에서 아래 명령어들을 통해 가상 메모리 동작을 눈으로 확인해 보세요.

1. **top 명령어로 가상/물리 메모리 분리해서 보기**  
   top

   * 프로세스 목록에서 VIRT (Virtual Image, 가상 메모리)와 RES (Resident size, 실제 물리 메모리에 올라간 크기 \= 프레임에 적재된 크기)를 비교해 보세요. 보통 VIRT가 훨씬 큽니다.  
2. **메이저 페이지 폴트(Major Page Fault) 발생 횟수 모니터링하기**  
   \# 시스템 전체의 페이지 인(pi, 디스크에서 읽어옴)과 페이지 아웃(po, 디스크로 씀)을 1초마다 보여줍니다.  
   vmstat 1

   * 평소에는 pi와 po 컬럼이 0입니다. 이 수치가 치솟는다면 서버가 Thrashing(잦은 스와핑으로 인한 성능 저하) 상태에 빠진 것입니다.  
3. **내 서버의 스와피니스(swappiness) 값 확인하기**  
   cat /proc/sys/vm/swappiness

   * K8s 노드라면 0\~1이어야 하고, 일반 리눅스 서버라면 60일 것입니다. 실무에서 이 값을 용도에 맞게 튜닝하는 것이 매우 중요합니다.

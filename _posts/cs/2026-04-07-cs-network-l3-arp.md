## **1\. 핵심 개념 정리**

### **① IP 주소 (Internet Protocol Address)**

* **정의**: 호스트를 식별하기 위해 부여된 논리적 주소. (IPv4는 32bit, IPv6는 128bit)  
* **왜 존재하는가**: MAC 주소(L2)는 전 세계 기기마다 무작위로 붙어있어서, 수조 개의 기기 중 하나를 찾기 위해 모든 기기에 물어볼 수(브로드캐스트) 없기 때문이야. '구(District)'와 '번지(Number)'처럼 **계층적 구조**를 가진 주소가 있어야 목적지까지 효율적으로 패킷을 전달할 수 있어.  
* **어떻게 동작하는가**: 네트워크 주소(어디 사는지)와 호스트 주소(누구인지)를 서브넷 마스크를 통해 구분해. 라우터는 패킷의 목적지 IP를 보고 자신의 '라우팅 테이블'과 대조해 다음 길(Next Hop)을 안내해.  
* **개념 간 관계**: L4의 포트 번호와 결합하면 '소켓(Socket)'이 되고, L2의 MAC 주소와 결합해야 비로소 전송 가능한 '프레임'이 완성돼.

### **② ARP (Address Resolution Protocol)**

* **정의**: 논리적 주소(IP)를 물리적 주소(MAC)로 변환해 주는 프로토콜.  
* **왜 존재하는가**: 캡슐화 원리에 따라 L2 프레임을 만들려면 목적지의 MAC 주소가 반드시 필요해. 하지만 우리는 보통 상대방의 IP 주소만 알고 있지. 이때 "이 IP 주소 가진 사람 MAC 주소 좀 알려줘\!"라고 물어보는 과정이 필요해.  
* **어떻게 동작하는가**:  
  1. **ARP Request**: 네트워크에 브로드캐스트를 쏴서 물어봄.  
  2. **ARP Reply**: 해당 IP를 가진 호스트가 자신의 MAC 주소를 유니캐스트로 답변함.  
  3. **ARP Cache**: 얻어낸 정보를 메모리에 일정 시간 저장해 재사용함.  
* **개념 간 관계**: L3(네트워크)와 L2(데이터 링크)를 잇는 징검다리 역할을 수행해.

## **2\. 장애·트러블슈팅 연결**

### **🚨 시나리오 A: "Destination Host Unreachable" vs "Request Timeout"**

* **이 원리가 드러나는 장애 상황**: 서버 A에서 B로 핑을 보낼 때 발생하는 에러 메시지의 차이.  
  * Destination Host Unreachable: 내 라우팅 테이블에 갈 길이 없거나, **ARP 응답이 없어서** L2 프레임을 아예 만들지 못할 때 발생.  
  * Request Timeout: 패킷은 보냈는데 돌아오지 않을 때(상대방 방화벽 차단 등) 발생.  
* **원리를 모르면 빠지는 함정**: 무조건 "상대방 서버가 죽었나?" 혹은 "방화벽 문제인가?"라고만 생각함. 실은 내 서버의 라우팅 설정이 빠졌거나 ARP 캐시가 꼬였을 수 있음.  
* **원리를 알면 보이는 것**: "Unreachable이 뜬다는 건 내 랜카드에서 패킷이 나가지도 못했다는 뜻이네? ip neigh로 ARP 캐시를 확인하거나 ip route로 경로가 있는지 먼저 봐야겠다."

### **🚨 시나리오 B: 클라우드 VPC 내의 IP 충돌 (IP Conflict)**

* **이 원리가 드러나는 장애 상황**: 드물지만 온프레미스에서 IP를 수동 할당하거나, K8s의 특정 CNI 버그로 인해 두 호스트가 같은 IP를 갖게 됨. 이때 통신이 됐다 안 됐다 하는 '간헐적 장애'가 발생함.  
* **원리를 모르면 빠지는 함정**: "네트워크가 좀 불안정한가?" 하고 대수롭지 않게 넘기거나 스위치 교체를 고민함.  
* **원리를 알면 보이는 것**: "IP 충돌이 나면 스위치의 MAC 테이블과 호스트의 ARP 캐시가 계속 요동치겠네(Flapping). arping 명령어를 써서 같은 IP에 대해 두 개의 MAC 주소가 응답하는지 확인해 봐야지."

## **3\. 성능·튜닝 연결**

* **ARP 캐시 크기 및 타임아웃 튜닝**  
  * **연결되는 원리**: 서버가 통신하는 대상이 수천 대(예: 대규모 마이크로서비스 환경)가 되면 ARP 캐시 테이블이 가득 참.  
  * **설정값**: sysctl \-a | grep gc\_thresh  
    * net.ipv4.neigh.default.gc\_thresh1, 2, 3: ARP 캐시 보관 개수 임계값.  
  * **튜닝**: 수만 개의 컨테이너가 통신하는 노드에서는 이 값을 늘려주지 않으면, ARP 캐시를 정리(GC)하느라 CPU를 과다 소모하거나 새로운 ARP 응답을 처리하지 못해 통신 지연이 발생함.  
* **Reverse Path Filtering (rp\_filter)**  
  * **연결되는 원리**: 들어온 패킷의 소스 IP를 보고, "내가 이 IP로 패킷을 보낼 때 나가는 인터페이스로 들어온 게 맞나?"를 확인하는 보안 기능.  
  * **상황**: 비대칭 라우팅(Asymmetric Routing) 환경이나 멀티 홈(NIC 2개) 서버에서 패킷이 조용히 드랍될 때 이 커널 파라미터를 튜닝함.

## **4\. 이전/이후 강의 연결**

* **선행 지식 (\[28강. L1/L2 계층\])**:  
  * 스위치가 MAC 주소를 학습하는 원리를 알아야, ARP가 왜 필요한지 이해할 수 있어. "MAC은 이웃집을 찾기 위한 주소고, IP는 먼 도시의 집을 찾기 위한 주소"라는 비유를 떠올려봐.  
* **이후 연결 (\[30강. 전송 계층\])**:  
  * IP가 목적지 '컴퓨터'까지는 배달해 주지만, 그 컴퓨터 안의 어떤 '프로그램'에게 줘야 할지는 몰라. 그 역할을 담당하는 포트(Port)와 TCP/UDP의 세계로 넘어가게 돼.

## **5\. 바로 해볼 수 있는 실험**

**실험 1: ARP 요청 과정 패킷 캡처하기**

\# 1\. ARP 캐시 삭제 (특정 IP)  
sudo ip neigh flush 192.168.1.10  
\# 2\. tcpdump로 ARP 패킷만 필터링해서 보기  
sudo tcpdump \-i any arp  
\# 3\. 다른 터미널에서 해당 IP로 핑 보내기  
ping \-c 1 192.168.1.10

* **포인트**: Who has 192.168.1.10? Tell 192.168.1.5 같은 브로드캐스트 요청과 그에 대한 응답을 직접 눈으로 확인해 봐.

**실험 2: 라우팅 테이블 분석하기**

ip route show

* **포인트**: default via ...라고 된 부분이 '디폴트 게이트웨이'야. 내 서브넷 밖의 IP로 가는 모든 패킷은 일단 이 게이트웨이(라우터)의 MAC 주소를 ARP로 알아낸 뒤 그쪽으로 던져진다는 걸 이해하는 게 중요해.

**실험 3: arping으로 중복 IP 찾기**

\# 특정 IP에 대해 L2 수준에서 응답 확인  
sudo arping \-I eth0 192.168.1.10

* **포인트**: 일반 ping은 상대방이 방화벽으로 ICMP를 막으면 대답하지 않지만, arping은 L2 수준의 응답이라 막기 힘들고, 만약 응답하는 MAC 주소가 두 개라면 100% IP 충돌이야.

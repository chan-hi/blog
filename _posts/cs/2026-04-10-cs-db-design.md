## **1\. 핵심 개념 정리**

### **① ERD (Entity-Relationship Diagram)**

* **정의**: 데이터베이스에 저장할 엔티티(객체)와 그들 간의 관계를 그림으로 나타낸 설계도야.  
* **왜 존재하는가**: 집을 짓기 전에 설계도를 그리듯, 실제 구축(CREATE TABLE) 전에 데이터 구조의 모순을 발견하고 협업하기 위해 존재해.  
* **어떻게 동작하는가**:  
  1. 엔티티(사각형), 속성(타원), 관계(마름모)를 식별해.  
  2. 관계의 기수성(1:1, 1:N, N:M)을 선으로 연결하여 데이터의 흐름을 정의해.  
* **개념 간 관계**: ERD를 기반으로 실제 물리적인 \*\*테이블(Table)\*\*과 **외래키(FK)** 제약조건이 생성돼.

### **② 이상 현상 (Anomaly)**

* **정의**: 잘못된 설계로 인해 데이터를 삽입, 삭제, 수정할 때 발생하는 논리적 오류나 데이터 불일치 현상이야.  
* **왜 발생하는가**: 하나의 테이블에 너무 많은 주제의 데이터가 중복되어 뭉쳐 있을 때 발생해.  
* **어떻게 동작하는가**:  
  * **삽입 이상**: 원치 않는 정보까지 삽입해야 하거나, 정보가 부족해 삽입이 불가능한 경우.  
  * **삭제 이상**: 특정 정보를 삭제할 때 연관된 다른 중요한 정보까지 같이 삭제되는 경우.  
  * **수정 이상**: 중복된 데이터 중 일부만 수정되어 데이터 간 불일치가 발생하는 경우.

### **③ 정규화 (Normalization)**

* **정의**: 이상 현상을 제거하기 위해 중복을 최소화하고 테이블을 분해하는 과정이야.  
* **왜 존재하는가**: 데이터의 무결성(Integrity)을 보장하고 저장 공간을 효율적으로 사용하기 위해 존재해.  
* **어떻게 동작하는가**:  
  * **제1정규형(1NF)**: 모든 필드 값이 원자값(Atomic value)이어야 함.  
  * **제2정규형(2NF)**: 부분 함수 종속 제거 (기본키의 일부분에만 종속되는 속성 분리).  
  * **제3정규형(3NF)**: 이행적 함수 종속 제거 (기본키가 아닌 속성 간의 종속성 분리).  
* **개념 간 관계**: 정규화가 진행될수록 테이블은 쪼개지고, 데이터를 합쳐서 보기 위해 **JOIN** 연산이 더 많이 필요하게 돼.

## **2\. 장애·트러블슈팅 연결**

### **🚨 트러블슈팅 시나리오 1: "수정 이상"으로 인한 데이터 부정합 장애**

* **장애 상황**: 고객이 주소를 변경했는데, 어떤 주문 내역에는 옛날 주소가 있고 어떤 곳엔 새 주소가 있음. 이로 인해 배송 사고 발생.  
* **주니어의 함정**: "백엔드 업데이트 로직(UPDATE 문)에 버그가 있나 봐요. 코드를 다시 짭시다\!"  
* **원리를 알면 보이는 것**: 주문 테이블에 고객 주소 정보가 직접 들어가 있는 비정규화된 구조가 문제야. UPDATE를 할 때 모든 행을 다 바꾸지 못하면 데이터가 꼬일 수밖에 없어. 이건 코드 수정이 아니라 테이블을 \*\*'고객'과 '주문'으로 정규화(분리)\*\*해야 근본적으로 해결되는 장애야.

### **🚨 트러블슈팅 시나리오 2: "삭제 이상"으로 인한 중요 데이터 소실**

* **장애 상황**: 특정 강의를 수강 취소했더니, 해당 강의 정보(교수명, 강의실 등) 자체가 DB에서 사라짐.  
* **주니어의 함정**: "백업본에서 강의 정보를 다시 복구해야겠어요."  
* **원리를 알면 보이는 것**: '수강 신청' 테이블에 '강의 정보'가 포함되어 설계되었기 때문에 발생하는 삭제 이상이야. 엔티티 간의 관계를 잘못 설정한 것이지. 인프라 백업으로 해결할 게 아니라 설계를 분리해야 해.

## **3\. 성능·튜닝 연결**

### **① 정규화와 JOIN의 성능 트레이드 오프**

* **설명**: 정규화는 데이터 무결성에는 좋지만, 조회 시 여러 테이블을 합쳐야 하는 **JOIN 비용**을 발생시켜.  
* **튜닝 (역정규화/Denormalization)**: 만약 특정 서비스에서 JOIN이 너무 많아 디스크 I/O와 CPU가 버티지 못한다면, 의도적으로 정규화를 깨고 데이터를 중복시키는 **역정규화**를 고려해야 해.  
* **실무 팁**: AWS RDS 성능 분석(Performance Insights)에서 Join\_buffer\_size 부족이나 Table\_open\_cache 문제가 빈번하다면 설계가 너무 파편화되어 있거나 JOIN 쿼리가 비효율적인지 점검해봐야 해.

### **② 스토리지 용량과 중복 데이터**

* **설명**: 비정규화된 테이블은 중복 데이터 때문에 스토리지(EBS)를 훨씬 많이 차지해.  
* **인프라 영향**: 정규화를 잘하면 데이터 용량이 줄어들고, 결과적으로 백업 시간 단축과 스냅샷 비용 절감으로 이어지지.

## **4\. 이전/이후 강의 연결**

* **선행 지식 (35/36강)**:  
  * **기본키(PK)와 외래키(FK)**: 정규화를 위해 테이블을 쪼갤 때, 이를 다시 잇기 위한 PK와 FK의 개념이 필수야.  
* **후속 지식 (이후 강의)**:  
  * **NoSQL (39강)**: 오늘 배운 정규화 원칙을 정면으로 거스르는(의도적 역정규화) NoSQL의 철학을 이해하는 밑거름이 돼. 대규모 분산 환경에서는 왜 정규화를 포기하는지 배우게 될 거야.

## **5\. 바로 해볼 수 있는 실험**

1. **이상 현상 직접 재현하기**  
   * 중복이 많은 테이블을 하나 만들고 데이터를 넣어봐.

   CREATE TABLE student\_course (  
         std\_id INT,  
         std\_name VARCHAR(10),  
         course\_name VARCHAR(20),  
         professor VARCHAR(10)  
     );

   * 한 학생이 강의를 두 개 들으면 학생 이름이 두 번 중복되지? 이때 이름을 한 곳만 수정(UPDATE)해보고 SELECT 결과가 어떻게 꼬이는지 확인해봐. 이게 바로 **수정 이상**이야.  
2. **테이블 분리(정규화) 연습**  
   * 위의 테이블을 student(id, name)와 course(id, name, professor), 그리고 enrollment(std\_id, course\_id)로 쪼개봐.  
   * 데이터를 수정할 때 student 테이블 한 곳만 바꾸면 모든 주문 내역에서 바뀐 이름이 조회되는(JOIN 활용) 무결성을 경험할 수 있어.  
3. **스키마 설계 도구 활용 (ERD 실습)**  
   * dbdiagram.io 같은 무료 툴을 써서 오늘 배운 엔티티 간의 관계를 시각적으로 그려봐. 인프라 아키텍처 다이어그램을 그리듯이 데이터의 구조를 시각화하는 습관을 들이는 게 좋아.

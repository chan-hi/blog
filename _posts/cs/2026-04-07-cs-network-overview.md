## **1\. 핵심 개념 정리**

### **① 패킷과 캡슐화/역캡슐화 (Packet & Encapsulation/Decapsulation)**

* **정의**: 데이터를 '패킷'이라는 작은 단위로 쪼개고, 각 계층(Layer)을 지날 때마다 프로토콜의 목적에 맞는 메타데이터(헤더/트레일러)를 덧붙이거나(캡슐화) 떼어내는(역캡슐화) 과정.  
* **왜 존재하는가**: 통매체(케이블, 무선)는 한 번에 보낼 수 있는 데이터 크기가 제한되어 있고, 목적지까지 안전하게 가려면 '주소표'와 '취급 주의사항'이 필요하기 때문.  
* **어떻게 동작하는가**: 송신자는 L7(응용) \-\> L1(물리)로 내려가며 헤더를 씌우고(Data \-\> Segment \-\> Packet \-\> Frame), 수신자는 L1 \-\> L7로 올라가며 헤더를 까보면서 확인하고 버림.  
* **개념 간 관계**: 프로토콜(Protocol)과 찰떡궁합. 각 계층의 프로토콜이 자신의 '목적과 특징'을 달성하기 위해 헤더에 특정 필드(IP 주소, 포트 번호 등)를 기록함.

### **② 전송 방식과 브로드캐스트 도메인 (Broadcast Domain)**

* **정의**: 유니캐스트(1:1), 브로드캐스트(1:All) 등 데이터를 전달하는 방식. 특히 브로드캐스트 도메인은 브로드캐스트 패킷이 도달할 수 있는 물리적/논리적 범위를 뜻함(통상적으로 같은 LAN, 같은 서브넷).  
* **왜 존재하는가**: 수신자의 정확한 주소를 모를 때(예: "IP 192.168.1.10 가진 애 누구야? 맥주소 내놔\!" \- ARP) 네트워크 전체에 물어봐야 하거나, 반대로 불필요한 트래픽이 전체 망을 마비시키는 것을 막기 위해 구역을 나눠야 하기 때문.  
* **어떻게 동작하는가**: 호스트가 목적지 주소를 브로드캐스트 주소(예: 255.255.255.255 또는 FF:FF:FF:FF:FF:FF)로 세팅해 던지면, 중간 노드(스위치)가 자신과 연결된 모든 포트로 이를 복제해서 뿌림.  
* **개념 간 관계**: 서브넷팅(Subnetting), VPC 설계와 직결됨. 라우터를 거치면 브로드캐스트는 차단되므로, 네트워크를 분리한다는 것은 곧 브로드캐스트 도메인을 나눈다는 의미.

## **2\. 장애·트러블슈팅 연결**

### **🚨 시나리오 A: K8s 오버레이 네트워크 패킷 드랍 (캡슐화 원리)**

* **이 원리가 드러나는 장애 상황**: EKS나 NKS에서 특정 Pod 간 통신이 간헐적으로 끊기거나, 대용량 파일 전송 시에만 Connection Timeout이 발생함. 로그에는 특별한 에러가 없고, tcpdump를 떠보면 \[S.\], \[S\] 플래그(SYN)는 가는데 실제 데이터 페이로드를 담은 패킷이 조용히 사라짐(Blackhole).  
* **원리를 모르면 빠지는 함정**: 주니어들은 흔히 "앱 문제인가?" 하고 애플리케이션 Timeout 설정만 무작정 늘리거나 팟을 재시작함. 하지만 원인은 **캡슐화 오버헤드**임.  
* **원리를 알면 보이는 것**: "아, K8s CNI(Flannel, Cilium 등)는 VXLAN/Geneve 등을 쓰니까 원본 패킷에 **L2/L3 헤더를 한 번 더 씌우지(캡슐화)**\! 그러면 원래 네트워크의 MTU(Max Transmission Unit, 1500바이트)를 초과해서 패킷이 쪼개지거나(Fragmentation) 버려질 수 있겠구나." \-\> CNI의 MTU를 노드 MTU보다 50바이트 작게(예: 1450\) 튜닝해야 함을 깨달음.

### **🚨 시나리오 B: 클라우드 환경의 브로드캐스트 특성 (브로드캐스트 원리)**

* **이 원리가 드러나는 장애 상황**: 온프레미스에서 잘 쓰던 L2 브로드캐스트 기반의 HA 솔루션(예: Keepalived의 VRRP, 특정 클러스터링 솔루션)을 AWS EC2에 그대로 올렸는데, 서로 Heartbeat를 못 찾고 두 서버 모두 자기가 마스터라고 우기는 Split-Brain 발생.  
* **원리를 모르면 빠지는 함정**: "Security Group에서 모든 포트를 열어줬는데 왜 통신이 안 되지?" 하며 방화벽만 계속 쳐다봄.  
* **원리를 알면 보이는 것**: "퍼블릭 클라우드(AWS, NCP)의 VPC는 전통적인 L2 스위치가 아니라 SDN(소프트웨어 정의 네트워크)으로 구현되어 있어서, 성능 및 보안을 위해 **네트워크 레벨의 브로드캐스트/멀티캐스트를 차단**하는구나." \-\> 유니캐스트 기반의 HA 구성으로 변경하거나 클라우드 네이티브 로드밸런서를 써야 한다는 결론 도출.

## **3\. 성능·튜닝 연결**

* **MTU (Maximum Transmission Unit) 튜닝과 점보 프레임(Jumbo Frame)**  
  * **연결되는 설정값**: 리눅스 인터페이스의 MTU 설정 (ifconfig eth0 mtu 9000 또는 ip link set dev eth0 mtu 9000), AWS VPC의 점보 프레임 지원.  
  * **내부 동작 원리**: 캡슐화/역캡슐화는 CPU 연산을 소모함. 대용량 데이터를 1500바이트 크기의 패킷 수만 개로 쪼개서 각각 헤더를 붙여(캡슐화) 보내면 CPU 사용률이 치솟음. MTU를 9000으로 늘리면 패킷 개수가 줄어들어 **캡슐화 오버헤드가 급감**하고 네트워크 처리량이 크게 향상됨.  
  * **부작용**: 중간 노드(스위치, 라우터) 중 하나라도 9000을 지원하지 않으면 패킷이 폐기되거나 파편화되어 오히려 성능이 극단적으로 저하됨. (End-to-End 지원 확인 필수).  
* **NIC Offloading (TSO/GSO 등)**  
  * **연결되는 설정값**: ethtool \-k eth0로 확인하는 tcp-segmentation-offload 등.  
  * **내부 동작 원리**: OS 커널이 해야 할 TCP 세그먼트 쪼개기 및 캡슐화 작업을 물리적 네트워크 카드(NIC) 하드웨어에 떠넘겨(Offload) CPU 부하를 줄이는 기술.

## **4\. 이전/이후 강의 연결**

* **선행 지식**:  
  * 컴퓨터 구조(CPU, 메모리 구조): 패킷을 생성하고 헤더를 붙이는 과정이 결국 메모리에 데이터를 복사하고 CPU가 연산하는 과정임을 이해하기 위해 필요함.  
* **이후 연결**:  
  * \[L2 계층 \- 이더넷과 MAC\]: 로컬 망에서의 캡슐화 실체.  
  * \[L3 계층 \- IP와 라우팅\]: 중간 노드들이 어떻게 최적의 간선(경로)을 찾는지.  
  * \[L4 계층 \- TCP/UDP\]: 신뢰성 있는 캡슐화(포트, 시퀀스 번호)의 실체.  
  * \[L7 계층 \- HTTP, 프록시\]: 응용 프로그램 페이로드가 어떻게 생겼는지. 이 모든 과정이 '캡슐화'라는 뼈대 위에 살을 붙이는 과정임.

## **5\. 바로 해볼 수 있는 실험**

지금 당장 운영 중인 Linux나 K8s 환경 터미널을 열고 아래 3가지를 쳐보자.

**실험 1: 캡슐화의 실체 두 눈으로 보기 (tcpdump)**

\# 다른 터미널에서 구글로 curl \[https://google.com\](https://google.com) 을 쏜 뒤, 아래 명령어로 패킷 까보기  
sudo tcpdump \-i eth0 \-n \-c 5 \-X port 443

* **포인트**: \-X 옵션을 주면 16진수(Hex)와 ASCII로 실제 패킷 내용이 출력됨. 강의 마지막에 와이어샤크로 봤던 것처럼, 앞에 의미 모를 영문/숫자(MAC, IP, TCP 헤더)가 잔뜩 붙어있고 뒤에 암호화된 데이터 페이로드가 있는 **캡슐화된 상태**를 리눅스 터미널에서 바로 확인할 수 있음.

**실험 2: 내 서버의 브로드캐스트 도메인(LAN) 정보 확인하기**

ip addr show

* **포인트**: brd (broadcast) 뒤에 나오는 IP(예: 192.168.1.255)를 확인해봐. 이 주소로 패킷을 쏘면 내 서버와 같은 네트워크(서브넷)에 있는 모든 노드에게 메시지가 뿌려진다는 뜻이야.

**실험 3: MTU와 파편화 방지(DF bit) 테스트**

\# 1500 꽉 채운 크기(1472 \+ IP헤더 20 \+ ICMP헤더 8 \= 1500)로 핑 보내기  
ping \-c 1 \-M do \-s 1472 8.8.8.8  
\# 크기를 살짝 늘려서 보내보기 (패킷이 쪼개져야만 전송되도록)  
ping \-c 1 \-M do \-s 1473 8.8.8.8

* **포인트**: \-M do는 "패킷을 절대 쪼개지 마\!(Don't Fragment)"라는 뜻이야. 1473으로 보냈을 때 Message too long 에러가 난다면? "아, 우리 망 중간 노드의 간선 크기가 1500이라서 캡슐화 덩치가 커지면 못 지나가는구나"를 체감할 수 있어.

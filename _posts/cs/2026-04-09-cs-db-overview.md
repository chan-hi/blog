## **1\. 핵심 개념 정리**

### **① DBMS는 '서버 프로그램'이다 (Client-Server 아키텍처)**

* **정의**: DBMS(Database Management System)는 데이터베이스를 관리하고, 외부(응용 프로그램)의 요청을 처리하는 **서버 애플리케이션**이야.  
* **왜 존재하는가**: 데이터에 접근하려는 수많은 클라이언트(백엔드 파드들)의 요청을 중앙에서 안전하게 수신하고 처리하기 위해 존재해.  
* **어떻게 동작하는가**: 일반적인 웹 서버처럼 클라이언트와 **TCP 3-way handshake**를 맺고, 인증(ID/PW)을 거친 뒤, SQL이라는 '쿼리(요청)'를 받아 결과를 '응답'해.  
* **개념 간 관계**: 네가 구성하는 AWS RDS나 K8s 안의 DB 파드들도 결국 리눅스 위에서 특정 포트(예: MySQL 3306)를 열고 Listen 상태로 대기하는 TCP 서버 프로세스일 뿐이야.

### **② 파일 시스템 vs 데이터베이스 (동시성/무결성 제어)**

* **정의**: 단순 텍스트 파일이 아닌 DB를 사용하는 본질적인 이유는 데이터의 **일관성, 무결성, 동시성 제어**를 시스템 차원에서 보장받기 위함이야.  
* **왜 존재하는가**: 하나의 파일에 여러 파드가 동시에 쓰기를 시도하면 데이터가 덮어씌워지거나 깨지는 레이스 컨디션(Race Condition)이 발생하기 때문이지.  
* **어떻게 동작하는가**: DBMS 내부적으로 OS의 뮤텍스(Mutex), 세마포어(Semaphore)와 유사한 **Lock 매커니즘**을 구현하여 여러 요청이 들어와도 안전하게 순서를 제어해.  
* **개념 간 관계**: 리눅스 커널에서 배우는 '동기화 문제'가 데이터베이스 레벨로 올라온 것이 바로 이 개념이야.

### **③ 트랜잭션(Transaction)과 ACID**

* **정의**: 데이터베이스의 상태를 변화시키는 논리적인 작업 단위(1개 이상의 쿼리 묶음)와, 이 단위가 안전하게 처리되기 위해 지켜야 할 4가지 원칙(원자성, 일관성, 격리성, 지속성)이야.  
* **왜 존재하는가**: 결제 중 서버가 다운되거나(원자성 위협), 두 명이 동시에 같은 좌석을 예매하려는(격리성 위협) 시스템 장애/동시성 문제로부터 데이터를 보호하기 위해 필요해.  
* **어떻게 동작하는가**:  
  * **원자성(A)**: Commit 하거나 전체를 Rollback (All or Nothing)  
  * **일관성(C)**: 트랜잭션 전후로 스키마(규칙) 유지  
  * **격리성(I)**: 여러 트랜잭션이 동시에 실행될 때 서로 간섭하지 않게 Lock 적용  
  * **지속성(D)**: Commit된 결과는 DB 서버(인스턴스)가 죽어도 디스크(WAL, Redo Log)에 남아 보존됨  
* **개념 간 관계**: 스토리지의 디스크 I/O(지속성)와 CPU의 스케줄링/Lock(격리성) 개념이 총망라된 DB의 핵심이야.

## **2\. 장애·트러블슈팅 연결**

### **🚨 트러블슈팅 시나리오 1: "DBMS는 TCP 서버다"가 드러나는 순간**

* **장애 상황**: 이벤트 기간 중 트래픽이 몰리면서 백엔드 파드 로그에 java.net.ConnectException: Connection refused 또는 Too many connections 에러가 대량 발생.  
* **주니어의 함정**: "웹 서버(파드) 부하가 높네\! K8s HPA로 파드 개수를 팍팍 늘려야겠다(Scale-out)\!"라고 대처함.  
* **원리를 알면 보이는 것**: 파드를 늘리면 파드 내의 Connection Pool 도 늘어나서 DB 서버로 향하는 TCP Connection이 DB의 max\_connections 한계치를 돌파하게 돼. 이 원리를 알면 무작정 앱을 스케일아웃 하는 게 아니라, DB 앞단에 PgBouncer나 RDS Proxy 같은 Connection Pooling 레이어를 두거나 DB 리소스를 스케일업 해야 함을 추론할 수 있어.

### **🚨 트러블슈팅 시나리오 2: 격리성(Isolation)이 드러나는 순간**

* **장애 상황**: 특정 배치 작업이 돌기 시작하면 RDS의 CPU가 치솟고, 정상적이던 API API 응답 속도가 급격히 느려짐. 앱 로그에는 Lock wait timeout exceeded; try restarting transaction 발생.  
* **주니어의 함정**: "DB 리소스가 부족한가 봐. 인스턴스 타입을 업그레이드하자\!"  
* **원리를 알면 보이는 것**: CPU가 높은데 디스크 I/O가 생각보다 낮다면, 이는 연산이 많은 게 아니라 트랜잭션 간의 '격리성'을 보장하기 위해 Lock이 걸려 대기(Wait)하는 스레드가 많아서 발생하는 현상(Context Switching 과다)이야. 리소스 증설이 아니라 SHOW PROCESSLIST나 innotop을 켜서 어떤 쿼리가 Lock을 쥐고 안 놓는지 찾아내야 해.

## **3\. 성능·튜닝 연결**

강의에 나온 개념들이 실제 클라우드/OS 파라미터와 어떻게 연결되는지 볼까?

* **지속성(Durability)과 innodb\_flush\_log\_at\_trx\_commit (MySQL)**  
  * **설명**: 트랜잭션이 Commit될 때마다 그 내역을 디스크(Redo Log)에 쓸 것인지 묻는 파라미터야.  
  * **내부 동작**:  
    * 값 1 (기본값): Commit 시 무조건 디스크에 Sync. 완벽한 ACID 보장. (느림)  
    * 값 2 또는 0: OS의 버퍼 캐시에만 쓰고 나중에 디스크로 내림.  
  * **부작용/튜닝**: 초당 수만 건의 로그성 데이터를 쏟아붓는다면 무조건 1로 두는 건 디스크 I/O 병목(IOPS 한계)을 유발해. 일부분 유실되어도 괜찮은 데이터라면 이 값을 2로 튜닝해서 **지속성을 조금 타협하고 엄청난 성능 향상**을 얻어낼 수 있어.  
* **TCP 연결과 커널 파라미터 (net.ipv4.tcp\_tw\_reuse)**  
  * **설명**: DBMS가 서버로서 수많은 단기 연결을 맺고 끊으면 서버 OS에 TIME\_WAIT 상태의 소켓이 쌓여 로컬 포트가 고갈될 수 있어.  
  * **튜닝**: 리눅스 커널의 TCP 관련 파라미터를 튜닝하여 소켓을 재사용하게 만들면, 데이터베이스 앞단의 네트워크 단에서 발생하는 병목을 줄일 수 있지.

## **4\. 이전/이후 강의 연결**

* **선행 지식 (이전)**  
  * **운영체제의 프로세스/스레드 동기화 (Mutex, Semaphore, Race Condition)**: 파일 대신 DB를 쓰는 이유가 바로 이 동기화 문제를 DB가 대신 해결(Lock)해 주기 때문이야.  
  * **네트워크 계층 (TCP 3-Way Handshake)**: DBMS가 클라이언트와 소통하는 근간 모델.  
* **후속 지식 (이후)**  
  * **RDBMS 심화**: 스키마(구조)를 엄격히 지키기 위한 정규화, 무결성 제약 조건, 빠른 검색을 위한 인덱스 구조 (B-Tree).  
  * **NoSQL**: 스키마리스(Schema-less) 구조가 클라우드 환경의 수평적 확장(Scale-out)과 어떻게 시너지를 내는지 배우게 됨.

## **5\. 바로 해볼 수 있는 실험**

인프라 엔지니어답게 터미널을 열고 직접 눈으로 확인해 보자.

1. **DBMS도 결국 리눅스의 TCP 프로세스임을 확인하기**  
   * DB가 떠 있는 서버(또는 파드 내부)에 접속해서 아래 명령어를 쳐봐.  
   * sudo netstat \-ntlp | grep 3306 (MySQL 기준)  
   * DB 프로세스가 0.0.0.0:3306을 LISTEN 하고 있는 걸 볼 수 있어. DBMS도 결국 아파치/엔진엑스 같은 하나의 네트워크 데몬이라는 걸 깨닫게 될 거야.  
2. **현재 DB에 붙어있는 Client(응용 프로그램) 세션 확인하기**  
   * DB 콘솔에 접속해서 쿼리를 날려봐: SHOW PROCESSLIST; (MySQL/MariaDB)  
   * 현재 몇 개의 커넥션이 맺어져 있고, 어떤 쿼리가 돌고 있으며, Sleep 상태로 커넥션만 잡고 있는 녀석이 얼마나 되는지 바로 보일 거야. (이게 트러블슈팅의 1순위 습관이야)  
3. **K8s 환경에서 앱 파드 스케일링 전후의 DB TCP 소켓 관찰**  
   * watch \-n 1 "ss \-s" 명령어를 DB 파드나 노드에 켜두고, K8s 백엔드 파드를 강제로 5개에서 50개로 늘려봐 (kubectl scale deploy ...).  
   * TCP estab (Established) 개수가 어떻게 치솟는지 관찰하면 "Client-Server" 관계가 직관적으로 다가올 거야.

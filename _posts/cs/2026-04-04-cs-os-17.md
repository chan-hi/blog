## **1\. 핵심 개념 정리**

### **① 임계 구역(Critical Section)과 레이스 컨디션(Race Condition)**

* **정의**: '임계 구역'은 동시다발적으로 실행하면 문제가 생기는 공유 자원 접근 코드이며, 관리가 안 되어 데이터가 꼬이는 현상을 '레이스 컨디션'이라고 합니다.  
* **왜 존재하는가**: 멀티 스레드나 멀티 프로세스 환경에서 여러 주체가 하나의 공유 자원(변수, 파일, DB 등)을 동시에 읽고 쓰려고 하면 일관성이 깨지기 때문입니다.  
* **어떻게 동작하는가**: A가 값을 읽고 1을 더해 저장하기 직전에(Context Switching 발생), B가 과거의 값을 읽어가서 1을 더해버리면 최종적으로 \+2가 아닌 \+1만 반영되는 현상입니다.  
* **개념 간 관계**: 16강에서 배운 스레드 간 'Heap/Data 영역 공유'의 치명적인 부작용이 바로 레이스 컨디션이며, 이를 막기 위한 기술이 \*\*동기화(Synchronization)\*\*입니다.

### **② 뮤텍스 락(Mutex)과 세마포어(Semaphore)**

* **정의**: 뮤텍스는 **단 1개**의 자원에 대한 접근을 통제하는 자물쇠이고, 세마포어는 **N개**의 자원(수용량)을 통제하는 신호기입니다.  
* **왜 존재하는가**: 임계 구역에 들어갈 때 '상호 배제(Mutual Exclusion)'를 보장하여 레이스 컨디션을 막기 위해 존재합니다.  
* **어떻게 동작하는가**:  
  * **뮤텍스**: 화장실 칸이 1개. 키(acquire)를 가진 사람만 들어가고, 나오면서 키를 반납(release)합니다.  
  * **세마포어**: 화장실 칸이 N개. 빈 칸 개수(S)를 확인하고 1 감소시키며 입장(wait), 나오면서 1 증가시키며 대기자를 깨움(signal).  
* **개념 간 관계**: 모니터(Monitor)나 자바의 synchronized 키워드 등 고수준의 동기화 기능들이 내부적으로 이 원리를 활용해 구현됩니다.

### **③ 교착 상태(Deadlock)와 4가지 발생 조건**

* **정의**: 두 개 이상의 프로세스/스레드가 서로가 가진 자원을 무한정 기다리며 시스템이 영원히 멈춰버리는 현상입니다.  
* **왜 존재하는가**: 자원을 점유한 상태에서 남의 자원을 탐내고(점유와 대기), 남의 것을 강제로 뺏을 수도 없으며(비선점), 그 관계가 꼬리를 물고 이어지기(원형 대기) 때문입니다.  
* **어떻게 동작하는가**:  
  1. 상호 배제 (자원은 한 번에 한 명만)  
  2. 점유와 대기 (내 자원을 쥔 채로 남의 자원 대기)  
  3. 비선점 (남의 자원을 강제로 뺏지 못함)  
  4. 원형 대기 (A ➔ B ➔ C ➔ A 형태로 물고 물림)  
     이 4가지 조건이 **모두 충족될 때만** 데드락이 발생합니다.

## **2\. 장애·트러블슈팅 연결**

### **💡 DB 트랜잭션 교착 상태 (MySQL Deadlock)**

* **이 원리가 드러나는 장애 상황**:  
  Spring Boot 백엔드 애플리케이션의 에러 로그에 Deadlock found when trying to get lock; try restarting transaction 에러가 무더기로 발생. 사용자 결제가 실패함.  
* **원리를 모르면 빠지는 함정**:  
  단순한 네트워크 타임아웃이나 DB 스펙(CPU/Memory) 부족인 줄 알고 RDS 인스턴스 클래스를 올림. (데드락은 논리적 문제라 스케일업을 해도 똑같이 발생함)  
* **원리를 알면 보이는 것**:  
  "트랜잭션 A가 X테이블을 물고 Y테이블을 업데이트하려는데, 트랜잭션 B는 Y테이블을 물고 X테이블을 업데이트하려다 \*\*원형 대기(Circular Wait)\*\*에 빠졌구나\!"  
  ➔ 트랜잭션 로직을 분석하여 테이블 락을 획득하는 **순서(오름차순 등)를 일치**시키는 방향으로 코드 수정을 가이드할 수 있음 (교착 상태 예방 전략).

### **💡 Terraform State Lock 경합 (뮤텍스의 실무 예)**

* **이 원리가 드러나는 장애 상황**:  
  Jenkins나 GitHub Actions에서 Terraform CI/CD 파이프라인이 돌다가 Error acquiring the state lock 로그를 뱉고 파이프라인이 실패함.  
* **원리를 모르면 빠지는 함정**:  
  "Terraform 코드가 고장 났나?" 하고 HCL 코드만 뒤적거림.  
* **원리를 알면 보이는 것**:  
  "S3/DynamoDB에 저장된 tfstate 파일은 '공유 자원'이고, 두 명의 엔지니어(또는 파이프라인)가 동시에 terraform apply를 실행하여 **뮤텍스 락(Mutex Lock)** 경합에서 한쪽이 튕겨 나간 거구나\!" ➔ 한 파이프라인이 끝날 때까지 얌전히 기다렸다가 실행(release 대기)하면 됨을 인지.

### **💡 분산 환경의 레이스 컨디션 (초과 예약 / 쿠폰 발급)**

* **이 원리가 드러나는 장애 상황**:  
  선착순 100명 쿠폰 발급 이벤트인데, DB를 까보니 105명에게 쿠폰이 발급되어 있음.  
* **원리를 모르면 빠지는 함정**:  
  프론트엔드에서 버튼 클릭을 여러 번 막지 않았다고 프론트 탓만 함.  
* **원리를 알면 보이는 것**:  
  "5대의 K8s 파드가 동시에 DB의 남은 쿠폰 수를 읽고(read) \-1을 한 뒤 덮어쓰는(write) **레이스 컨디션**이 발생했구나. 여러 서버(프로세스) 간의 **상호 배제**가 보장되지 않았어." ➔ Redis Distributed Lock(분산 락)을 도입하여 K8s 클러스터 전체 레벨의 뮤텍스를 구현해야 함을 조언할 수 있음.

## **3\. 성능·튜닝 연결**

강의에서 배운 개념은 애플리케이션 및 인프라의 '스레드 풀' 파라미터 튜닝과 직결됩니다.

* **관련 시스템 파라미터/설정값**:  
  * HikariCP (DB Connection Pool)의 maximumPoolSize (실질적인 세마포어의 S 값 역할)  
  * K8s Ingress Controller의 worker-connections (동시 접속 허용량)  
* **내부 동작과 튜닝의 원리**:  
  DB 커넥션은 '공유 자원'이고, 커넥션 풀은 일종의 \*\*세마포어(Semaphore)\*\*입니다.  
  S=10으로 설정하면, 11번째 요청부터는 커넥션을 얻기 위해 대기(wait 상태로 블로킹)하게 됩니다. 트래픽이 몰리면 서버 스레드들이 모두 wait 상태에 빠져 애플리케이션이 먹통이 됩니다.  
* **부작용**:  
  대기를 없애겠다고 세마포어 허용량(maximumPoolSize)을 1000으로 너무 크게 잡으면? DB 입장에서는 수많은 트랜잭션이 한 번에 락(Mutex)을 얻기 위해 경합(Lock Contention)하면서 CPU가 폭발하여 데이터베이스 전체가 다운(Crash)됩니다. 따라서 시스템의 한계점에 맞춘 적절한 크기의 세마포어 제어가 성능 튜닝의 핵심입니다.

## **4\. 이전/이후 강의 연결**

* **선행 지식 (16강)**:  
  멀티 스레드 구조를 배웠습니다. 스레드들은 Code/Data/Heap 영역을 공유하기 때문에 서로 통신하기 좋지만, 그 '공유'가 비극(레이스 컨디션)의 씨앗이 됨을 배웠습니다.  
* **이 강의의 위치 (현재)**:  
  그 공유로 인한 문제를 막기 위해 '순서'를 정하고 '자물쇠'를 채우는 **동기화 기법**과, 자물쇠를 잘못 채웠을 때 다 같이 죽는 **교착 상태**를 배웠습니다.  
* **후행 지식 (이후 내용)**:  
  CPU와 프로세스 관리에 대한 이야기가 일단락되었습니다. 다음부터는 "물리적인 메모리는 한정되어 있는데 이 수많은 프로세스들을 어떻게 다 올려놓을까?"에 대한 해답인 \*\*메모리 관리 기법(가상 메모리, 페이징 등)\*\*으로 넘어갑니다. (OOM 커널 패닉을 이해하는 핵심)

## **5\. 바로 해볼 수 있는 실험**

본인의 실무 환경에서 아래 현상들을 직접 재현해 보세요\!

1. **MySQL 데드락(Deadlock) 눈으로 직접 만들기**  
   터미널 2개를 열어 같은 MySQL DB에 접속하고 아래를 순서대로 한 줄씩 실행해보세요.  
   * \[세션 1\] START TRANSACTION; UPDATE table\_a SET val=1 WHERE id=1; (락 획득)  
   * \[세션 2\] START TRANSACTION; UPDATE table\_a SET val=1 WHERE id=2; (락 획득)  
   * \[세션 1\] UPDATE table\_a SET val=1 WHERE id=2; (세션 2의 락 대기 \- 점유와 대기)  
   * \[세션 2\] UPDATE table\_a SET val=1 WHERE id=1; (세션 1의 락 대기 ➔ **원형 대기 완성**)  
   * 결과: 즉시 ERROR 1213 (40001): Deadlock found 에러가 발생하며 커널(DBMS)이 하나를 강제 종료(회복)시키는 것을 볼 수 있습니다.  
2. **Terraform 뮤텍스 락 확인하기**  
   * 터미널 1에서 terraform apply를 실행하고 프롬프트(Yes/No)에서 대기합니다. (락 획득 상태)  
   * 터미널 2에서 동시에 terraform apply를 실행해 봅니다.  
   * 결과: Error acquiring the state lock 에러 발생. 이것이 인프라 레벨의 \*\*상호 배제(Mutual Exclusion)\*\*입니다.  
3. **Java/Node.js 애플리케이션 Thread Dump에서 BLOCKED 상태 확인하기**  
   \# 자바 애플리케이션의 PID를 찾아 스레드 덤프를 뜹니다.  
   jstack \<PID\> | grep \-i "blocked"

   * 덤프를 열어보면 특정 스레드가 waiting for monitor entry (모니터 진입 대기 \= 뮤텍스 락 대기) 상태로 멈춰있는 모습을 실무에서 아주 자주 보게 될 것입니다.

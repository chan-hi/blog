## **1\. 핵심 개념 정리 (클라우드 인프라 관점)**

인터럽트는 CPU가 일일이 하드웨어를 감시하지 않도록 만들어진 "알림(Alarm)" 시스템이야.

### **① 인터럽트 (Interrupt)**

* **정의**: CPU가 정상적으로 명령어를 실행하고 있는 흐름을 '잠깐 멈추게(방해)' 하는 신호.  
* **왜 존재하는가**: 이 알림 시스템이 없으면 CPU는 "프린터 다 됐어? 네트워크 패킷 왔어?" 하고 주기적으로 물어보는 폴링(Polling)을 하느라 다른 일을 아예 못 하기 때문. (세탁기 다 돌아갈 때까지 앞에서 기다리는 것과 같음)

### **② 비동기 인터럽트 (Hardware Interrupt)**

* **정의**: 키보드, 마우스, 특히 클라우드에서는 \*\*네트워크 카드(NIC)\*\*나 **디스크(EBS)** 같은 외부 하드웨어가 발생시키는 알림.  
* **실무의 의미**: K8s 노드(EC2)로 엄청난 트래픽(네트워크 패킷)이 쏟아져 들어올 때, 네트워크 카드는 CPU에게 "패킷 왔어\! 빨리 처리해 줘\!"라며 무수히 많은 하드웨어 인터럽트를 날림.

### **③ 동기 인터럽트 (Exception / 예외)**

* **정의**: CPU가 명령어 자체를 실행하다가 마주친 예외 상황.  
* **대표적 예 (페이지 폴트 \- Page Fault)**: 앱이 특정 메모리를 읽으려 했는데, 그 데이터가 아직 물리 메모리(RAM)에 안 올라와 있고 디스크(Swap)에 있을 때 발생. CPU는 작업을 멈추고(Fault), OS가 디스크에서 데이터를 메모리로 긁어올 때까지 기다린 후 다시 원래 명령어를 재시도함.

### **④ 인터럽트 처리 흐름과 ISR (Interrupt Service Routine)**

* **어떻게 동작하는가**:  
  1. 외부 장치가 **요청 신호**를 보냄.  
  2. CPU가 **플래그 레지스터**를 보고 "지금 딴 거 처리해도 되나?" 판단.  
  3. 괜찮다면 현재 하던 일(프로그램 카운터, 레지스터 값)을 \*\*스택(Stack)\*\*에 백업.  
  4. 어떤 알림인지(인터럽트 벡터) 확인 후, OS 커널에 있는 해당 처리 함수(**인터럽트 서비스 루틴, ISR**)를 실행.  
  5. 처리가 끝나면 스택에서 백업된 값을 복구해 원래 하던 일 계속함.

## **2\. 장애·트러블슈팅 연결**

### **🚨 이 원리가 드러나는 장애 상황**

**"이벤트 기간 트래픽이 몰리자, Nginx나 HAProxy 컨테이너가 떠 있는 노드의 특정 CPU 코어만 사용률 100%를 치고 응답이 타임아웃 나는 현상"**

* 로그/메트릭: 노드에서 top을 쳤을 때 특정 CPU 코어의 **%si (Soft Interrupt) 혹은 %hi (Hardware Interrupt)** 수치가 90% 이상으로 빨갛게 달아오르고, 커널 스레드인 ksoftirqd 프로세스가 CPU 자원을 다 먹고 있음.  
* **CS 지식의 발현**: 엄청난 네트워크 패킷이 들어오면서 네트워크 카드가 CPU에게 \*\*"비동기 인터럽트"\*\*를 초당 수만 번씩 때리고 있는 거야. CPU는 앱(Nginx)의 로직(사용자 모드)을 실행할 새도 없이, 스택에 백업하고 \*\*ISR(패킷 처리 루틴)\*\*을 실행하느라(커널 모드) 모든 시간을 다 써버린 상태지.

### **🕳️ 원리를 모르면 빠지는 함정**

* **주니어의 흔한 실수**: "어? CPU가 100%네? Nginx 워커 프로세스 수를 늘리거나, 컨테이너 CPU Limit을 올려야겠다\!"  
* **부작용**: Nginx 프로세스를 아무리 늘려도 소용없음. 병목은 앱(Nginx)이 아니라 **OS 커널이 하드웨어 인터럽트를 처리하는 구간**에 있기 때문. 결국 서버는 계속 뻗어있게 됨.

### **💡 원리를 알면 보이는 것**

"아, 네트워크 인터럽트가 특정 1번 코어에만 몰려서 ksoftirqd가 터지고 있구나\! **네트워크 카드의 인터럽트를 노드의 여러 CPU 코어로 분산(IRQ Balancing / RPS 설정)** 시켜서 여러 코어가 알림을 나눠서 받게 만들어야겠다\!"

## **3\. 성능·튜닝 연결**

* **irqbalance와 하드웨어 인터럽트 튜닝**:  
  * 리눅스에는 하드웨어 인터럽트 요청을 여러 CPU 코어에 골고루 분산해 주는 irqbalance 데몬이 있어. 클라우드에서 고성능 로드밸런서나 NAT 인스턴스를 직접 구축할 때는 이 인터럽트 친화도(IRQ Affinity)를 튜닝해서, 네트워크 패킷 인터럽트만 전담으로 처리하는 코어와 앱 로직만 도는 코어를 물리적으로 분리하는 최적화(CPU Pinning)를 하기도 해.  
* **페이지 폴트(Page Fault)와 메모리 튜닝**:  
  * 강의에서 배운 **동기 인터럽트(Fault)** 중 '페이지 폴트'는 클라우드 DB(Redis, MySQL) 성능 저하의 주범이야. 메모리가 부족해서 OS가 메모리 데이터를 디스크(Swap)로 쫓아냈다가 다시 부르는 과정에서 수많은 페이지 폴트 인터럽트가 발생해. 그래서 성능이 생명인 K8s 환경에서는 아예 노드의 **Swap 기능을 꺼버리도록(Kubelet 설정)** 강제하는 거야. (디스크까지 다녀오는 간접 사이클의 비용이 너무 크기 때문)

## **4\. 이전/이후 강의 연결**

* **이전 지식 (6강)**:  
  * 저번 강의의 "프로그램 카운터(PC)"와 "스택(Stack)" 기억나? 인터럽트가 발생하면 하던 일을 멈추기 위해 현재 PC 값을 스택에 PUSH 해. ISR 루틴이 끝나면 스택에서 POP 해서 PC 값을 복원하는 거지.  
* **이후 강의 (운영 체제)**:  
  * 강의 마지막에 "시스템 콜(System Call)이 바로 소프트웨어 인터럽트다"라고 했어. 우리가 짠 코드가 파일 입출력을 하거나 네트워크 통신을 할 때 바로 이 '인터럽트'를 통해 OS(운영체제) 커널을 깨워서 일을 시킨다는 걸, 다음 운영체제 파트에서 깊게 다루게 될 거야.

## **5\. 바로 해볼 수 있는 실험**

내 클라우드 노드에서 인터럽트가 어떻게 발생하고 있는지 직접 관찰해 보자.

1. **현재 내 서버의 실시간 인터럽트 현황 보기**  
   * 터미널 명령어: watch \-n 1 cat /proc/interrupts  
   * 서버 안의 각 CPU 코어(CPU0, CPU1 등)가 어떤 하드웨어(NIC, Timer 등)로부터 몇 번의 인터럽트를 받았는지 실시간 카운트가 올라가는 걸 볼 수 있어. 특정 네트워크 어댑터(eth0) 쪽에 숫자가 미친 듯이 올라가면 그게 바로 패킷 인터럽트야.  
2. **인터럽트와 컨텍스트 스위칭 카운트 보기**  
   * 터미널 명령어: vmstat 1  
   * 1초마다 시스템 상태를 찍어주는데, system 카테고리 아래의 in (인터럽트 발생 횟수)과 cs (컨텍스트 스위칭 횟수) 항목을 봐. 트래픽 테스트(Load Test) 도구를 돌려보면 이 숫자가 폭발적으로 증가하는 걸 목격할 수 있어.  
3. **CPU의 인터럽트 처리 점유율 (si, hi) 확인하기**  
   * 터미널 명령어: top (이후 숫자 1 누르기)  
   * 상단 CPU 지표에 hi (Hardware Interrupts)와 si (Software Interrupts) 퍼센티지가 보일 거야. 클라우드에서 네트워크 트래픽이나 디스크 I/O가 빡세게 돌 때 이 수치가 올라가면 커널이 인터럽트 처리하느라 땀 빼고 있다는 증거야.

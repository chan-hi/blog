## **1\. 핵심 개념 정리**

### **① NoSQL (Not Only SQL)**

* **정의**: 전통적인 관계형 데이터베이스(RDBMS)의 한계를 극복하기 위해 만들어진, 고정된 스키마가 없고 수평적 확장이 용이한 데이터베이스 시스템이야.  
* **왜 존재하는가**:  
  1. **빅데이터**: 데이터 양이 너무 많아져서 RDBMS 하나로는 감당이 안 됨.  
  2. **유연성**: 소셜 미디어처럼 데이터 구조가 수시로 변하는 환경에서 고정된 스키마는 장애물이 됨.  
  3. **확장성**: RDBMS의 수직 확장(Scale-up) 비용이 너무 비싸지거나 물리적 한계에 부딪힘.  
* **어떻게 동작하는가**: 테이블 간의 복잡한 JOIN을 포기하는 대신, 데이터를 한 곳에 모아두거나(Document), 단순한 키-값(Key-Value) 형태로 저장하여 읽기/쓰기 속도를 극대화해.  
* **개념 간 관계**: RDBMS가 \*\*ACID(강한 일관성)\*\*를 중시한다면, NoSQL은 \*\*BASE(가용성과 성능)\*\*를 중시해. 클라우드 인프라에서는 보통 이 둘을 섞어 쓰는 'Polyglot Persistence' 전략을 사용해.

### **② CAP 정리 (CAP Theorem)**

* **정의**: 분산 시스템에서 **일관성(Consistency), 가용성(Availability), 분할 내성(Partition Tolerance)** 세 가지를 동시에 모두 만족시킬 수 없다는 이론이야.  
* **왜 존재하는가**: 분산 환경(클라우드)에서 장애가 발생했을 때, 시스템이 어떤 가치를 우선시할지 결정하는 기준이 돼.  
* **어떻게 동작하는가**:  
  * **CP**: 데이터 정확성이 최우선 (장애 시 서비스 중단 감수).  
  * **AP**: 서비스 가용성이 최우선 (데이터가 잠시 달라도 일단 응답).  
* **개념 간 관계**: 인프라 설계 시 "우리 서비스는 결제 시스템인가(CP)? 아니면 SNS 좋아요 기능인가(AP)?"를 판단하는 근거가 되지.

## **2\. 장애·트러블슈팅 연결**

### **🚨 트러블슈팅 시나리오 1: 세션 불일치 장애 (Session Clustering)**

* **장애 상황**: 사용자가 로그인을 했는데, 페이지를 새로고침할 때마다 로그아웃이 됨. (L4/L7 로드밸런서 뒤에 여러 대의 웹 서버가 있는 환경)  
* **주니어의 함정**: "로드밸런서의 Sticky Session 설정이 풀렸나 봐요\! 아니면 웹 서버 로컬 디스크의 세션 파일이 깨졌나?"  
* **원리를 알면 보이는 것**: 세션 정보를 각 서버의 메모리나 로컬 파일(RDBMS 방식의 사고)에 저장하면 확장성이 떨어져. 이때 **Redis(NoSQL)** 같은 키-값 저장소를 도입해 세션을 공통으로 관리(Session Store)하면, 서버가 수백 대로 늘어나도(Scale-out) 세션 불일치 장애를 완벽히 막을 수 있어.

### **🚨 트러블슈팅 시나리오 2: 'Eventual Consistency'로 인한 혼란**

* **장애 상황**: 사용자가 게시글을 수정했는데, 목록에서는 수정 전 내용이 보이다가 몇 초 뒤에야 수정된 내용이 보임. 사용자는 "데이터가 안 바뀌었다"고 클레임을 제기함.  
* **주니어의 함정**: "DB에 저장이 안 됐나 봐요\! 인서트 쿼리에 버그가 있어요\!"  
* **원리를 알면 보이는 것**: 이건 장애가 아니라 NoSQL의 **'결과적 일관성(Eventual Consistency)'** 특성 때문이야. 분산된 노드들에 데이터가 복제되는 동안 아주 짧은 시간 불일치가 발생하는 거지. 인프라 엔지니어는 애플리케이션 팀에 "NoSQL의 읽기 일관성 옵션(Strong Consistency vs Eventual Consistency)을 확인해 보세요"라고 조언할 수 있어.

## **3\. 성능·튜닝 연결**

### **① 샤딩(Sharding)과 파티션 키(Partition Key)**

* **설명**: 데이터를 여러 서버에 쪼개서 저장하는 기술이야.  
* **내부 동작**: 데이터를 나눌 기준(Partition Key)을 정하고, 그 값에 따라 특정 노드로 요청을 보냄.  
* **부작용/튜닝**: 파티션 키를 잘못 잡으면 특정 노드에만 트래픽이 몰리는 **'Hot Partition'** 현상이 발생해. AWS DynamoDB 같은 NoSQL을 쓸 때 특정 파티션만 'Throttling'이 걸린다면, 키 설계를 다시 해야 하는 신호야.

### **② TTL(Time To Live) 설정**

* **설명**: 데이터의 유효 기간을 정하는 설정이야.  
* **원리**: NoSQL(특히 Redis나 DynamoDB)은 메모리나 스토리지 비용이 발생하므로, 오래된 로그나 임시 데이터는 자동으로 지워지게 설정해야 해.  
* **튜닝**: TTL을 너무 짧게 잡으면 DB 히트율(Hit Rate)이 떨어져서 성능이 낮아지고, 너무 길게 잡으면 스토리지 비용이 폭증해.

## **5\. 바로 해볼 수 있는 실험**

1. **Docker로 Redis 띄워보고 Key-Value 체험하기**  
   * 터미널에서 간단하게 테스트해 봐:

docker run \-d \--name my-redis \-p 6379:6379 redis  
docker exec \-it my-redis redis-cli  
\> set user:1 "Alice"  
\> get user:1  
\> expire user:1 10  \# 10초 뒤에 사라지게 설정 (TTL 체험)

2. **MongoDB에서 스키마리스(Schema-less) 확인하기**  
   * 관계형 DB와 달리 필드를 내 맘대로 넣어도 들어가는지 확인해 봐:

// MongoDB Shell (mongosh)  
db.users.insertOne({ name: "Alice", age: 25 })  
db.users.insertOne({ name: "Bob", email: "bob@example.com", hobby: "Coding" })   
// 필드 구조가 달라도 에러 없이 잘 들어감\!

3. **CAP 정리 시뮬레이션 고민해보기**  
   * "만약 우리 서비스의 DB 노드 3대 중 1대가 네트워크 단절(Partition)이 된다면? 나머지 2대라도 서비스를 계속하게 할 것인가(Availability), 아니면 데이터 정합성을 위해 전체를 멈출 것인가(Consistency)?" 이 질문에 답을 내려보는 연습이 인프라 설계의 시작이야.

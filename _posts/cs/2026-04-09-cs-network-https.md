## **1\. 핵심 개념 정리**

### **① 대칭키와 공개키(비대칭키) 암호화**

* **정의**: 데이터를 암복호화할 때 동일한 키를 쓰느냐(대칭), 서로 다른 한 쌍의 키를 쓰느냐(공개키)의 차이.  
* **왜 존재하는가**:  
  * **대칭키**: 속도는 빠르지만, 상대방에게 키를 전달하는 과정에서 탈취될 위험이 있음.  
  * **공개키**: 키 전달 문제는 해결되지만, 연산이 너무 복잡해서 대용량 데이터 전송에는 부적합함.  
* **어떻게 동작하는가**: HTTPS는 이 둘의 장점을 합친 **하이브리드 방식**을 써. 공개키로 '대칭키'를 안전하게 주고받은 뒤, 실제 데이터는 그 대칭키로 빠르게 주고받는 식이지.  
* **개념 간 관계**: 이 암호화 원리가 L4 이상의 전송 계층 보안인 **SSL/TLS**의 뼈대가 돼.

### **② 인증서(Certificate)와 CA(Certificate Authority)**

* **정의**: 서버가 믿을 수 있는 곳인지 증명해주는 디지털 문서와 이를 발급해주는 공인 기관.  
* **왜 존재하는가**: 암호화만 한다고 끝이 아니야. "내가 접속한 구글이 진짜 구글 서버가 맞나?"라는 **신원 확인**이 안 되면 피싱 사이트에 속을 수 있거든.  
* **어떻게 동작하는가**: 서버는 CA의 비밀키로 서명된 인증서를 브라우저에게 전달해. 브라우저는 이미 내장된 CA의 공개키로 이 서명을 검증해서 신뢰 여부를 판단해.  
* **개념 간 관계**: 클라우드에서는 \*\*AWS ACM(Certificate Manager)\*\*이 일종의 CA 관리 도구 역할을 수행해.

### **③ SSL/TLS 핸드쉐이크 (Handshake)**

* **정의**: TCP 연결(3-Way)이 맺어진 후, 실제로 암호화 통신을 시작하기 위해 암호화 알고리즘을 협의하고 키를 교환하는 과정.  
* **왜 존재하는가**: 클라이언트와 서버가 서로 지원하는 보안 수준이 다를 수 있기 때문에, 최선의 보안 방식을 맞추기 위해 필요해.  
* **어떻게 동작하는가**: Client Hello(나 이거 쓸 수 있어) \-\> Server Hello(그럼 이거 쓰자 \+ 인증서) \-\> 키 교환 \-\> 완료 순으로 진행돼.  
* **개념 간 관계**: 3-Way Handshake 이후에 일어나기 때문에, HTTPS는 HTTP보다 연결 수립 과정이 더 길고 복잡해(Latency 발생 원인).

## **2\. 장애·트러블슈팅 연결**

### **🚨 시나리오 A: "인증서 체인(Certificate Chain) 누락" 장애**

* **이 원리가 드러나는 장애 상황**: PC 브라우저에서는 잘 되는데, 특정 안드로이드 앱이나 오래된 서버의 curl 명령어로 접속하면 SSL Certificate Error: self signed certificate in certificate chain 에러가 발생함.  
* **원리를 모르면 빠지는 함정**: "내 브라우저에선 잘 되는데? 사용자 폰이 이상한가?" 하고 방치하거나 인증서를 다시 발급받음.  
* **원리를 알면 보이는 것**: "서버가 인증서를 보낼 때 자기 것만 보내고 상위 CA로 이어지는 \*\*중간 인증서(Intermediate Certificate)\*\*를 안 보냈구나\! PC 브라우저는 미리 캐싱해둬서 괜찮지만, 엄격한 클라이언트는 검증을 못 하는 거야." \-\> Nginx나 로드밸런서 설정에서 Full-Chain 인증서를 적용해야 함을 파악.

### **🚨 시나리오 B: TLS 버전 미스매치 (Legacy 시스템 연동)**

* **이 원리가 드러나는 장애 상황**: 최신 보안 가이드에 따라 로드밸런서에서 TLS 1.0, 1.1을 끄고 **TLS 1.2/1.3만 허용**함. 그랬더니 오래된 자바 1.7 기반의 내부 시스템들이 우리 서버에 접속을 못 하고 Handshake Failure를 뱉음.  
* **원리를 모르면 빠지는 함정**: 네트워크 방화벽이 막힌 줄 알고 보안 그룹(Security Group)만 계속 체크함.  
* **원리를 알면 보이는 것**: "핸드쉐이크 과정에서 서로 합의할 수 있는 TLS 버전이나 \*\*사이퍼 스위트(Cipher Suite)\*\*가 하나도 없어서 연결이 깨진 거구나." \-\> 클라이언트 환경을 업그레이드하거나, 보안 위험을 감수하고 서버에서 특정 사이퍼 스위트를 일시적으로 허용하는 조치 수행.

## **3\. 성능·튜닝 연결**

* **SSL/TLS 오프로딩 (Termination)**  
  * **연결되는 원리**: 암복호화는 CPU 연산량이 매우 많아.  
  * **설정**: AWS ALB나 Nginx에서 443(HTTPS)을 받고, 실제 백엔드 서버로는 80(HTTP)으로 넘겨주는 방식.  
  * **효과**: 개별 웹 서버들이 암복호화 연산에 CPU를 뺏기지 않고 비즈니스 로직 처리에만 집중할 수 있게 함.  
* **TLS 1.3 업그레이드**  
  * **설정**: Nginx ssl\_protocols TLSv1.2 TLSv1.3;  
  * **효과**: TLS 1.2는 핸드쉐이크에 2-RTT가 필요하지만, 1.3은 **1-RTT**로 줄어듦. 클라이언트 입장에서 첫 접속 속도가 눈에 띄게 빨라짐.  
* **HSTS (HTTP Strict Transport Security)**  
  * **설정**: 응답 헤더에 Strict-Transport-Security 추가.  
  * **효과**: 브라우저에게 "앞으로 1년 동안은 80으로 오지 말고 무조건 443으로만 와\!"라고 명령함. 80 \-\> 443 리다이렉션 오버헤드를 줄이고 보안을 강화함.

## **4\. 이전/이후 강의 연결**

* **이전 강의 (\[31\~32강. HTTP 기초/심화\])**: HTTP 메시지가 평문으로 전달될 때의 위험성을 알아야 HTTPS의 가치를 이해할 수 있어. 또한 리버스 프록시(ALB)가 인증서 종단점(Termination) 역할을 수행한다는 연결점이 있음.  
* **이후 강의 (\[네트워크 장 마무리\])**: 이제 네트워크의 큰 그림을 다 그렸어. 다음 장부터는 이 인프라 위에서 돌아가는 데이터의 핵심인 \*\*데이터베이스(DB)\*\*로 넘어가게 될 거야.

## **5\. 바로 해볼 수 있는 실험**

### **실험 1: 특정 사이트의 SSL 인증서 상세 정보 확인**

\# openssl 명령어로 특정 도메인의 인증서 체인과 만료일 확인  
openssl s\_client \-connect google.com:443 \-showcerts

* **포인트**: Certificate chain 부분에서 0, 1, 2번 인증서가 어떻게 계층 구조(CA)를 이루고 있는지 눈으로 확인해 봐.

### **실험 2: curl로 TLS 버전 강제 지정 테스트**

\# TLS 1.3으로 접속 시도  
curl \-I \-v \--tlsv1.3 \[https://www.google.com\](https://www.google.com)  
\# 만약 서버가 지원하지 않는 아주 낮은 버전(예: 1.0)으로 시도하면 어떻게 될까?  
curl \-I \-v \--tlsv1.0 \[https://www.google.com\](https://www.google.com)

* **포인트**: 핸드쉐이크 과정(\* ALPN, offering h2, \* TLSv1.3 (OUT), TLS handshake) 로그를 직접 읽어보며 연결 수립 과정을 체감해 봐.

### **실험 3: HSTS 헤더 확인**

\# 대형 포털 사이트의 응답 헤더 확인  
curl \-I \[https://www.github.com\](https://www.github.com)

* **포인트**: strict-transport-security 헤더가 있는지 찾아봐. 이 헤더가 있는 사이트는 브라우저 주소창에 http://로 입력해도 강제로 https://로 전환돼.

## 이야기: 창고가 필요해

클라우드빵집이 성장하면서 데이터가 쌓이기 시작했다.

- **주문 DB**: 초당 수천 건의 읽기/쓰기
- **레시피 파일**: 10명의 직원이 함께 편집
- **상품 이미지**: 수십만 장의 사진과 동영상

민준은 세 가지를 다 같은 디스크에 넣으려 했다.

> "다 한 곳에 넣으면 안 돼?"

지수가 고개를 저었다.

> "용도가 달라. 빠른 DB용 창고, 여럿이 공유하는 창고, 인터넷에 공개하는 창고.  
> 각각 다른 종류의 스토리지를 써야 해."

## 🔑 핵심 개념: 스토리지 3대장 완전 비교 (⭐⭐ 최빈출!)

| 구분 | **블록 스토리지 (Block)** | **파일 스토리지 (File)** | **오브젝트 스토리지 (Object)** |
|------|------------------------|----------------------|------------------------------|
| **저장 단위** | 블록 (Block) | 폴더 & 파일 | 객체 (Object) |
| **접근 방식** | iSCSI, FC, NVMe-oF | NFS, SMB | HTTP/HTTPS |
| **내부 구조** | 블록 직접 접근 | 파일 서버 | HTTPS 서버 + KVS |
| **경로 예시** | `/dev/sda` | `//182.168.0.1/data` | `https://example.com/storage` |
| **성능** | **고속** (블록 직접 접근) | 중간 (폴더 검색 필요) | **저속** (HTTP API) |
| **동시 접근** | 서버 **1대만** | **여러 서버** 공유 가능 | 인터넷으로 **무제한** |
| **주요 용도** | DB, OS (부팅 디스크) | 파일 공유 | CDN (사진, 동영상 저장) |
| **NCP 서비스** | 서버 Block Storage | **NAS** | **Object Storage** |

## 비유로 이해하기 🍞

```
블록 스토리지 (Block)
  = 주방 조리대 위의 작업 공간
  나만 쓰는 빠른 공간. 여럿이 쓰면 충돌.
  → OS 설치, DB 저장

파일 스토리지 (File)
  = 직원 공용 선반
  폴더/파일 구조로 정리. 여럿이 함께 쓰기 가능.
  → 레시피 파일 공유, 팀 작업 공간

오브젝트 스토리지 (Object)
  = 외부 창고 (인터넷으로 접근)
  URL로 아무나 접근 가능. 용량 무제한.
  → 상품 이미지, 영상, 정적 웹사이트
```

## 접근 방식 상세

### 블록 스토리지 접근
```
서버 OS
  └─ /dev/sda (블록 디바이스로 직접 인식)
       └─ OS가 파일시스템으로 포맷해서 사용
```
- 가장 빠른 I/O
- OS 입장에서는 그냥 "로컬 디스크"처럼 보임

### 파일 스토리지 접근
```
서버 A ──NFS──→ NAS 서버 (//192.168.0.1/shared)
서버 B ──NFS──→ NAS 서버 (공유!)
서버 C ──SMB──→ NAS 서버 (Windows용)
```
- 네트워크를 통해 폴더/파일 형태로 접근
- NFS: Linux 계열 / SMB(CIFS): Windows 계열

### 오브젝트 스토리지 접근
```
앱 서버 ──HTTPS REST API──→ Object Storage
웹 브라우저 ──URL──→ 이미지 파일
CDN ──Origin Pull──→ Object Storage
```
- HTTP/HTTPS로 REST API 호출
- URL만 있으면 어디서든 접근 가능

## 🔑 핵심 개념: 파일 시스템(FileSystem)이란?

파일 시스템은 OS에서 보조기억장치에 저장되는 파일을 관리하는 시스템의 통칭이다. 파일을 저장할 **클러스터**라는 저장 공간 단위를 만들고, 파일명에 대한 규칙과 데이터의 저장/검색을 관장한다.

### 저장 방식의 종류

| 방식 | 구조 | 특징 | 한계 |
|------|------|------|------|
| **연속 할당** | 시작 위치 + 크기로 저장 | 단순하고 고전적 | 저장 장치 크기 제약, 공간 낭비 심함 |
| **연결(Linked) 할당** | 링크 구조 (각 블록이 다음 블록 포인터 보유) | 비연속 공간 사용 가능 | Sequential search만 빠름, 포인터 메모리 낭비 |
| **색인(Indexed) 할당** | Index block에 해당 파일의 블록 리스트 저장 | Random access 가능 | **현재 대부분의 파일 시스템이 채택** |

> ⚠️ **시험 포인트**  
> 최근의 많은 파일 시스템은 **비연속 할당 방식 + 색인 할당 방식**을 사용한다.  
> Index block을 통해 파일의 Random access를 구현하여, 기존 링크 방식의 순차 탐색 단점을 해결했다.

## 파일 시스템 종류 (Linux / Windows)

### Linux 파일 시스템

NCP 서버에서 OS가 설치된 기본 스토리지는 **XFS**로 구성된다. 그 외에도 ext2, ext3, ext4, JFS 등 다양한 파일 시스템을 선택할 수 있다.

| 파일 시스템 | 단일 최대 파일 크기 | 전체 파일 시스템 크기 | 특징 |
|------------|-------------------|-------------------|------|
| **ext2** | 16GB ~ 2TB | 2TB ~ 32TB | Journal 미사용 → write 성능 우수하나 안정성 낮음. fsck로 복구 가능. |
| **ext3** | 16GB ~ 2TB | 2TB ~ 32TB | ext2에 **journaling 기능 추가** → 안정성 향상, 단 성능 저하 |
| **ext4** | 16GB ~ **16TB** | ~**1EB (1024PB)** | 저널 체크섬, Extent/지연 블록 할당으로 read/write 성능 향상. fast fsck 지원. CPU 리소스 소모 낮음. |
| **XFS** | - | - | NCP 기본 OS 스토리지. 고성능 64비트 저널링 파일 시스템. |

### Windows 파일 시스템

| 파일 시스템 | 최대 볼륨 크기 | 단일 최대 파일 크기 | 특징 |
|------------|-------------|-------------------|------|
| **NTFS** | 2⁶⁴-1 클러스터 (기본 클러스터 4KB → **16TB**) | **16TB** | 시스템 고장 및 디스크 손상 복구 기능 내장 |

## 로컬 파일 시스템의 한계

지수가 설명을 이어갔다.

> "로컬 파일 시스템은 원래 **로컬에 장착된 하드디스크**에 맞춰 설계된 시스템이야. 고정된 크기의 섹터와 블록으로 저장하지. 근데 클라우드 시대에는 한계가 있어."

로컬 파일 시스템의 세 가지 근본적 한계:

1. **용량의 한계** — 물리 디스크 크기 이상으로 확장 불가
2. **장애 복구의 한계** — 디스크 고장 시 데이터 손실
3. **다양한 접근 경로 미제공** — 단일 서버에서만 접근 가능

---

## 파일 시스템의 진화: 로컬 → 네트워크 → 분산

스토리지 기술이 어떻게 발전해왔는지 이해하면 Object Storage의 존재 이유가 명확해진다.

```
[1단계] 로컬 파일 시스템
  → 로컬 하드디스크 직접 접근
  → 한계: 용량 제한, 장애 취약, 단일 접근

[2단계] 네트워크 파일 시스템 (NFS / CIFS)
  → 네트워크를 통해 원격 파일 시스템에 접근
  → 사용자가 원격 컴퓨터의 파일을 로컬처럼 처리 가능
  → 한계: 용량 확장 시 고가의 인프라 비용 발생

[3단계] 분산 파일 시스템 (HDFS, CODA 등)
  → 파일을 '쪼개서(Split)' 여러 노드에 분산 + 복제
  → 용량 한계 극복, 장애 복구 가능, 다양한 접근 경로 제공
  → Namenode가 메타데이터 관리, Datanode가 블록 저장

[4단계] 오브젝트 스토리지 (분산 파일 시스템의 발전형)
  → 파일을 '쪼개지 않고' 통째로 여러 노드에 복제
  → 메타데이터 서버(MDS)가 Owner map 관리
  → 용량 한계 없음, 장애에도 강함
```

> ⚠️ **시험 포인트**  
> **분산 파일 시스템(HDFS 등)**: 파일을 **쪼개서** 분산 저장  
> **오브젝트 스토리지**: 파일을 **쪼개지 않고** 통째로 복제

### 분산 파일 시스템의 종류

오브젝트 스토리지 방식으로 작동하는 분산 파일 시스템에는 다음과 같은 것들이 있다:

- 아마존 S3
- 앤드루 파일 시스템 (AFS)
- DCE 분산 파일 시스템
- 넷웨어 코어 프로토콜
- 코다 (CODA) — 분산형 고장 방지 파일 시스템
- InterMezzo
- Tahoe-LAFS
- 9P

## NCP 스토리지 서비스 매핑

| NCP 서비스 | 스토리지 타입 | 주요 특징 |
|-----------|-------------|---------|
| **서버 Block Storage** | Block | 서버 부팅 디스크, DB 전용 |
| **NAS** | File | NFS/CIFS, 여러 서버 공유, 병렬 가상 파일 시스템 |
| **Object Storage** | Object | S3 호환, 정적 웹 호스팅 |
| **Archive Storage** | Object | 장기 보관, OpenStack Swift |

## 📝 시험 대비 체크리스트

- [ ] Block: 고속, OS/DB용, **서버 1대만** 연결, 접근: iSCSI/FC/NVMe-oF
- [ ] File(NAS): NFS/SMB, **여러 서버 공유**, 내부 구조: 파일 서버
- [ ] Object: HTTP REST API, CDN/대용량 미디어, 내부 구조: KVS
- [ ] Block 접근 경로: `/dev/sda` 형태 (블록 디바이스)
- [ ] File 접근 경로: `//IP/폴더` 형태 (네트워크 경로)
- [ ] Object 접근 경로: `https://...` 형태 (URL)
- [ ] Linux 기본 파일 시스템: **XFS**
- [ ] Windows 파일 시스템: **NTFS** (최대 볼륨 16TB, 클러스터 기본 4KB)
- [ ] ext4: 단일 파일 최대 **16TB**, 전체 파일 시스템 최대 **1EB**
- [ ] ext2: journal 없음 → 빠르지만 안정성 낮음
- [ ] ext3: ext2 + journaling → 안정성 향상, 성능 저하
- [ ] 로컬 FS 3대 한계: **용량 한계 / 장애 복구 한계 / 접근 경로 제한**
- [ ] 분산 FS: 파일 **쪼개서** 분산 (HDFS)
- [ ] Object Storage: 파일 **통째로** 복제, MDS가 메타데이터 관리

## 🧠 암기 핵심 문장

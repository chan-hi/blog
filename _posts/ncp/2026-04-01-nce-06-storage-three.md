## 이야기: 창고가 필요해

클라우드빵집이 성장하면서 데이터가 쌓이기 시작했다.

- **주문 DB**: 초당 수천 건의 읽기/쓰기
- **레시피 파일**: 10명의 직원이 함께 편집
- **상품 이미지**: 수십만 장의 사진과 동영상

민준은 세 가지를 다 같은 디스크에 넣으려 했다.

> "다 한 곳에 넣으면 안 돼?"

지수가 고개를 저었다.

> "용도가 달라. 빠른 DB용 창고, 여럿이 공유하는 창고, 인터넷에 공개하는 창고.  
> 각각 다른 종류의 스토리지를 써야 해."

---

## 🔑 핵심 개념: 스토리지 3대장 완전 비교 (⭐⭐ 최빈출!)

| 구분 | **블록 스토리지 (Block)** | **파일 스토리지 (File)** | **오브젝트 스토리지 (Object)** |
|------|------------------------|----------------------|------------------------------|
| **저장 단위** | 블록 (Block) | 폴더 & 파일 | 객체 (Object) |
| **접근 방식** | iSCSI, FC, NVMe-oF | **NFS, SMB** | **HTTP/HTTPS (REST API)** |
| **경로 예시** | `/dev/sda` | `//182.168.0.1/data` | `https://kr.object.ncloudstorage.com/...` |
| **성능** | **최고속** (직접 블록 접근) | 중간 | **저속** (HTTP 오버헤드) |
| **동시 접근** | 서버 **1대만** | **여러 서버** 공유 가능 | 인터넷으로 **무제한** |
| **주요 용도** | OS 부팅 디스크, DB | 공유 파일 서버 | CDN, 대용량 미디어/이미지 |
| **NCP 서비스** | 서버 Block Storage | **NAS** | **Object Storage** |

---

## 비유로 이해하기 🍞

```
블록 스토리지 (Block)
  = 주방 조리대 위의 작업 공간
  나만 쓰는 빠른 공간. 여럿이 쓰면 충돌.
  → OS 설치, DB 저장

파일 스토리지 (File)
  = 직원 공용 선반
  폴더/파일 구조로 정리. 여럿이 함께 쓰기 가능.
  → 레시피 파일 공유, 팀 작업 공간

오브젝트 스토리지 (Object)
  = 외부 창고 (인터넷으로 접근)
  URL로 아무나 접근 가능. 용량 무제한.
  → 상품 이미지, 영상, 정적 웹사이트
```

## 접근 방식 상세

### 블록 스토리지 접근
```
서버 OS
  └─ /dev/sda (블록 디바이스로 직접 인식)
       └─ OS가 파일시스템으로 포맷해서 사용
```
- 가장 빠른 I/O
- OS 입장에서는 그냥 "로컬 디스크"처럼 보임

### 파일 스토리지 접근
```
서버 A ──NFS──→ NAS 서버 (//192.168.0.1/shared)
서버 B ──NFS──→ NAS 서버 (공유!)
서버 C ──SMB──→ NAS 서버 (Windows용)
```
- 네트워크를 통해 폴더/파일 형태로 접근
- NFS: Linux 계열 / SMB(CIFS): Windows 계열

### 오브젝트 스토리지 접근
```
앱 서버 ──HTTPS REST API──→ Object Storage
웹 브라우저 ──URL──→ 이미지 파일
CDN ──Origin Pull──→ Object Storage
```
- HTTP/HTTPS로 REST API 호출
- URL만 있으면 어디서든 접근 가능

## 🔑 핵심 개념: 파일 시스템의 진화

스토리지 기술이 어떻게 발전해왔는지 이해하면 Object Storage의 존재 이유가 명확해진다.

| 방식 | 특징 | 한계 |
|------|------|------|
| **연속 할당** | 연이어 저장 | 공간 낭비 심함 |
| **연결(Linked) 할당** | 링크 구조로 쪼개서 저장 | 순차 접근만 빠름 |
| **색인(Indexed) 할당** | 인덱스 블록으로 관리 | **현재 대부분의 파일 시스템이 사용** |

```
전통적 NFS/CIFS (네트워크 파일 시스템)
  → 용량 확장에 고가의 인프라 필요

분산 파일 시스템 (HDFS 등)
  → 파일을 '쪼개서(Split)' 여러 노드에 분산 + 복제

오브젝트 스토리지
  → 파일을 '쪼개지 않고' 통째로 여러 노드에 복제
  → 용량 한계 없음, 장애에도 강함
```

> ⚠️ **시험 포인트**  
> **분산 파일 시스템**: 파일을 **쪼개서** 분산  
> **오브젝트 스토리지**: 파일을 **쪼개지 않고** 통째로 복제

## NCP 스토리지 서비스 매핑

| NCP 서비스 | 스토리지 타입 | 주요 특징 |
|-----------|-------------|---------|
| **서버 Block Storage** | Block | 서버 부팅 디스크, DB 전용 |
| **NAS** | File | NFS/CIFS, 여러 서버 공유 |
| **Object Storage** | Object | S3 호환, 정적 웹 호스팅 |
| **Archive Storage** | Object | 장기 보관, OpenStack swift |

---

## 📝 시험 대비 체크리스트

- [ ] Block: 고속, OS/DB용, **서버 1대만** 연결
- [ ] File(NAS): NFS/SMB, **여러 서버 공유**
- [ ] Object: HTTP REST API, CDN/대용량 미디어
- [ ] Block 접근: `/dev/sda` 형태 (블록 디바이스)
- [ ] File 접근: `//IP/폴더` 형태 (네트워크 경로)
- [ ] Object 접근: `https://...` 형태 (URL)
- [ ] 분산 FS: 파일 **쪼개서** 분산
- [ ] Object Storage: 파일 **통째로** 복제

## 핵심 개념 1: Docker란?

**Docker**는 컨테이너 기반의 오픈소스 가상화 플랫폼이다.

2013년 3월, Python 컨퍼런스(PyCon)에서 **Solomon Hykes**가 "The Future of Linux Container" 세션을 통해 처음 세상에 소개했다. 당시 발표는 기존에 있던 다양한 리소스 격리 기술과 네트워크, 파일시스템 기술들을 잘 조합하고 편의성을 높인 것으로, 많은 개발자와 엔지니어들에게 즉각적인 호응을 받았다.

Docker는 리눅스 커널의 두 가지 핵심 기능을 활용한다.

- **Linux Namespace**: 각 프로세스가 파일시스템 마운트, 네트워크, 유저, 호스트명 등에 대해 시스템에 독립적인 뷰를 제공한다.
- **Linux Control Groups (cgroups)**: 프로세스가 소비할 수 있는 리소스 양(CPU, 메모리, I/O, 네트워크 대역폭 등)을 제한한다.

여기에 **Union Filesystem**, **Overlay Network**, **Docker Hub** 등의 기술을 더해, 컨테이너를 쉽고 빠르게 만들고 배포할 수 있게 해준다.

> Docker = Linux의 Namespace + cgroups + Union Filesystem 등을 조합한 컨테이너 관리 플랫폼

## 핵심 개념 2: VM vs 컨테이너 (Container)

### 가상화 방식의 차이

**Virtualization(VM)**은 단일 시스템에서 여러 OS가 동시에 실행되는 방식이다. 반면 **Container**는 동일한 OS 커널을 공유하며, 시스템의 나머지 부분으로부터 프로세스를 격리하는 방식이다.

기존 가상화에서 많이 쓰이던 OS 전체 가상화와 달리, 컨테이너는 하나의 OS 커널 위에 각각의 개별 프로세스와 그에 따른 환경을 격리화한다. 그 결과 OS 가상화보다 오버헤드가 적고 성능 손실이 적다.

| 항목 | **VM (가상 머신)** | **Container** |
|------|-----------------|--------------|
| **구조** | 하이퍼바이저 → Guest OS → 앱 | Host OS → 컨테이너 런타임 → 앱 |
| **Guest OS** | 각 VM마다 **별도 OS** | **없음** (Host OS 커널 공유) |
| **무게** | 무거움 (OS 포함) | **가벼움** |
| **성능** | 오버헤드 있음 | **손실 최소** |
| **격리 방식** | 하드웨어 가상화 | **Namespace, Cgroup** 기술 |
| **부팅 시간** | 분 단위 | **초 단위** |

```
[VM 구조]
Physical Server
  └─ Hypervisor
       ├─ VM1: Guest OS + Bins/Libs + App A
       ├─ VM2: Guest OS + Bins/Libs + App B
       └─ VM3: Guest OS + Bins/Libs + App C
       (각각 OS가 따로 있어 무거움)

[Container 구조]
Physical Server
  └─ Host OS + Docker Engine
       ├─ Container 1: Bins/Libs + App A (OS 커널 공유)
       ├─ Container 2: Bins/Libs + App B (OS 커널 공유)
       └─ Container 3: Bins/Libs + App C (OS 커널 공유)
       (OS를 공유하니 가볍고 빠름!)
```

## 핵심 개념 3: Docker 이미지 레이어 구조

Docker의 가장 큰 특징 = **레이어(Layer) 구조**.

Docker는 **Union Filesystem**이라는 기술을 이용해 여러 Layer를 하나의 파일시스템으로 인식할 수 있도록 한다. 이미지에 수정 사항이 생기면 Layer를 추가하여 변경분만 기록하고, 최상단에는 항상 **R/W(읽기/쓰기) 레이어**가 존재한다.

```
[Docker 이미지 레이어]

Layer 4: R/W (읽기/쓰기) ← 컨테이너 실행 시 추가
Layer 3: 앱 소스코드    ← Read Only
Layer 2: Python 3.10   ← Read Only
Layer 1: Ubuntu 22.04  ← Read Only (베이스 이미지)
```

- **베이스 레이어**: 변경 불가, 읽기 전용(Read-Only)
- **최상단 레이어**: 컨테이너 실행 시 추가되는 읽기/쓰기(R/W) 레이어
- **Union Filesystem**: 여러 레이어를 **하나의 파일시스템처럼** 보이게 하는 기술

### Dockerfile 빌드와 레이어의 관계

Dockerfile을 빌드하여 Docker 이미지를 작성하면, **Dockerfile 명령 한 줄마다** 이미지(레이어)가 작성된다. 실제로 빌드 로그를 확인하면 명령어 한 줄이 실행될 때마다 새로운 레이어가 쌓이는 것을 확인할 수 있다.

### 레이어 구조의 장점

```
이미지 A: Ubuntu + Apache + WebApp A
이미지 B: Ubuntu + Apache + WebApp B
           ↑ 공통 레이어 공유!

→ Ubuntu + Apache 레이어는 한 번만 저장
→ 디스크 절약, 이미지 배포 속도 향상
```

공통의 베이스 이미지를 바탕으로 여러 이미지를 작성한 경우, 베이스 이미지의 레이어가 공유된다. 공통 부분은 컨테이너들이 함께 사용함으로써 디스크 용량을 효율적으로 이용할 수 있다.

## 핵심 개념 4: Dockerfile 명령어 (시험 빈출!)

**Dockerfile**은 Docker 상에서 작동시킬 컨테이너의 구성 정보를 기술하기 위한 파일이다. Dockerfile을 통해 Docker 이미지를 만드는 설계도 역할을 한다.

### 주요 명령어 전체 목록

| 명령어 | 역할 |
|--------|------|
| **FROM** | 베이스(기본) 이미지 지정 |
| **MAINTAINER** | 작성자 지정 |
| **RUN** | 컨테이너에서 명령 실행 (이미지 빌드 시) |
| **CMD** | 데몬(컨테이너 시작 시) 기본 실행 명령 지정 |
| **LABEL** | 라벨 설정 |
| **EXPOSE** | 컨테이너가 외부에 공개하는 포트 선언 |
| **ENV** | 환경 변수 설정 |
| **ADD** | 파일 추가 |
| **COPY** | 파일 및 디렉터리를 컨테이너에 복사 |
| **VOLUME** | 볼륨 마운트 |
| **ENTRYPOINT** | 데몬 실행 |
| **USER** | 사용자 설정 |
| **WORKDIR** | 컨테이너의 작업 폴더(디렉터리) 지정 |
| **ONBUILD** | 빌드 후 실행 명령 |

### 핵심 명령어 요약

| 명령어 | 역할 | 예시 |
|--------|------|------|
| **FROM** | 베이스 이미지 지정 | `FROM ubuntu:22.04` |
| **WORKDIR** | 컨테이너 내부 작업 디렉터리 설정 | `WORKDIR /app` |
| **COPY** | 호스트 파일을 컨테이너 안으로 복사 | `COPY . /app` |
| **RUN** | 이미지 빌드 시 실행할 명령어 | `RUN pip install -r requirements.txt` |
| **EXPOSE** | 외부와 통신할 포트 선언 | `EXPOSE 8080` |
| **CMD** | 컨테이너 **시작 시** 실행할 기본 명령 | `CMD ["python", "app.py"]` |

```dockerfile
# 클라우드빵집 웹앱 Dockerfile 예시
FROM python:3.10-slim          # 베이스 이미지

WORKDIR /app                   # 작업 폴더 설정

COPY requirements.txt .        # 의존성 파일 복사
RUN pip install -r requirements.txt  # 설치 (빌드 시 실행)

COPY . .                       # 소스코드 복사

EXPOSE 8000                    # 포트 선언

CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]  # 시작 명령
```

> **RUN vs CMD 차이**
> `RUN`: **이미지 빌드 시** 실행 (패키지 설치 등)
> `CMD`: **컨테이너 시작 시** 실행 (앱 구동)

## 핵심 개념 5: 컨테이너 상태 변화

도커 명령의 실행에 의해 컨테이너의 상태가 변한다. 컨테이너는 생성, 실행, 정지, 일시정지, 삭제 상태를 취할 수 있다.

```
생성(docker container create)
    ↓
실행(start / run)
    ↓
    ├─ 일시정지(pause)
    │       ↓
    │   재시작(unpause)
    │
    └─ 정지(stop / kill)
            ↓
         삭제(rm)
```

## 핵심 개념 6: 이미지 생성부터 배포까지

```
1. 이미지 생성   → Dockerfile 작성 후 docker build로 이미지 빌드
2. 레지스트리 등록 → docker push로 Container Registry에 이미지 업로드
3. 레지스트리 Pull → Container Registry에서 이미지 다운로드
4. 배포 업데이트  → kubectl apply로 쿠버네티스 클러스터에 배포
```

각 단계를 좀 더 자세히 보면:

1. Docker 이미지를 빌드하기 위해 **Dockerfile을 작성**하고, 클라이언트가 Dockerfile과 함께 이미지 빌드를 요청한다.
2. 클라이언트가 이미지 push를 요청하면 서버는 로컬의 이미지를 **Docker 레지스트리에 업로드**한다.
3. 클라이언트가 컨테이너를 실행할 Docker 서버에 컨테이너 실행을 요청한다.
4. 다운로드한 Docker 이미지로 컨테이너를 생성하여 실행한다.

## 핵심 개념 7: NCP Container Registry

### Docker Registry란?

**Docker Registry**는 컨테이너의 바탕이 되는 Docker 이미지를 공개 및 공유하기 위한 레지스트리 기능이다. Docker의 공식 레지스트리 서비스인 **Docker Hub**도 Docker Registry를 사용한다.

### NCP Container Registry

NCP에서 제공하는 **프라이빗 Docker 이미지 저장소**이다. 도커 컨테이너(Docker Container) 이미지를 손쉽게 저장, 관리 및 배포할 수 있는 네이버 클라우드 플랫폼 서비스다.

| 항목 | 내용 |
|------|------|
| **저장 위치** | **Object Storage 기반** (이미지를 Object Storage에 저장, 안정성 및 확장성 제공) |
| **호환성** | **Docker Registry V2 (HTTP API V2)** 와 호환 |
| **CLI 지원** | 기존의 **Docker CLI**를 이용하여 컨테이너 관리 가능 |
| **연동** | CI/CD, **NKS(Kubernetes Service)**와 연동으로 생산성 향상 |
| **보안** | **CVE 기반** 컨테이너 취약점 분석으로 취약점 유무 확인 |
| **엔드포인트** | **Public/Private Endpoint** 제공으로 내부 혹은 외부 공개 여부 선택 가능 |

### 주요 기능 상세

- **Object Storage 연동**: Container Registry는 Object Storage와 연동하여 프라이빗 Docker Container 이미지에 대한 Push, Pull 기능을 제공한다.
- **Private Endpoint**: 네이버 클라우드 플랫폼의 네트워크 내에서만 사용할 수 있는 Private Endpoint를 제공하여, 모든 네트워크 트래픽이 네이버 클라우드 플랫폼의 네트워크로만 제한되므로 보안을 향상시킨다.
- **CVE 기반 보안 분석**: CVE(Common Vulnerabilities and Exposures, 취약점 표준 코드) 기반으로 컨테이너 취약점을 분석한다.

> **시험 포인트**
> Container Registry는 **Object Storage에** 이미지를 저장한다!
> CVE = Common Vulnerabilities and Exposures (공개 취약점 표준 코드)

```
[Container Registry 활용 흐름]

개발자 PC
  └─ docker build → 이미지 생성
  └─ docker push  → Container Registry(Object Storage)에 업로드

NKS (Kubernetes Cluster)
  └─ kubectl apply → 매니페스트로 배포
  └─ Pod 배포 시 Container Registry에서 이미지 pull
```

## 시험 대비 체크리스트

- [ ] Docker: 2013년 3월 PyCon에서 **Solomon Hykes**가 "The Future of Linux Container"로 소개
- [ ] Docker = Linux **Namespace** + **cgroups** + **Union Filesystem** 기반 플랫폼

- [ ] 베이스 레이어: **Read-Only**, 최상단은 R/W 레이어
- [ ] Union Filesystem: 여러 레이어를 하나처럼
- [ ] FROM: 베이스 이미지 지정
- [ ] MAINTAINER: 작성자 지정
- [ ] WORKDIR: 작업 디렉터리 지정
- [ ] COPY: 호스트 → 컨테이너 파일 복사
- [ ] ADD: 파일 추가
- [ ] RUN: **빌드 시** 명령 실행
- [ ] EXPOSE: 포트 선언
- [ ] CMD: **컨테이너 시작 시** 기본 명령 실행
- [ ] ENTRYPOINT: 데몬 실행
- [ ] ENV: 환경 변수 설정
- [ ] VOLUME: 볼륨 마운트
- [ ] ONBUILD: 빌드 후 실행 명령
- [ ] 컨테이너 상태: 생성 → 실행 → 일시정지/정지 → 삭제

- [ ] Container Registry: **CVE 기반 취약점 분석**

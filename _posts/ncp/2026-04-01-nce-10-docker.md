## 🔑 핵심 개념 1: VM vs 컨테이너 (Container)

| 항목 | **VM (가상 머신)** | **Container** |
|------|-----------------|--------------|
| **구조** | 하이퍼바이저 → Guest OS → 앱 | Host OS → 컨테이너 런타임 → 앱 |
| **Guest OS** | 각 VM마다 **별도 OS** | **없음** (Host OS 커널 공유) |
| **무게** | 무거움 (OS 포함) | **가벼움** |
| **성능** | 오버헤드 있음 | **손실 최소** |
| **격리 방식** | 하드웨어 가상화 | **Namespace, Cgroup** 기술 |
| **부팅 시간** | 분 단위 | **초 단위** |

```
[VM 구조]
Physical Server
  └─ Hypervisor
       ├─ VM1: Guest OS + App A
       ├─ VM2: Guest OS + App B
       └─ VM3: Guest OS + App C
       (각각 OS가 따로 있어 무거움)

[Container 구조]
Physical Server
  └─ Host OS + Container Runtime
       ├─ Container 1: App A (OS 커널 공유)
       ├─ Container 2: App B (OS 커널 공유)
       └─ Container 3: App C (OS 커널 공유)
       (OS를 공유하니 가볍고 빠름!)
```

> 💡 **Docker** = Namespace, Cgroup 등 Linux 기술을 조합하여  
> 컨테이너를 쉽게 만들고 관리할 수 있게 해주는 오픈소스 플랫폼

## 🔑 핵심 개념 2: Docker 이미지 레이어 구조

Docker의 가장 큰 특징 = **레이어(Layer) 구조**.

```
[Docker 이미지 레이어]

Layer 4: R/W (읽기/쓰기) ← 컨테이너 실행 시 추가
Layer 3: 앱 소스코드    ← Read Only
Layer 2: Python 3.10   ← Read Only
Layer 1: Ubuntu 22.04  ← Read Only (베이스 이미지)
```

- **베이스 레이어**: 변경 불가, 읽기 전용(Read-Only)
- **최상단 레이어**: 컨테이너 실행 시 추가되는 읽기/쓰기(R/W) 레이어
- **Union Filesystem**: 여러 레이어를 **하나의 파일시스템처럼** 보이게 하는 기술

### 레이어 구조의 장점
```
이미지 A: Ubuntu + Python + 앱A
이미지 B: Ubuntu + Python + 앱B
           ↑ 공통 레이어 공유!

→ Ubuntu + Python 레이어는 한 번만 저장
→ 디스크 절약, 이미지 배포 속도 향상
```

## 🔑 핵심 개념 3: Dockerfile 명령어 (⭐ 빈출!)

Dockerfile = Docker 이미지를 만드는 **설계도 파일**

| 명령어 | 역할 | 예시 |
|--------|------|------|
| **FROM** | 베이스 이미지 지정 | `FROM ubuntu:22.04` |
| **WORKDIR** | 컨테이너 내부 작업 디렉터리 설정 | `WORKDIR /app` |
| **COPY** | 호스트 파일을 컨테이너 안으로 복사 | `COPY . /app` |
| **RUN** | 이미지 빌드 시 실행할 명령어 | `RUN pip install -r requirements.txt` |
| **EXPOSE** | 외부와 통신할 포트 선언 | `EXPOSE 8080` |
| **CMD** | 컨테이너 **시작 시** 실행할 기본 명령 | `CMD ["python", "app.py"]` |

```dockerfile
# 클라우드빵집 웹앱 Dockerfile 예시
FROM python:3.10-slim          # 베이스 이미지

WORKDIR /app                   # 작업 폴더 설정

COPY requirements.txt .        # 의존성 파일 복사
RUN pip install -r requirements.txt  # 설치 (빌드 시 실행)

COPY . .                       # 소스코드 복사

EXPOSE 8000                    # 포트 선언

CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]  # 시작 명령
```

> ⚠️ **RUN vs CMD 차이**  
> `RUN`: **이미지 빌드 시** 실행 (패키지 설치 등)  
> `CMD`: **컨테이너 시작 시** 실행 (앱 구동)

## 🔑 핵심 개념 4: 컨테이너 라이프사이클

```
생성(create)
    ↓
실행(start / run)
    ↓
    ├─ 일시정지(pause)
    │       ↓
    │   재시작(unpause)
    │
    └─ 정지(stop / kill)
            ↓
         삭제(rm)
```

### 표준 배포 파이프라인

```
1. docker build   → Dockerfile 읽어서 이미지 생성
2. docker push    → 이미지를 Container Registry에 업로드
3. docker pull    → Registry에서 이미지 다운로드
4. docker run     → 컨테이너 실행 (쿠버네티스 환경이면 apply)
```

## 🔑 핵심 개념 5: NCP Container Registry

NCP에서 제공하는 **프라이빗 Docker 이미지 저장소**.

| 항목 | 내용 |
|------|------|
| **저장 위치** | **Object Storage 기반** (이미지를 Object Storage에 저장) |
| **보안 기능** | **CVE 기반 보안 취약점 분석** |
| **연동** | NKS(Kubernetes Service)와 쉽게 연동 |

> ⚠️ **시험 포인트**  
> Container Registry는 **Object Storage에** 이미지를 저장한다!  
> CVE = Common Vulnerabilities and Exposures (공개 취약점 목록)

```
[Container Registry 활용 흐름]

개발자 PC
  └─ docker build → 이미지 생성
  └─ docker push → Container Registry에 업로드

NKS (Kubernetes Cluster)
  └─ Pod 배포 시 Container Registry에서 이미지 pull
```

## 📝 시험 대비 체크리스트

- [ ] VM: 각각 별도 Guest OS 있음
- [ ] Container: **Host OS 커널 공유**, Guest OS 없음
- [ ] Docker = Namespace + Cgroup 기반 컨테이너 플랫폼
- [ ] Docker 이미지: **레이어 구조**, 베이스 레이어 **Read-Only**
- [ ] Union Filesystem: 여러 레이어를 하나처럼
- [ ] FROM: 베이스 이미지 지정
- [ ] WORKDIR: 작업 디렉터리
- [ ] COPY: 호스트 → 컨테이너 파일 복사
- [ ] RUN: **빌드 시** 실행
- [ ] EXPOSE: 포트 선언
- [ ] CMD: **컨테이너 시작 시** 실행
- [ ] Container Registry: **Object Storage 기반** 저장
- [ ] Container Registry: **CVE 기반 취약점 분석**

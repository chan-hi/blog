## 핵심 개념: NACL vs ACG 완전 비교 (최빈출!)

| 구분 | **Network ACL (NACL)** | **ACG (Access Control Group)** |
|------|----------------------|-------------------------------|
| **적용 단위** | **Subnet** (서브넷 내 모든 서버 자동 적용) | **서버 NIC** (명시적으로 지정 필요) |
| **규칙 종류** | 허용(Allow) + **차단(Deny) 모두 가능** | **허용(Allow) 규칙만** 가능 |
| **기본 정책** | 기본 **허용** (Deny를 명시적으로 추가) | 기본 **차단** (Allow를 명시적으로 추가) |
| **상태 유지** | **Stateless** (In/Out 각각 별도 설정) | **Stateful** (In 허용 → Out 자동 허용) |
| **규칙 처리** | **우선순위(Rule Number)** 순서대로 처리 | **모든 규칙 평가** 후 허용 여부 결정 |
| **적용 대상** | 대상 Subnet 내 모든 서버에 자동 적용 | 서버 시작 시 ACG 지정, 또는 나중에 연결 |

## Stateless vs Stateful 완벽 이해

가장 많이 헷갈리는 부분. 실무에서도 매우 중요한 개념.

### NACL = Stateless (상태 비저장)

```
[클라이언트 → 서버 HTTP 요청]

클라이언트 ──────[요청 패킷]──────→ 서버  (Inbound 규칙: 허용)
클라이언트 ←──────[응답 패킷]────── 서버  (Outbound 규칙: ???)
                                           ← 별도로 허용해야 함!
```

NACL은 **연결 상태를 기억하지 않는다**.  
요청을 허용했어도, 응답 패킷이 나가려면 **Outbound 규칙도 따로 허용**해야 한다.  
Outbound 규칙에 포함되지 않으면 트래픽이 차단된다.

```
[NACL 설정 예시 - HTTP 서버]
Inbound: TCP 80 허용 (클라이언트 → 서버)
Outbound: TCP 1024-65535 허용 (서버 → 클라이언트 응답)
          ← 이걸 빼먹으면 응답이 안 나감!
```

---

### ACG = Stateful (상태 저장)

```
[클라이언트 → 서버 HTTP 요청]

클라이언트 ──────[요청 패킷]──────→ 서버  (Inbound 규칙: 허용)
클라이언트 ←──────[응답 패킷]────── 서버  (자동 허용! ← Stateful)
```

ACG는 **연결 상태를 기억한다**.  
Inbound로 허용된 트래픽은 응답 패킷이 **자동으로 Outbound 허용**된다.  
Outbound 규칙에 명시하지 않아도 자동으로 적용된다.

```
[ACG 설정 예시 - HTTP 서버]
Inbound: TCP 80 허용
Outbound: (따로 설정 불필요 — 자동 허용됨)
```

## 비유로 이해하기

```
NACL (Stateless):
"들어오는 사람은 허용했어요.
 근데 나가는 사람은 다시 확인해야 해요."
→ 출입 기록 없음. 입장/퇴장 각각 따로 확인.

ACG (Stateful):
"한번 들어오면, 나갈 때는 자동으로 통과예요.
 왔다 갔다 세트로 처리해요."
→ 출입 기록 있음. 입장 허용하면 퇴장도 자동.
```

## 규칙 처리 방식 차이

### NACL: 우선순위(Rule Number) 기반
```
Rule 100: TCP 80 허용
Rule 200: TCP 22 허용
Rule 300: TCP * 차단
Rule *:   전체 차단 (기본값)

→ 숫자 낮은 순서대로 처리. 먼저 매칭되면 이후 규칙 무시.
```

### ACG: 전체 규칙 평가
```
규칙1: TCP 80 허용
규칙2: TCP 22 허용
규칙3: TCP 443 허용

→ 모든 규칙을 다 평가한 뒤, 하나라도 허용 규칙에 걸리면 허용.
  우선순위 없음. AND/OR 평가.
```

## ACG 규칙 설정 상세 (서버 Operation)

ACG에서 규칙을 설정할 때는 **프로토콜 / 접근 소스 / 허용 포트** 세 가지를 지정한다.

### 특정 IP에서 SSH 접근 허용

```
프로토콜  접근 소스          허용 포트
TCP       192.168.77.17      22
```

단일 IP를 지정해 해당 관리자 PC에서만 SSH 접속을 허용하는 가장 기본적인 보안 설정.

### 특정 IP 대역에서 SSH 접근 허용

```
프로토콜  접근 소스          허용 포트
TCP       192.168.77.0/24    22
```

CIDR 표기로 서브넷 전체를 허용. 사무실 네트워크 대역을 통째로 허용할 때 사용.

### ACG 그룹명을 소스로 지정 — 서버 간 SSH 허용

```
프로토콜  접근 소스        허용 포트
TCP       lab1-web-ACG     22
```

`lab1-web-ACG`라는 이름의 ACG에 할당된 서버들 간 SSH 접근을 허용한다.  
IP가 바뀌어도 ACG 그룹 기반으로 자동 관리되므로 동적 환경에서 매우 유용하다.

### UDP 포트 범위 허용

```
프로토콜  접근 소스        허용 포트
UDP       192.168.77.17    22-1025
```

UDP 프로토콜로 22~1025 포트 범위를 한 번에 허용하는 예시.  
포트 단일 지정뿐 아니라 **범위(range) 지정**도 가능하다.

## ACG 제한 사항

| 항목 | 제한 |
|------|------|
| VPC당 ACG 최대 개수 | **500개** |
| NIC당 ACG 적용 최대 | **3개** |
| ACG당 Inbound 규칙 | 최대 **50개** |
| ACG당 Outbound 규칙 | 최대 **50개** |

### ACG 특수 기능: ACG 그룹명을 소스로 지정

```
[일반적인 방화벽]
접근 소스: IP 주소 (예: 192.168.1.0/24)

[ACG의 특수 기능]
접근 소스: 다른 ACG 그룹명 자체도 지정 가능!
예: "web-server-acg"에서 오는 트래픽만 허용
→ IP가 바뀌어도 ACG 그룹 기반으로 자동 관리
```

## 자주 이용하는 포트 번호 (기본 상식)

포트 번호는 세 가지 범위로 구분된다.

| 범위 | 명칭 | 용도 |
|------|------|------|
| 0 ~ 1023 | 웰 노운 포트 | 인터넷상의 표준 서비스용 |
| 1024 ~ 49151 | 등록된 포트 | IANA에 신청하면 이용 가능 |
| 49152 ~ 65535 | 동적/비공식 포트 | 자유롭게 이용 가능 |

자주 시험에 나오는 포트와 프로토콜은 다음과 같다.

| 포트 | TCP/UDP | 서비스/프로토콜 | 역할 |
|------|---------|----------------|------|
| **21** | TCP | FTP | 파일 전송 |
| **22** | TCP/UDP | SSH | 보안 원격 컴퓨터 접속 |
| **23** | TCP | TELNET | 원격 컴퓨터 접속 |
| **25** | TCP/UDP | SMTP | 이메일 전송 |
| **53** | TCP/UDP | DNS | IP 주소 검색 |
| **80** | TCP/UDP | HTTP | 인터넷 통신 |
| **110** | TCP | POP3 | 이메일 수신 |
| **443** | TCP/UDP | HTTPS | 보안 인터넷 통신 |
| **3389** | TCP | RDP | Windows 원격 데스크톱 |

## 실무 적용: 클라우드빵집 방화벽 설계

```
[서브넷 레벨 — NACL]
Inbound:
  Rule 100: TCP 80 허용 (웹 트래픽)
  Rule 110: TCP 443 허용 (HTTPS)
  Rule 900: 전체 차단

Outbound:
  Rule 100: TCP 1024-65535 허용 (응답)
  Rule 900: 전체 차단
  ← Stateless라서 반드시 Outbound도 설정!

[서버 NIC 레벨 — ACG]
Inbound:
  TCP 80 허용 (웹서버용)
  TCP 22 허용 (관리용 SSH, 관리자 IP에서만)
Outbound:
  (Stateful이라 자동 처리)
```

## 시험 대비 체크리스트

- [ ] NACL: **Subnet** 단위 적용
- [ ] ACG: **서버 NIC** 단위 적용
- [ ] NACL: Allow + **Deny 모두 가능**, 기본 허용
- [ ] ACG: **Allow만 가능**, 기본 차단
- [ ] NACL: **Stateless** → In/Out 각각 따로 설정, Outbound 미설정 시 차단
- [ ] ACG: **Stateful** → Inbound 허용 시 Outbound 자동
- [ ] NACL: **우선순위(Rule Number)** 순서 처리
- [ ] ACG: **전체 규칙 평가** 후 결정
- [ ] ACG: VPC당 **500개**, NIC당 **3개**, 규칙 각 **50개**
- [ ] ACG 소스: IP뿐 아니라 **다른 ACG 그룹명** 지정 가능
- [ ] ACG 규칙: 프로토콜(TCP/UDP/ICMP) + 접근 소스 + 허용 포트(단일 또는 범위)
- [ ] SSH = **22**, RDP = **3389**, HTTP = **80**, HTTPS = **443**, DNS = **53**, SMTP = **25**
- [ ] 포트 범위: 웰 노운(0~1023) / 등록된(1024~49151) / 동적(49152~65535)

## 암기 핵심 문장

> "**NACL은 Subnet에 Stateless, ACG는 NIC에 Stateful.**"  
> "**NACL은 Deny도 가능, ACG는 Allow만.**"  
> "**NACL은 번호 순서, ACG는 모든 규칙 평가.**"  
> "**ACG 소스는 IP 또는 다른 ACG 그룹명.**"

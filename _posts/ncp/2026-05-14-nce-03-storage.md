## 핵심 요약

- Object Storage: 버킷 최대 1,000개, 콘솔 업로드 1GB, API 업로드 10TB
- Archive Storage: Segment 최대 5GB, 초과 시 반드시 분할
- NAS: 최소 500GB, 최대 20TB, 100GB 단위 추가, NFS/CIFS 제공
- Backup: 보관 기간 7일~364일, 단일 서버 하루 최대 1TB
- NCP Linux 기본 파일시스템: XFS / ext4 최대 파일: 16TB, 최대 FS: 1EB

## 스토리지 종류 비교

| 구분 | Block Storage | File Storage (NAS) | Object Storage |
|------|--------------|-------------------|----------------|
| 구조 | iSCSI, FC, NVMe-oF | NFS, SMB | HTTPS |
| 저장 방식 | 블록 단위 | 폴더 & 파일 구조 | 객체 단위 |
| 성능 | **고속** | 중간 | 저속 |
| 주요 용도 | DB, OS | 파일 공유 | CDN, 미디어 |

## Object Storage

- 객체 기반 무제한 파일 저장
- **S3 Compatibility API** 지원 (AWS CLI/API와 호환)
- 파일마다 고유한 접근 URL → 인터넷으로 누구나 접근 가능
- 정적 웹사이트 호스팅 가능
- Data Lifecycle: Object Storage → Archive Storage 자동 이동 가능

### 스펙 제한

| 항목 | 값 |
|------|-----|
| 버킷 최대 개수 | **1,000개** |
| 객체명 길이 | **1,024 Byte** 이하 |
| 콘솔 업로드 단일 파일 | **1 GB** 이하 |
| 콘솔 다운로드 단일 파일 | **2 GB** 이하 |
| API 업로드 최대 | **10 TB** 이하 |
| API 다운로드 최대 | 제한 없음 |

### 공개 및 권한 관리

- **버킷 공개**: 누구나 객체 목록 확인 + 파일 업로드 가능
- **파일 공개**: 누구나 파일 열람 및 다운로드 가능
- 권한 관리: 버킷/파일 단위로 NCP 사용자에게 특정 권한 부여

## Archive Storage

- 데이터 아카이빙 및 장기 백업 최적화
- 콘솔, API(swift, s3), CLI, SDK로 관리
- 데이터 최소 보관 기간 없음
- **긴급 다운로드**: 수분 내 다운로드 가능
- 페타바이트(PB) 규모까지 확장 가능

### Object Storage vs Archive Storage

| 항목 | Object Storage | Archive Storage |
|------|---------------|----------------|
| 입출력 속도 | **빠름** | 느림 |
| 저장 비용 | 비쌈 | **저렴** |
| API 비용 | 저렴 | **비쌈** |
| 주요 용도 | 자주 사용하는 데이터 | 장기 보관 |

### 대용량 파일 업로드 (DLO/SLO)

- Segment 최대 크기: **5 GB**
- **5 GB 초과 오브젝트는 반드시 분할하여 업로드**
- **DLO**: Dynamic Large Object
- **SLO**: Static Large Object

## NAS (Network Attached Storage)

- 다수의 서버에서 공유하는 스토리지
- **최소 500 GB ~ 최대 20 TB**
- 추가 단위: **100 GB**
- 프로토콜: **NFS / CIFS**
- 서버 사설 IP를 이용한 ACL 오픈으로 타 계정 서버에도 마운트 가능

### Block Storage vs NAS 비교

| 항목 | Block Storage | NAS |
|------|--------------|-----|
| 공유 여부 | 불가 (단일 서버 전용) | 가능 (다수 서버 공유) |
| 최소 용량 | 10 GB | **500 GB** |
| 용량 단위 | 10 GB | **100 GB** |
| 스냅샷 | 별도 생성 | 자동 스냅샷 제공 |
| 동적 볼륨 조정 | 불가 | 가능 |

## Backup

- 서버 내 파일 및 Preinstall DB 백업 서비스
- **보관 기간**: 7일 ~ **364일(52주)**
- 백업 수행 주기: 일간, 주간, 월간, 일회성
- 최소 용량: **100 GB**
- **단일 서버 하루 최대 백업 용량: 1 TB**

### 백업 방식 비교

| 방식 | 설명 | 복원 방법 |
|------|------|-----------|
| **전체 백업 (Full)** | 모든 데이터 매번 백업 | 최근 전체 백업 1개 |
| **차등 백업 (Differential)** | 마지막 전체 백업 이후 변경분 **전체** | 전체 + 최신 차등 |
| **증분 백업 (Incremental)** | **이전 백업 이후** 변경분만 | 전체 + 모든 증분 |

> 차등 vs 증분: 차등은 복원 시 2개 필요, 증분은 전체+모든 증분 필요 (저장↓, 복원시간↑)

### 소산/교차 존 백업

| 방식 | 설명 |
|------|------|
| **소산 백업** | 로컬 존 백업 데이터를 지리적으로 다른 존에 추가 백업 |
| **교차 존 백업** | 백업 소스가 위치한 존이 아닌 다른 존에 백업 |

## 파일시스템

### 종류별 특징

| 파일시스템 | OS | Journaling | 단일 최대 파일 | 최대 FS 크기 | 특징 |
|-----------|-----|-----------|-------------|------------|------|
| **ext2** | Linux | **없음** | 16GB~2TB | 2TB~32TB | write 빠름, 안전성 낮음 |
| **ext3** | Linux | **있음** | 16GB~2TB | 2TB~32TB | ext2 + journal |
| **ext4** | Linux | **있음 (체크섬)** | 16GB~**16TB** | **~1EB** | 현재 표준, CPU 효율 |
| **NTFS** | **Windows** | 있음 | **16TB** | 16TB | Windows 기본 |
| **XFS** | Linux | 있음 | 큼 | 큼 | **NCP Linux 기본** |

- **ext2**: journal 없음 → write 빠르지만 안전성 낮음
- **ext3**: ext2에 journaling 추가
- **ext4**: 저널 체크섬 + Extent + fast fsck + 타임스탬프 개선
- **XFS**: NCP Linux OS 기본 파일시스템

## 시험 암기 포인트

- **Object Storage 버킷**: 최대 1,000개
- **Object Storage 콘솔 업로드**: 최대 1 GB
- **Object Storage API 업로드**: 최대 10 TB
- **Archive Segment**: 최대 5 GB (초과 시 반드시 분할), DLO/SLO
- **NAS**: 최소 500 GB, 최대 20 TB, 100 GB 단위
- **NAS 프로토콜**: NFS / CIFS
- **Backup 보관**: 7일 ~ 364일(52주), 단일 서버 하루 최대 1 TB
- **ext2**: journal 없음 / **ext3**: journal / **ext4**: journal+체크섬, 최대 파일 16TB, FS 1EB
- **NTFS**: Windows 기본, 최대 파일 16TB
- **XFS**: NCP Linux 기본 파일시스템

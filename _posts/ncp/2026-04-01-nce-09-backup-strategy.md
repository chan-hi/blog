## 핵심 개념 1: NCP Backup 서비스란?

NCP Backup은 서버 내 파일 및 Preinstall DB에 대한 백업을 제공하는 관리형 서비스다.  
콘솔에서 전부 설정할 수 있으며, 설정 단계는 다음과 같다:

```
① 리소스 생성 및 에이전트 설치  →  콘솔
② 저장소 및 정책 생성           →  콘솔
③ 백업 실행을 위한 작업 및 백업 주기 일정 생성  →  콘솔
④ 백업본 소산 설정              →  콘솔
⑤ 백업본 복원                  →  콘솔
```

### 지원 환경

| 구분 | 지원 버전 | 지원 비트 |
|------|----------|----------|
| **MSSQL** | 2005, 2008, 2008R2, 2012, 2014, 2016, 2017, 2019, 2022 | x86, x64 |
| **MySQL** | 5.5, 5.6, 5.7, 8.x | Redhat/CentOS 6~8.x: x86, x64 |
| **MariaDB** | 5.5, 10.0, 10.1~10.6 | Ubuntu 14.04~20.04: x64 |
| **PostgreSQL** | 9.2~14.x | Redhat/CentOS 6.x~8.x: x64, Ubuntu 12.04~22.04: x86, x64 |
| **Windows** | 7, 8, 8.1, 10, 2008, 2008R2, 2016, 2019, 2022 | x86, x64 |
| **Linux (Redhat/CentOS)** | 5.x~8.x | 5.x~7.x: x86, x64 / 8.x: x64 |
| **Linux (Rocky Linux)** | 8.x~9.x | x64 |
| **Linux (Ubuntu)** | 18.04~21.04 | x64 |

### 백업 서비스 유형

NCP Backup은 두 가지 유형으로 제공된다:

- **파일시스템 백업**: 네이버 클라우드 플랫폼에서 제공하는 서버(VM)의 데이터 백업. 파일 또는 디렉토리 단위로 백업 가능.
- **데이터베이스 온라인 백업**: 서버 이미지로 제공하는 버전의 MSSQL, MySQL, PostgreSQL의 온라인 백업. 데이터 복구를 위해 검증된 버전에 한해 지원하며, 향후 다양한 DBMS 버전으로 확대 예정.

## 핵심 개념 2: 유연한 백업 정책

NCP Backup의 핵심 강점 중 하나는 **유연한 정책 설정**이다.

| 항목 | 내용 |
|------|------|
| **백업 수행 주기** | 일간 / 주간 / 월간 / 일회성 중 선택 |
| **보관 기간** | 최소 **7일** ~ 최대 **364일 (52주)** |
| **보관 기간 최소 조건** | 백업 수행 주기의 **최소 2배 이상**으로 설정해야 함 |
| **백업 용량** | 최소 **100GB**에서 수십 **TB**까지 유연하게 사용 |
| **단일 서버 하루 최대 백업** | **1TB** |

> 주의: 보관 기간은 백업 주기의 최소 2배 이상이어야 한다. 예를 들어 주간 백업이라면 보관 기간을 최소 2주(14일) 이상으로 설정해야 한다.

### 소산 백업 & 교차 존 백업

장애 대응을 강화하려면 데이터를 지리적으로 분리하는 것이 핵심이다.

- **소산 백업 (이중화 백업)**: 로컬 존(zone)에 백업한 데이터를 지리적으로 떨어진 **다른 존**에도 추가로 백업. 재해 발생 시 원격 데이터로 복구 가능.
- **교차 존 백업**: 백업을 수행할 때 백업 소스가 위치한 로컬 존이 아닌 **다른 존**에 백업. 백업 소스와 백업 데이터를 지리적으로 분리하여 안정적으로 보관.

```
[소산 백업 구조]
서울 존 (소스 + 로컬 백업)
       └──────────────→ 춘천 존 (소산 백업)
       ↑ 지리적 분리로 재해 시에도 안전

[교차 존 백업 구조]
서울 존 (소스)
       └──────────────→ 춘천 존 (백업 저장)
       ↑ 처음부터 다른 존에 저장
```

## 핵심 개념 3: 백업 영역 — 시스템 vs 데이터

| 항목 | 내용 |
|------|------|
| **시스템 백업** | OS나 애플리케이션이 설치된 영역 |
| **데이터 백업** | 애플리케이션이 사용할 데이터가 저장된 영역 |

> Best Practice: OS 영역과 데이터 영역을 **분리**하여 백업.  
> OS가 날아가도 데이터는 살릴 수 있다!

## 핵심 개념 4: 백업 방식 3종 비교 (시험 최빈출!)

### 전체 백업 (Full Backup)

```
월 [████████████████] ← 100% 전체
화 [████████████████] ← 100% 전체
수 [████████████████] ← 100% 전체
목 [████████████████] ← 100% 전체

용량: 매일 동일하게 100% 사용
```

- 매번 **모든 데이터를 통째로** 백업
- 복원: 해당 날짜 백업본 **1개**만 있으면 됨
- **장점**: 복원 가장 빠르고 간단
- **단점**: 시간 오래 걸림, 용량 많이 차지

---

### 차등 백업 (Differential Backup)

```
월 [████████████████] ← 전체 백업 (기준점 설정)
화 [▓▓▓▒▒▒          ] ← 월 이후 변경된 것 전부
수 [▓▓▓▒▒▒▒▒▒       ] ← 월 이후 변경된 것 전부 (더 많아짐)
목 [▓▓▓▒▒▒▒▒▒▓▓▓▓▓  ] ← 월 이후 변경된 것 전부 (더더 많아짐)
                            ↑ 갈수록 파일 커짐
```

- 기준: **마지막 전체 백업** 이후 변경된 부분 **모두** 백업
- 복원: **전체 백업 1개 + 해당 날짜 차등 백업 1개** = **딱 2개!**

```
[목요일에 장애 발생 시 복원]
① 월요일 전체 백업 복원
② 목요일 차등 백업 복원
   끝! (2개만으로 완성)
```

- **장점**: 복원 빠름 (항상 딱 2개)
- **단점**: 갈수록 차등 백업 파일 커짐

---

### 증분 백업 (Incremental Backup)

```
월 [████████████████] ← 전체 백업 (기준점 설정)
화 [▓▒▒              ] ← 월→화 사이 변경분만
수 [▓               ] ← 화→수 사이 변경분만
목 [▓▓              ] ← 수→목 사이 변경분만
   ↑ 항상 최소한의 용량
```

- 기준: **직전 백업(전체든 증분이든)** 이후 변경된 부분**만** 백업
- 복원: **전체 백업 + 중간 모든 증분 백업 순서대로** 쌓아야 함

```
[목요일에 장애 발생 시 복원]
① 월요일 전체 백업 복원
② 화요일 증분 백업 적용
③ 수요일 증분 백업 적용
④ 목요일 증분 백업 적용
   끝! (총 4개, 순서 틀리면 안 됨!)
```

- **장점**: 백업 시간 가장 짧음, 용량 가장 적게 사용
- **단점**: 복원이 복잡하고 번거로움

## 복원 과정 수치로 이해하기

백업 방식에 따라 복원 과정이 어떻게 달라지는지, 실제 용량으로 살펴보자.

### 차등 백업 복원 예시 (4일차 복구)

```
1일차: 전체 백업 100GB
2일차: 차등 백업 10GB (1일차 이후 변경분 누적)
3일차: 차등 백업 20GB (1일차 이후 변경분 누적)
4일차: 차등 백업 30GB (1일차 이후 변경분 누적)

→ 4일차 복원 시:
   마지막 전체 백업 (100GB) + 4일차 차등 백업 (30GB)
   = 총 2개 파일, 130GB 처리
```

### 증분 백업 복원 예시 (4일차 복구)

```
1일차: 전체 백업 100GB
2일차: 증분 백업 10GB (1→2일차 사이 변경분만)
3일차: 증분 백업 10GB (2→3일차 사이 변경분만)
4일차: 증분 백업 10GB (3→4일차 사이 변경분만)

→ 4일차 복원 시:
   전체 백업 (100GB)
   + 2일차 증분 (10GB)
   + 3일차 증분 (10GB)
   + 4일차 증분 (10GB)
   = 총 4개 파일, 순서대로 모두 적용 필수
```

차등 백업은 누적이라 파일 하나하나가 커지지만 복원은 2개로 끝난다.  
증분 백업은 매번 작지만 복원 시엔 전체 체인을 순서대로 다 쌓아야 한다.

## 비교 요약표 (시험에서 반드시 외울 것!)

| 항목 | 전체 백업 | 차등 백업 | **증분 백업** |
|------|---------|---------|------------|
| **기준점** | 없음 | **마지막 전체 백업** | **직전 어떤 백업이든** |
| **백업 대상** | 전체 | 전체 백업 이후 **변경된 것 전부** | 직전 백업 이후 **변경된 것만** |
| **백업 파일 크기** | 최대 | 중간 (갈수록 커짐) | **최소** |
| **백업 소요 시간** | 최장 | 중간 | **최단** |
| **복원 시 필요 파일** | **1개** | **2개** (전체+차등) | **여러 개** (전체+모든증분) |
| **복원 난이도** | 쉬움 | **쉬움** | 복잡 |

## 시험 문제 패턴 완전 정복

### 패턴 1: 복원 파일 개수
> Q: "차등 백업으로 복원할 때 필요한 백업본은 최소 몇 개인가?"  
> **A: 2개** (마지막 전체 백업 1개 + 해당 날짜 차등 백업 1개)

### 패턴 2: 백업 시간/용량
> Q: "백업 소요 시간이 가장 짧은 방식은?"  
> **A: 증분 백업**

> Q: "저장 용량을 가장 적게 쓰는 방식은?"  
> **A: 증분 백업**

### 패턴 3: 복원 시 번거로운 것
> Q: "복원이 가장 복잡한 방식은?"  
> **A: 증분 백업** (모든 증분 파일을 순서대로 적용해야 함)

### 패턴 4: 2개면 복원 가능한 방식
> Q: "마지막 전체 백업과 해당 날짜 백업 2개만 있으면 복원이 가능한 방식은?"  
> **A: 차등 백업**

## 시각화: 5일간의 백업 비교

```
설정: 매주 월요일 전체 백업, 화~금 각 방식으로 백업

        월     화     수     목     금
-----------------------------------------
전체   [100]  [100]  [100]  [100]  [100]
차등   [100]  [10]   [25]   [45]   [70]   ← 누적 증가
증분   [100]  [10]   [5]    [8]    [3]    ← 항상 최소

금요일 복원 시:
전체: 금요일 파일 1개만
차등: 월요일 + 금요일 2개
증분: 월 + 화 + 수 + 목 + 금 총 5개 (순서대로!)
```

## 시험 대비 체크리스트

- [ ] Backup 보관 기간: 최소 **7일** ~ 최대 **364일 (52주)**
- [ ] 보관 기간은 백업 수행 주기의 **최소 2배 이상** 설정
- [ ] 단일 서버 하루 최대 백업 용량: **1TB**
- [ ] 백업 용량 범위: 최소 **100GB** ~ 수십 **TB**
- [ ] 백업 수행 주기: **일간 / 주간 / 월간 / 일회성**
- [ ] **파일시스템 백업**: 서버(VM) 파일/디렉토리 단위 백업
- [ ] **DB 온라인 백업**: MSSQL, MySQL, PostgreSQL 지원
- [ ] **소산 백업**: 로컬 존 + 지리적으로 떨어진 다른 존에도 백업
- [ ] **교차 존 백업**: 소스와 다른 존에 백업 저장
- [ ] **시스템 백업**: OS/앱 영역 | **데이터 백업**: 데이터 저장 영역
- [ ] **전체 백업**: 매번 모든 데이터, 복원 = **1개**
- [ ] **차등 백업**: 마지막 전체 이후 변경분 전부, 복원 = **2개**
- [ ] **증분 백업**: 직전 백업 이후 변경분만, 복원 = **전체+모든증분**
- [ ] 백업 용량 작은 순: **증분 < 차등 < 전체**
- [ ] 백업 시간 짧은 순: **증분 < 차등 < 전체**
- [ ] 복원 쉬운 순: **전체 > 차등 > 증분**
- [ ] "2개만으로 복원" = **차등 백업**
- [ ] "용량/시간 최소" = **증분 백업**

## Part 3 마무리

그날 이후, 민준은 단순히 백업을 켜두는 것만으로는 부족하다는 걸 깨달았다.

> "어떤 방식으로 백업하느냐, 어디에 보관하느냐, 얼마나 보관하느냐 — 이게 전부 전략이야."

클라우드빵집의 스토리지 구성이 완성됐다.

```
[클라우드빵집 스토리지 최종 설계]

서버 OS / DB
  └→ Block Storage (KVM G3 CB/FB 타입)

직원 공유 레시피 파일
  └→ NAS (500GB, NFS 마운트)

상품 이미지 (수십만 장)
  └→ Object Storage (AWS CLI로 관리, CDN 연동)

5년 이상 된 주문 데이터
  └→ Archive Storage (OpenStack swift, Lifecycle 자동 이동)

백업 전략
  └→ 매주 월요일: 전체 백업 (파일시스템 백업)
     매일 화~일: 증분 백업
     DB: MSSQL 온라인 백업 별도 설정
     보관 기간: 52주(364일) ← 주간 백업의 52배, 조건 충족
     소산 백업: 서울 존 → 춘천 존 교차 보관

```

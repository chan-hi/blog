## 🔑 핵심 개념 1: Backup 서비스 기본

| 항목 | 내용 |
|------|------|
| **보관 기간** | 최소 **7일** ~ 최대 **364일 (52주)** |
| **시스템 백업** | OS/앱 영역 (Linux: `/`, Windows: `C:\`) |
| **데이터 백업** | 서비스 데이터 영역 (Linux: `/data`, Windows: `D:\`) |

> 💡 Best Practice: OS 영역과 데이터 영역을 **분리**하여 백업.  
> → OS가 날아가도 데이터는 살릴 수 있다!

---

## 🔑 핵심 개념 2: 백업 방식 3종 비교 (⭐⭐ 최빈출!)

### 전체 백업 (Full Backup)

```
월 [████████████████] ← 100% 전체
화 [████████████████] ← 100% 전체
수 [████████████████] ← 100% 전체
목 [████████████████] ← 100% 전체

용량: 매일 동일하게 100% 사용
```

- 매번 **모든 데이터를 통째로** 백업
- 복원: 해당 날짜 백업본 **1개**만 있으면 됨
- **장점**: 복원 가장 빠르고 간단
- **단점**: 시간 오래 걸림, 용량 많이 차지

---

### 차등 백업 (Differential Backup)

```
월 [████████████████] ← 전체 백업 (기준점 설정)
화 [▓▓▓▒▒▒          ] ← 월 이후 변경된 것 전부
수 [▓▓▓▒▒▒▒▒▒       ] ← 월 이후 변경된 것 전부 (더 많아짐)
목 [▓▓▓▒▒▒▒▒▒▓▓▓▓▓  ] ← 월 이후 변경된 것 전부 (더더 많아짐)
                            ↑ 갈수록 파일 커짐
```

- 기준: **마지막 전체 백업** 이후 변경된 부분 **모두** 백업
- 복원: **전체 백업 1개 + 해당 날짜 차등 백업 1개** = **딱 2개!**

```
[목요일에 장애 발생 시 복원]
① 월요일 전체 백업 복원
② 목요일 차등 백업 복원
   끝! (2개만으로 완성)
```

- **장점**: 복원 빠름 (항상 딱 2개)
- **단점**: 갈수록 차등 백업 파일 커짐

---

### 증분 백업 (Incremental Backup)

```
월 [████████████████] ← 전체 백업 (기준점 설정)
화 [▓▒▒              ] ← 월→화 사이 변경분만
수 [▓               ] ← 화→수 사이 변경분만
목 [▓▓              ] ← 수→목 사이 변경분만
   ↑ 항상 최소한의 용량
```

- 기준: **직전 백업(전체든 증분이든)** 이후 변경된 부분**만** 백업
- 복원: **전체 백업 + 중간 모든 증분 백업 순서대로** 쌓아야 함

```
[목요일에 장애 발생 시 복원]
① 월요일 전체 백업 복원
② 화요일 증분 백업 적용
③ 수요일 증분 백업 적용
④ 목요일 증분 백업 적용
   끝! (총 4개, 순서 틀리면 안 됨!)
```

- **장점**: 백업 시간 가장 짧음, 용량 가장 적게 사용
- **단점**: 복원이 복잡하고 번거로움

---

## 비교 요약표 (⭐ 이 표 외울 것!)

| 항목 | 전체 백업 | 차등 백업 | **증분 백업** |
|------|---------|---------|------------|
| **기준점** | 없음 | **마지막 전체 백업** | **직전 어떤 백업이든** |
| **백업 대상** | 전체 | 전체 백업 이후 **변경된 것 전부** | 직전 백업 이후 **변경된 것만** |
| **백업 파일 크기** | 최대 | 중간 (갈수록 커짐) | **최소** |
| **백업 소요 시간** | 최장 | 중간 | **최단** |
| **복원 시 필요 파일** | **1개** | **2개** (전체+차등) | **여러 개** (전체+모든증분) |
| **복원 난이도** | 쉬움 | **쉬움** | 복잡 |

## 시험 문제 패턴 완전 정복

### 패턴 1: 복원 파일 개수
> Q: "차등 백업으로 복원할 때 필요한 백업본은 최소 몇 개인가?"  
> **A: 2개** (마지막 전체 백업 1개 + 해당 날짜 차등 백업 1개)

### 패턴 2: 백업 시간/용량
> Q: "백업 소요 시간이 가장 짧은 방식은?"  
> **A: 증분 백업**

> Q: "저장 용량을 가장 적게 쓰는 방식은?"  
> **A: 증분 백업**

### 패턴 3: 복원 시 번거로운 것
> Q: "복원이 가장 복잡한 방식은?"  
> **A: 증분 백업** (모든 증분 파일을 순서대로 적용해야 함)

### 패턴 4: 2개면 복원 가능한 방식
> Q: "마지막 전체 백업과 해당 날짜 백업 2개만 있으면 복원이 가능한 방식은?"  
> **A: 차등 백업**

## 시각화: 5일간의 백업 비교

```
설정: 매주 월요일 전체 백업, 화~금 각 방식으로 백업

        월     화     수     목     금
-----------------------------------------
전체   [100]  [100]  [100]  [100]  [100]
차등   [100]  [10]   [25]   [45]   [70]   ← 누적 증가
증분   [100]  [10]   [5]    [8]    [3]    ← 항상 최소

금요일 복원 시:
전체: 금요일 파일 1개만
차등: 월요일 + 금요일 2개
증분: 월 + 화 + 수 + 목 + 금 총 5개 (순서대로!)
```

## 📝 시험 대비 체크리스트

- [ ] Backup 보관 기간: 최소 **7일** ~ 최대 **364일**
- [ ] **전체 백업**: 매번 모든 데이터, 복원 = **1개**
- [ ] **차등 백업**: 마지막 전체 이후 변경분 전부, 복원 = **2개**
- [ ] **증분 백업**: 직전 백업 이후 변경분만, 복원 = **전체+모든증분**
- [ ] 백업 용량 작은 순: **증분 < 차등 < 전체**
- [ ] 백업 시간 짧은 순: **증분 < 차등 < 전체**
- [ ] 복원 쉬운 순: **전체 > 차등 > 증분**
- [ ] "2개만으로 복원" = **차등 백업**
- [ ] "용량/시간 최소" = **증분 백업**

```
[클라우드빵집 스토리지 최종 설계]

서버 OS / DB

  └→ 매주 월요일: 전체 백업

```

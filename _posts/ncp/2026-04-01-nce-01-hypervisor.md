## 🔑 핵심 개념 1: 서버 세대와 하이퍼바이저

**하이퍼바이저(Hypervisor)** = 물리 서버의 CPU, Memory, Storage를 쪼개서 여러 가상 서버(VM)에 나눠주는 가상화 관리 소프트웨어.

| 구분 | **1세대(G1) / 2세대(G2)** | **3세대(G3)** |
|------|--------------------------|--------------|
| **하이퍼바이저** | **XEN** | **KVM** |
| CPU 성능 | 기본 | **비약적으로 향상** (KVM 전환 이유) |
| AMD 기반 서버 | 지원 안 함 | 지원 (High CPU, High Memory, Standard) |
| 지원 리전 | 전체 리전 | 한국, 싱가포르, 일본 **(VPC 환경만)** |

> **시험 포인트 #1**  
> **하이퍼바이저가 다른 리소스 간에는 호환 불가!**  
> XEN으로 만든 서버 이미지를 KVM 서버에서 사용할 수 없다.

```
[호환 불가 예시]
G2 서버의 내 서버 이미지 → G3 서버에서 사용 X
G3 서버의 내 서버 이미지 → G2 서버에서 사용 X
같은 세대끼리만 가능
```

## 🔑 핵심 개념 2: CPU vs Memory 할당 규칙

민준이 물었다.

> "물리 서버 CPU가 8코어면, 가상 서버에 총 8코어만 줄 수 있어요?  
> 100개 서버 만들면 CPU는 어떻게 되는 거야?"

지수의 답:

> "CPU는 **Over Commit**이 허용돼.  
> 실제보다 더 많이 약속해줄 수 있어. 어차피 다 동시에 100% 쓰는 경우는 드물거든.  
> 근데 **메모리는 안 돼**. 메모리는 실제로 있는 만큼만 줄 수 있어."

| 자원 | 할당 단위 | **Over Commit** | 스펙 변경 |
|------|-----------|----------------|-----------|
| **CPU** | vCPU | **허용** | 가능 |
| **Memory** | GB | **불허** | 가능 |

CPU는 물리 코어(Core)가 vCPU에 매핑되는 구조다. vCPU는 Over Commit이 허용되므로 물리 코어 수보다 더 많은 vCPU를 VM에 할당할 수 있다.

> **비유**: CPU는 "자리 예약"처럼 오버부킹 가능. Memory는 실제 물리 공간이라 있는 만큼만.

## 🔑 핵심 개념 3: 가상화 기반 기술 (VT-x, VT-d, SR-IOV)

> "하이퍼바이저가 작동하려면 CPU나 하드웨어가 가상화를 지원해야 한다고 하던데요?"

지수가 고개를 끄덕였다.

> "맞아. Intel에는 가상화 관련 기술이 여러 가지 있어. 시험에도 나오니까 개념을 잡아 두자."

### HyperThreading (하이퍼스레딩)

**SMT(Simultaneous Multi-Threading)** 을 지원하는 CPU 아키텍처로, 하나의 물리 코어에서 여러 개의 스레드를 동시에 처리하는 구조다. 2000년대 들어 인텔의 **HyperThreading** 이라는 상표명으로 인텔 CPU에 적용되었다.

```
[Without Intel HT Technology]   [With Intel HT Technology]
  Thread 0 Architectural State    Thread 0 Architectural State
  (1코어 = 1스레드)                Thread 1 Architectural State
                                  (1코어 = 2스레드)
```

### Intel VT (Virtualization Technology)

가상화를 위해 복수 OS의 병렬 동작을 보다 효율적으로 지원하는 기술. AMD의 AMD-V와 대응된다.

- **VT-x**: 인텔의 가상화 기술로, 가상 머신이 **물리 메모리에 직접 접근**하는 것을 가능하게 하는 기술
- **VT-d**: 인텔의 가상화 기술로, 가상 머신이 **물리 디바이스에 직접 접근**하는 것을 가능하게 하는 기술

| 기술 | 핵심 역할 |
|------|----------|
| **VT-x** | VM → 물리 **메모리** 직접 접근 |
| **VT-d** | VM → 물리 **디바이스** 직접 접근 |

### SR-IOV (Single Root I/O Virtualization)

다수의 가상 머신이 **하나의 I/O를 공유**할 수 있도록 하는 가상화 지원 디바이스 기술이다.

SR-IOV를 사용하면:
- 가상 머신 입장에서 디바이스가 **직접 연결된 것과 같은 효과**를 가짐
- 가상화로 인한 **성능 저하 및 제약을 방지** 가능
- 실제 디바이스 드라이버를 사용하므로, 디바이스 드라이버가 SR-IOV를 **반드시 지원**해야 함
- 단, 하나의 SR-IOV 장치에 할당할 수 있는 가상 머신의 수에는 **제한**이 있음

## 🔑 핵심 개념 4: 하이퍼바이저별 OS 스토리지 차이 (⭐ 매우 중요!)

가장 많이 헷갈리는 부분. OS가 설치되는 **부팅 디스크(OS 영역)**의 규칙이 세대마다 완전히 다르다.

| 구분 | **XEN (G1/G2)** | **KVM (G3)** |
|------|----------------|--------------|
| **Linux OS 영역** | **50GB 고정** | **10GB ~ 2TB** (선택 가능) |
| **Windows OS 영역** | **100GB 고정** | **30GB ~ 2TB** (선택 가능) |
| **OS 영역 확장** | **불가** | **서버 정지 후 확장 가능** |
| **OS 영역 축소** | **불가** | **불가** |

> **시험 포인트 #2**  
> XEN: OS 영역 **확장도 축소도 불가**. 처음부터 고정!  
> KVM: OS 영역 **확장 가능** (단, 서버를 반드시 **정지**한 후에만)  
> KVM Windows 최솟값은 **30GB** (100GB가 아님!)

```
[민준의 상황]
G1 서버로 시작 → OS 디스크 50GB 꽉 참 → 늘리고 싶음
→ XEN이라 불가능! → G3(KVM)으로 마이그레이션 필요

G3 서버로 시작 → OS 디스크 10GB 꽉 참 → 늘리고 싶음
→ 서버 정지 → 확장 가능!
```

### KVM 추가 스토리지 규칙

| 항목 | 내용 |
|------|------|
| **추가 스토리지 용량** | 10GB ~ 최대 **16TB** |
| **추가 스토리지 개수** | 최대 **20개** |
| **서버당 총 스토리지** | 기본 스토리지 포함 최대 **21개** |

> XEN(G1/G2)의 추가 디스크는 디스크당 최대 2,000GB이며, OS가 지원하는 개수만큼 생성 가능.

## 🔑 핵심 개념 6: GPU 서버

GPU 서버의 특징:

- GPU는 **GPU 단위**로 할당 (GPU Core 단위가 아님)
- **PassThrough 방식**으로 제공 (가상화 오버헤드 없이 직접 연결)
- 지원 GPU 모델: **A100, T4, V100** 등
- 그래픽카드별로 **서버 한 대당 매핑 가능한 개수**가 다름

---

## 🔑 핵심 개념 7: 네트워크 스펙

| 항목 | 값 |
|------|-----|
| **물리(Physical)** | 10Gbps |
| **논리(Logical)** | 1Gbps |
| **실제 제공 대역폭** | 500Mbps ~ 1Gbps (RX + TX 합산) |

- 서버 간 **사설(Private) 통신** 가능
- 외부 서비스 제공 방법: **공인 IP 직접 할당** 또는 **Load Balancer** 연결
- 공인 IP는 서버 간 이동 가능 → 신규 서버 전환 시 **짧은 중단 후 서비스 연속 제공** 가능

## 🔑 핵심 개념 8: 서버 타입

| 서버 타입 | CPU:Memory 비율 | 용도 | 예시 스펙 |
|----------|----------------|------|----------|
| **Micro** | 1:1 | 신규 가입 후 최초 결제 수단 등록 월부터 **1년간 무료** (이후 자동 과금) | vCPU 1개, 메모리 1GB |
| **Standard** | 1:4 | 일반 웹 또는 데이터베이스 용도 | s2: vCPU 2개, 메모리 8GB |
| **High Memory** | 1:8 | 고성능 데이터베이스 용도 | m4: vCPU 4개, 메모리 32GB |
| **High CPU** | 1:2 | 컴퓨팅 집약적인 워크로드 (과학적 모델링 등) | c16: vCPU 16개, 메모리 32GB |
| **CPU-Intensive** | 1:2 | 많은 연산이 필요한 머신/딥러닝, 고성능 웹서버 | c16: vCPU 16개, 메모리 32GB |

> Micro 서버는 KVM G3에서 AMD 기반으로도 제공. G3 AMD 기반 서버 중 Standard는 vCPU 2개, 메모리 16GB 사양도 있음.

## 🔑 핵심 개념 10: 요금 체계

> "서버 쓰다가 잠깐 멈추면 요금은 어떻게 돼요?"

지수가 설명했다.

> "**월 또는 시간** 단위로 요금이 부과돼.  
> 월 단위로 신청했는데 한 달 안에 종료하면? 일할 계산해서 청구돼.  
> 그리고 서버를 **정지**시키면 운영 요금은 안 내지만, **기본 디스크 요금은 계속 나와.**"

| 상황 | 청구 내용 |
|------|----------|
| **서버 운영 중** | 서버 요금 전체 |
| **서버 정지 시** | **기본 디스크(OS 영역) 요금만** 청구 |
| **월 단위, 1개월 미만 사용** | **일할 계산** 청구 |

> Windows 서버에 한해 기본 디스크로 100GB 서비스 활용 가능.

## 🔑 핵심 개념 11: Bare Metal Server

> "가상화 없이 물리 서버를 그대로 쓸 수는 없나요?"

지수가 말했다.

> "있어. **Bare Metal Server**야. 하이퍼바이저 없이 물리 서버에 바로 OS를 설치해서 쓰는 거야.  
> 성능에 민감한 서비스에 적합하고, RAID 구성을 선택할 수 있어."

Bare Metal Server의 특징:

- 단독으로 사용하는 **고성능 물리 서버**를 클라우드 형태로 제공
- **하이퍼바이저 없이** 물리 서버에 바로 OS 설치
- **RAID 구성** 선택 가능:
  - **RAID 5**: 일반적으로 사용되는 RAID 레벨, 웹 서비스 같은 일반 애플리케이션에 적합
  - **RAID 1+0**: DBMS처럼 높은 성능과 안정성이 필요한 애플리케이션에 적합
- 지원 OS: CentOS, Red Hat Enterprise Linux, Ubuntu Server, Windows

**Bare Metal Server 제공 사양**

| 구성 | CPU | 코어 수 | 메모리 | 디스크 |
|------|-----|--------|-------|-------|
| Single | Intel Xeon Silver 4215 (2.6GHz) | 8개 | 128GB | 4 x 1.9TB SSD |
| Dual | Intel Xeon Silver 4214 (2.2GHz) | 24개 | 256GB | 4 x 1.9TB SSD |
| Dual | Intel Xeon Gold 5220 (2.2GHz) | 36개 | 384GB | 8 x 1.9TB SSD |
| Dual | Intel Xeon Gold 6248R (3.0GHz) | 48개 | 512GB | 8 x 1.9TB SSD |
| GPU (Tesla A100 x 8EA) | Dual AMD Epyc Milan 7313 (3.0GHz) | 32개 | 2TB | 2 x 240GB SSD + 4 x 7.68TB NVMe SSD |

## 📝 시험 대비 체크리스트

- [ ] G1/G2 = **XEN** 하이퍼바이저
- [ ] G3 = **KVM** 하이퍼바이저
- [ ] 서로 다른 하이퍼바이저 간 **호환 불가**
- [ ] CPU: **Over Commit 허용** (물리 코어가 vCPU에 매핑, 오버부킹 가능)
- [ ] Memory: **Over Commit 불허**
- [ ] XEN OS 영역: Linux **50GB** / Windows **100GB** — **확장/축소 모두 불가**
- [ ] KVM OS 영역: Linux 최소 **10GB** / Windows 최소 **30GB** — **서버 정지 후 확장 가능**
- [ ] KVM 추가 스토리지: 최대 **16TB**, 최대 **20개** (기본 포함 서버당 **21개**)
- [ ] G3는 **VPC 환경**에서만 지원 (Classic 환경 불가)
- [ ] **VT-x**: VM → 물리 메모리 직접 접근 / **VT-d**: VM → 물리 디바이스 직접 접근
- [ ] **SR-IOV**: 다수의 VM이 하나의 I/O를 공유, 가상화 성능 저하 방지
- [ ] GPU는 **GPU 단위** 할당 (GPU Core 단위 아님), **PassThrough** 방식
- [ ] 서버 정지 시 **기본 디스크 요금만** 청구
- [ ] 월 단위 요금, 1개월 미만 사용 시 **일할 계산**
- [ ] Bare Metal: RAID 5 (일반 앱) / RAID 1+0 (DBMS, 고성능)

## 이야기: 도메인을 사다

클라우드빵집이 `cloudbakery.co.kr` 도메인을 샀다.
그리고 전국 고객들이 빠르게 이미지를 볼 수 있도록 CDN도 붙이기로 했다.

> "DNS가 정확히 뭘 하는 거야?"

재현이 설명했다.

> "전화번호부야. 사람들은 'cloudbakery.co.kr'을 입력하는데,

## 🔑 핵심 개념 1: DNS 기본 개념

**DNS (Domain Name System)** = 도메인 이름을 IP 주소로 변환하는 시스템.

```
[DNS 조회 과정]
사용자: cloudbakery.co.kr 입력
        ↓
DNS 서버: cloudbakery.co.kr = 110.10.20.30 (IP)
        ↓
브라우저: 110.10.20.30 서버로 접속
```

---

## 🔑 핵심 개념 2: NCP Global DNS 특징

| 항목 | 내용 |
|------|------|
| **TTL 기본값** | **300초 (5분)** |
| **TTL 최대값** | **7일** |
| **레코드 한 번에 추가** | 최대 **10개** |
| **트래픽 분기** | 동일 도메인 → **Round Robin** 방식 |
| **Alias 기능** | 다른 레코드의 별칭 지정 가능 |

> 💡 **TTL 팁**: 도메인 변경 작업 전에는 TTL을 짧게 줄여야 변경이 빠르게 전파된다.

## 🔑 핵심 개념 3: DNS 레코드 타입 9가지 (⭐⭐ 최빈출!)

| 레코드 | 역할 | 특이사항 |
|--------|------|---------|
| **A** | 도메인 → **IPv4** 주소 매핑 | 가장 기본 |
| **AAAA** | 도메인 → **IPv6** 주소 매핑 | A의 IPv6 버전 |
| **CNAME** | 도메인에 **별명(Alias)** 지정 | **다른 레코드와 공존 불가** |
| **MX** | 메일 수신 **메일 서버** 지정 | **Preference 값 작을수록 우선순위 높음** |
| **SPF** | 발송자와 메일 서버 정보 일치 검증 → **스팸/사칭 방지** | TXT 레코드 형태로도 사용 |
| **TXT** | 도메인에 텍스트 정보 추가 | 유효성 판단, SPF와 유사 용도 |
| **SRV** | 서비스를 호스팅하는 서버의 **위치(호스트명, 포트번호)** 저장 | - |
| **CAA** | 인증서 발급 가능한 **인증기관(CA) 지정** → 가짜 인증서 방지 | - |
| **DS** | **DNSSEC**에서 서명 있는 영역 식별 데이터 저장 | - |

> ⚠️ **CNAME 주의**: CNAME은 같은 도메인에 다른 레코드(A, MX 등)와 **공존 불가!**

### 자주 나오는 문제 패턴

```
Q: "메일 서버를 지정하는 레코드 타입은?" → MX
Q: "MX 레코드에서 우선순위가 높은 것은?" → Preference 값이 작은 것
Q: "도메인에 별명을 지정하는 레코드는?" → CNAME
Q: "가짜 인증서 발급을 막는 레코드는?" → CAA
Q: "스팸 메일 방지를 위한 레코드는?" → SPF (또는 TXT)
```

## 🔑 핵심 개념 4: CDN (Content Delivery Network)

**CDN** = 콘텐츠를 사용자 가까운 캐시 서버에 저장하여 빠르게 전달하는 서비스.

```
[CDN 없을 때]
서울 사용자 → 원본 서버 (부산) → 시간 걸림
뉴욕 사용자 → 원본 서버 (부산) → 엄청 오래 걸림

[CDN 있을 때]
서울 사용자 → 서울 CDN 캐시 서버 → 빠름!
뉴욕 사용자 → 뉴욕 CDN 캐시 서버 → 빠름!
```

### Cache Hit vs Cache Miss (⭐)

| 상태 | 설명 | 동작 |
|------|------|------|
| **Cache Hit** | 캐시 서버에 콘텐츠가 **있음** | 캐시에서 바로 응답 (빠름!) |
| **Cache Miss** | 캐시 서버에 콘텐츠가 **없음** | 원본 서버에 요청 후 캐시에 저장 |

```
[처음 요청 — Cache Miss]
사용자 → CDN 캐시 서버 (없음) → 원본 서버 → CDN 캐시 저장 → 응답

[이후 요청 — Cache Hit]
사용자 → CDN 캐시 서버 (있음!) → 즉시 응답 ⚡
```

## 🔑 핵심 개념 5: NCP Global Edge 주요 기능

| 기능 | 내용 |
|------|------|
| **엣지 유형** | 한국/일본 엣지, 글로벌 성능우선(네이버+파트너), 글로벌 가용성우선(파트너) |
| **룰빌더** | 캐시 제어, URL 리다이렉션 설정 가능 |
| **최신 프로토콜** | **HTTP/2**, **TLSv1.3** 지원 |
| **보안** | Secure-Token, **JWT 기반 보안 URL** |
| **접근 제어** | IP 기반, **GEO(지역) 기반**, Referer 기반 접근 제어 |
| **압축** | 최신 압축 알고리즘 **brotli** 지원 |
| **대용량 전송** | **byte range** 지원 (대용량 파일 분할 다운로드) |

## 📝 시험 대비 체크리스트

- [ ] DNS TTL 기본값: **300초** / 최대: **7일**
- [ ] 레코드 한 번에 추가 최대: **10개**
- [ ] A: IPv4 매핑 / AAAA: IPv6 매핑
- [ ] **CNAME**: 별명, 다른 레코드와 **공존 불가**
- [ ] **MX**: 메일 서버, Preference **작을수록** 우선순위 높음
- [ ] SPF: 스팸 방지 / CAA: 가짜 인증서 방지 / DS: DNSSEC
- [ ] Cache Hit: 캐시에 있음 / Cache Miss: 없어서 원본 요청
- [ ] Global Edge: brotli 압축, byte range, JWT 보안 URL

## 🧠 암기 핵심 문장

> "**A=IPv4, AAAA=IPv6, CNAME=별명(공존불가), MX=메일(작을수록우선)**"  
> "**Cache Hit = 있어서 빠름 / Cache Miss = 없어서 원본에서 가져옴**"

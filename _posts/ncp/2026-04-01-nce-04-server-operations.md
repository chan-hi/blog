## 🔑 핵심 개념 0: 서버 관련 자원 제한

본격적인 운영에 앞서, NCP 서버에 적용되는 리소스 기본 제한값을 알아두자.

| 리소스 | 기본값 / 제한 |
|--------|-------------|
| **서버당 최대 공인 IP 수** | 서버당 **1개** |
| **VPC당 최대 ACG 수** | **500개** |
| **서버당 최대 ACG 수** | **3개** |
| **ACG당 최대 규칙 수** | **50개** |
| **최대 Network Interface 수** | **3개** (단, Micro-g3는 **1개**) |
| **서버 정지 가능 일수** | 1회 최대 **90일**, 12개월 누적 **180일** |

> ⚠️ **주의**: 공인 IP, 로드밸런서, 추가 디스크 등 서버에 연결된 자원의 요금은 **서버 정지 시에도 정상 청구**된다. 사용하지 않는 리소스는 반드시 반납 처리할 것.

## 🔑 핵심 개념 1: 서버 상태 값

서버는 다음 상태 값을 거쳐 운영된다.

**생성 시**: 생성중 → 부팅중 → 설정중 → **운영중**

**정지 시**: 종료중 → **정지**

**시작 시**: 부팅중 → **운영중**

## 🔑 핵심 개념 2: 서버 Operation — 상태별 가능 여부 (⭐⭐ 최빈출!)

| 작업 | 필요 서버 상태 | 핵심 포인트 |
|------|--------------|------------|
| **서버 스펙 변경** | **정지** 필수 | 운영 중에는 절대 불가 |
| **내 서버 이미지 생성** | **운영 중 / 정지 모두 가능** | ← 이게 함정 포인트! |
| **스토리지 용량 변경** | **정지 또는 연결 해제** | 증설 후 OS 내에서 추가 작업 필요 |
| **인증키 변경** | **정지** 필수 | 결과는 관리자 이메일로 전송됨 |

> ⚠️ **시험 포인트 #1**  
> **내 서버 이미지는 운영 중에도 생성 가능!**  
> 보통 "서버 켜져 있으면 스냅샷 못 찍는다"고 생각하기 쉬운데, NCP는 **운영 중에도 가능**.

```
[민준의 시험 함정]
Q: "서버 이미지를 생성하려면 서버를 정지해야 한다. (O/X)"
A: X — 운영 중에도 가능!
```

## 🔑 핵심 개념 3: 서버 Operation 1 — 스펙 변경 / 이미지 생성 / 유사 서버 생성

### 서버 스펙 변경

반드시 **정지된 상태**에서 변경해야 한다. 운영 중에는 불가.

### 내 서버 이미지 생성

서버의 현재 상태를 이미지로 생성하는 기능. **정지 중 / 운영 중 상태 모두**에서 생성 가능.

### 유사 서버 생성

선택한 서버와 동일한 스펙의 서버 생성 시 사용한다. 다음 정보를 그대로 적용한다:

- 서버 이미지
- 스토리지
- 서버 타입
- 요금제
- Zone
- 인증키
- ACG

> 동일한 설정의 서버를 빠르게 여러 대 만들 때 유용하다.

## 🔑 핵심 개념 4: 서버 Operation 2 — 스토리지 생성 및 용량 변경

### 스토리지 생성 규칙

OS 영역(50GB)에 추가적으로 디스크를 생성할 때 사용한다.

| 하이퍼바이저 | OS 영역 (Linux) | OS 영역 (Windows) | 추가 가능한 스토리지 용량 | 최대 추가 수 | 스토리지 타입 |
|------------|----------------|-------------------|----------------------|-----------|-------------|
| **XEN** (1·2세대) | 50GB | 100GB | 최소 **10GB** ~ 최대 **2TB** | **15개** | SSD, HDD |
| **KVM** (3세대) | 10GB~2TB | 30GB~2TB | 최소 **10GB** ~ 최대 **16TB** | **20개** | CB1, FB1, CB2, FB2 |

> KVM은 XEN보다 스토리지 측면에서 훨씬 유리하다. 또한 KVM 기반 3세대 서버의 OS 영역은 **정지 후 사이즈 변경이 가능**하다.

### 용량 변경 규칙

| 방향 | 가능 여부 |
|------|----------|
| **증설 (늘리기)** | 가능 |
| **축소 (줄이기)** | **불가** |

> ⚠️ **중요**: 용량 증설 후 **데이터 삭제 없이 해당 공간을 사용하기 위해서는 볼륨 관리 툴을 이용**해야 한다.  
> 물리 디스크가 커졌다고 자동으로 OS에 반영되지 않는다.

증설은 **서버 정지** 또는 **스토리지 연결 해제** 후 가능하다.

### 스토리지 detach 기능

스토리지를 서버에서 해제(detach)하여 다른 서버에 연결(attach)하는 기능도 제공한다.

### 물리 배치 그룹 관리

서버들을 최대한 다른 클러스터에 분산시키기 위한 방법으로, 분산 결과를 보장하지 않는 **Best-Effort 방식**이다.

## 🔑 핵심 개념 5: 서버 Operation 3 — 반납 보호 / 모니터링 / 이름 변경

| 항목 | 내용 | 주의사항 |
|------|------|---------|
| **반납 보호 설정** | 사용자 부주의로 서버가 반납되지 않도록 설정하는 기능 | 보호 설정 시 삭제 요청 불가 |
| **상세 모니터링** | 서버들의 모니터링에 대해 보다 상세하게 모니터링하는 기능 | 설정 시 모니터링 서비스에서 정보 확인 가능 |
| **네트워크 모니터링 설정** | 서버에서 발생하는 트래픽에 대한 상세 내용을 확인할 수 있도록 설정 | 네트워크 트래픽 모니터링 서비스에서 정보 확인 가능 |
| **서버 이름 변경** | 콘솔에서 보여지는 서버 이름 변경 | **실제 서버 내부에서 호스트명이 변경되지 않음** |

> ⚠️ **시험 포인트 #2**  
> 서버 이름을 바꿔도 **OS 내부의 호스트명은 그대로!**  
> 실제 호스트명을 바꾸려면 OS 내부에서 직접 변경해야 한다.

## 🔑 핵심 개념 6: 서버 Operation 4 — 인증키 변경

- 서버 생성 시 만든 인증키를 **다른 인증키로 변경** 가능
- 초기 패스워드를 변경하지 않은 상태에서 인증키 분실 시, 초기 패스워드를 확인하기 위해 인증키 변경이 필요
- **서버 정지 후** 변경 가능
- 인증키 변경 시 ID인 **이메일로 인증 정보 전송**

## 🔑 핵심 개념 7: 내 서버 이미지 (Server Image Operation)

**내 서버 이미지** = 현재 서버의 OS + 설치된 앱 + 추가 스토리지 상태를 캡처한 것.  
나중에 이 이미지로 **똑같은 서버를 찍어낼 수** 있다.

| 항목 | 내용 |
|------|------|
| **생성 가능 상태** | 운영 중 / 정지 **모두 가능** |
| **포함 내용** | **서버의 모든 스토리지**를 이미지로 생성 (추가 스토리지까지 포함) |
| **추가 스토리지 용량이 클 경우** | 스토리지 해제 후 OS 영역만 이미지로 생성 |
| **리전 간 복제** | **OS 영역만** 이미지로 전송하고, 데이터는 SFTP 방식으로 전송 |
| **계정 공유** | 공유 권한 설정에 공유받는 계정을 추가하여 다른 NCP 계정과 이미지 공유 가능 |

> ⚠️ **시험 포인트 #3**  
> 리전 간 이미지 복제 시 → **OS 영역만** 이미지로 전송!  
> 데이터는 SFTP 등으로 따로 옮겨야 한다.

### 이미지를 이용한 서버 생성

서버의 마스터 이미지를 생성하고, 이를 이용하여 서버를 생성한다.

## 🔑 핵심 개념 8: Storage Operation — 디스크 파티션 / 포맷 / 마운트

디스크를 논리적으로 분할하는 방식: 물리적 디스크(하드디스크, ROM 등)를 파티션으로 분할한다.

**리눅스 순서: 파티셔닝 → 파일시스템 → 마운트**

### 신규 스토리지 추가 후 Linux 설정 순서 (⭐ 빈출!)

```bash
1. lsblk          # 물리 디스크 인식 확인  (예: /dev/sde 확인)
2. parted         # 파티셔닝             (예: /dev/sde → /dev/sde1, /dev/sde2)
3. mkfs           # 파일시스템 포맷       (예: mkfs.ext4 /dev/sde1)
4. mount          # 마운트              (예: mount /dev/sde1 /data)
```

> 💡 **순서 암기**: "lsblk → parted → mkfs → mount"  
> "**인식 → 파티션 → 포맷 → 마운트**"

### 스냅샷 기본 컨셉

기본적인 애플리케이션은 OS 영역에 설치하고, **데이터는 추가 스토리지에 저장**하는 것이 권장 패턴이다. 이렇게 하면 시스템 Crash 발생 시 데이터 유실을 최소화하고 빠른 복구가 가능하다.

## 🔑 핵심 개념 9: 스냅샷 (Snapshot Operation)

"내 서버 이미지"와 "스냅샷"의 차이를 반드시 구분해야 한다!

| 항목 | **내 서버 이미지** | **스냅샷** |
|------|-----------------|-----------|
| **생성 단위** | **서버** 단위 | **스토리지(디스크)** 단위 |
| **복원 형태** | 서버로 바로 복원 | **추가 스토리지로만** 변환 가능 |
| **백업 방식** | - | **증분(Incremental) 스냅샷** 지원 → 스냅샷 용량 효율화 가능 |

> ⚠️ **시험 포인트 #4**  
> 스냅샷으로는 **서버를 바로 만들 수 없다!**  
> 오직 **추가 스토리지**로만 변환 가능.

### 스냅샷 장애 복구 시나리오

```
[OS 영역 손상으로 부팅 불가]
       ↓
OS 영역의 스토리지를 스냅샷으로 생성
       ↓
해당 스냅샷을 다른 서버의 "추가 스토리지"로 변환
       ↓
해당 서버에서 원하는 데이터를 추출하여 복구
```

> 이 시나리오: 스냅샷 → 추가 스토리지 → 다른 서버에 붙이기 → 데이터 구출

## 🔑 핵심 개념 10: 네트워크 인터페이스 (Network Interface Operation)

### NIC 제한

서버에 네트워크 카드(NIC)를 여러 개 달 수 있다. 하지만 개수 제한이 있다.

| 항목 | 제한 |
|------|------|
| **NIC 최대 수** | 서버 1대당 **최대 3개** (단, Micro-g3는 **1개**) |
| **NIC당 Secondary IP** | 최대 **5개** |
| **NIC당 총 IP** | Primary 1개 + Secondary 5개 = **총 6개** |

```
[IP 계산]
NIC 1개 → IP 6개 (1 + 5)
NIC 3개 (최대) → IP 최대 18개 (6 × 3)
```

> ⚠️ 멀티 IP 이용을 위해서는 **부가적인 network script 작성 및 적용**이 필요하다.

### NIC를 여러 개 생성하는 경우

- 서버에서 사용할 애플리케이션의 특성에 따라 또 다른 Subnet이 필요한 경우
- 용도에 따라 Subnet을 분리하여 설계하였으나 한 서버로 통합해서 사용해야 하는 경우
- 여러 대의 서버를 사용 목적에 따라 직관적으로 구분하기 위해 활용 (예: 10.1.1.1x는 개발 서버, 10.1.1.2x는 운영 서버로 구분)

### VIP (고가용성) 구성

**HA(High Availability)** 구성 시 VIP 사용:

```
Secondary IP + Pacemaker + corosync
→ 하나의 서버가 죽으면 VIP가 다른 서버로 자동 이동
```

## 🔑 핵심 개념 11: ACG (Access Control Group)

ACG는 IP 기반 필터링 기능으로, 서버로의 네트워크 접근을 관리하는 서비스다. 서버 방화벽 역할을 한다.

| 항목 | 내용 |
|------|------|
| **종류** | Default ACG와 Custom ACG로 구분 |
| **프로토콜** | TCP, UDP, ICMP 중 선택 |
| **접근 소스** | IP(CIDR), ACG Group 설정 가능 |
| **최대 생성 개수** | VPC당 **500개** |
| **적용 대상** | 서버 NIC |
| **NIC당 적용 ACG 개수** | **3개** |
| **Rule 속성** | In/Outbound |
| **Rule 최대 생성 개수** | In **50개** / Out **50개** |

### ACG 규칙 설정 예시

| 설정 목적 | 프로토콜 | 접근 소스 | 허용 포트 |
|----------|---------|----------|---------|
| 특정 IP에서 SSH 접근 허용 | TCP | 192.168.77.1 | 22 |
| 특정 IP 대역에서 SSH 접근 허용 | TCP | 192.168.77.0/24 | 22 |
| 특정 ACG에 할당된 서버 간 SSH 허용 | TCP | lab1-web-ACG | 22 |
| 특정 IP에서 UDP 22-1025 포트 허용 | UDP | 192.168.77.1 | 22-1025 |

### NACL vs ACG 비교

| 항목 | **Network ACL** | **ACG** |
|------|----------------|---------|
| **적용 범위** | Subnet 접근 시 작동 | 서버 접근 시 작동 |
| **규칙 설정** | Inbound/Outbound 허용/차단 규칙 모두 설정 | Inbound/Outbound **허용 규칙만** 설정 |
| **기본 규칙** | 기본적으로 차단 규칙 적용 | 기본적으로 허용 규칙 적용 |
| **상태 추적** | **Stateless** — 트래픽 상태 저장 안 함, 규칙 별도 설정 필요 | **Stateful** — 트래픽 상태 저장, Inbound 허용 시 Outbound 자동 허용 |
| **규칙 처리 방식** | 우선순위로 처리 | 모든 규칙을 평가 |
| **적용 대상** | Subnet 내 모든 서버에 적용 | 서버 시작 시 또는 나중에 지정 |

## 🔑 핵심 개념 12: CLI & API 제어 도구 (⭐ 함정 포인트!)

NCP CLI는 **Java 어플리케이션**으로 제작되었으며, API와의 호환성을 위해 Java로 만들어졌다. 따라서 **JRE가 설치**되어 있어야 하며, REST API 방식으로 동작한다.

### CLI 구조

```bash
ncloud <command> <subcommand> [options and parameters]
```

주요 command 예시:

| command | 설명 |
|---------|------|
| `Server` | `getServerInstanceList`, `createServerInstances` 등 |
| `Loadbalancer` | 로드밸런서 관련 |
| `Autoscaling` | `startServerInstances`, `stopServerInstances` 등 |
| `NAS` | NAS 관련 |
| `CloudDB` | `rebootServerInstances`, `terminateServerInstances` 등 |
| `CloudHadoop` | Hadoop 관련 |

주요 옵션:
- `--output` : 결과값 출력 포맷 (json \| xml)
- `--ReadTimeout`
- `--ConnectTimeout`

### 서비스별 CLI/API 도구 구분

| 서비스 | 사용 CLI/API | 이유 |
|--------|------------|------|
| **일반 서버** | **ncloud CLI** (`ncloud <command>`) | NCP 전용 |
| **Object Storage** | **AWS CLI / AWS API** | **S3 호환**이기 때문 |
| **Archive Storage** | **OpenStack keystone/swift CLI** | OpenStack 기반 |

```bash
# 서버 목록 조회
ncloud vserver getServerInstanceList

# Object Storage 파일 업로드 (AWS CLI)
aws s3 cp ./photo.jpg s3://my-bucket/ --endpoint-url=https://kr.object.ncloudstorage.com

# Archive Storage 파일 업로드 (swift CLI)
swift upload my-container ./archive.tar.gz
```

## 📝 시험 대비 체크리스트

- [ ] 서버 정지 가능 일수: 1회 최대 **90일**, 12개월 누적 **180일**
- [ ] Micro-g3의 Network Interface 최대 수: **1개** (일반 서버는 3개)
- [ ] 스펙 변경 → **정지** 필수
- [ ] 이미지 생성 → **운영 중에도 가능**
- [ ] 유사 서버 생성 → 이미지, 스토리지, 서버타입, 요금제, Zone, 인증키, ACG 모두 동일 적용
- [ ] 내 서버 이미지 = **모든 스토리지** 포함 (추가 스토리지까지)
- [ ] 리전 간 이미지 복제 → **OS 영역만**, 데이터는 SFTP로 별도 이전
- [ ] 인증키 변경 → 서버 **정지 후**, 결과는 **이메일로 전송**
- [ ] 스냅샷 = 스토리지 단위, **추가 스토리지로만** 복원
- [ ] 스냅샷: **증분(Incremental)** 백업 지원 → 용량 효율화 가능
- [ ] XEN 추가 스토리지: **10GB~2TB**, 최대 **15개**
- [ ] KVM 추가 스토리지: **10GB~16TB**, 최대 **20개**
- [ ] 용량 증설만 가능, **축소 불가**
- [ ] 용량 증설 후 공간 사용: **볼륨 관리 툴** 이용 필요
- [ ] Linux 디스크 설정: **lsblk → parted → mkfs → mount**
- [ ] 스토리지 **detach → 다른 서버에 attach** 가능
- [ ] NIC: 서버당 최대 **3개** (Micro-g3는 1개), NIC당 Secondary IP 최대 **5개**
- [ ] NIC 1개 = IP 최대 **6개** (Primary 1 + Secondary 5)
- [ ] 멀티 IP 사용 시 **network script 작성 및 적용** 필요
- [ ] 반납 보호 = 부주의로 서버가 반납되지 않도록 설정하는 기능
- [ ] 서버 이름 변경 = 콘솔 이름만 변경, **호스트명은 그대로**
- [ ] 물리 배치 그룹: **Best-Effort** (100% 보장 안 함)
- [ ] ACG: **허용 규칙만**, Stateful
- [ ] NACL: **허용/차단 규칙 모두**, Stateless
- [ ] Object Storage CLI: **AWS CLI**
- [ ] Archive Storage CLI: **OpenStack swift**
- [ ] NCP CLI는 **Java 기반**, JRE 필요, REST API 방식 동작

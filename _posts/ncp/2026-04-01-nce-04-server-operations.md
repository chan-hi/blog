## 🔑 핵심 개념 1: 서버 Operation — 상태별 가능 여부 (⭐⭐ 최빈출!)

| 작업 | 필요 서버 상태 | 핵심 포인트 |
|------|--------------|------------|
| **서버 스펙 변경** | **정지** 필수 | 운영 중에는 절대 불가 |
| **내 서버 이미지 생성** | **운영 중 / 정지 모두 가능** | ← 이게 함정 포인트! |
| **스토리지 용량 변경** | **정지 또는 연결 해제** | 증설 후 OS 내에서 추가 작업 필요 |
| **인증키 변경** | **정지** 필수 | 결과는 관리자 이메일로 전송됨 |

> ⚠️ **시험 포인트 #1**  
> **내 서버 이미지는 운영 중에도 생성 가능!**  
> 보통 "서버 켜져 있으면 스냅샷 못 찍는다"고 생각하기 쉬운데, NCP는 **운영 중에도 가능**.

```
[민준의 시험 함정]
Q: "서버 이미지를 생성하려면 서버를 정지해야 한다. (O/X)"
A: X — 운영 중에도 가능!
```

## 🔑 핵심 개념 2: 내 서버 이미지 (Server Image)

**내 서버 이미지** = 현재 서버의 OS + 설치된 앱 + 추가 스토리지 상태를 캡처한 것.  
나중에 이 이미지로 **똑같은 서버를 찍어낼 수** 있다.

| 항목 | 내용 |
|------|------|
| **생성 가능 상태** | 운영 중 / 정지 **모두 가능** |
| **포함 내용** | OS 영역 + 추가 스토리지 모두 포함 |
| **리전 간 복제** | **OS 영역만** 이미지로 전송 |
| **데이터 디스크** | SFTP 등으로 **별도** 이전 필요 |
| **계정 공유** | 다른 NCP 계정과 이미지 공유 가능 |

> ⚠️ **시험 포인트 #2**  
> 리전 간 이미지 복제 시 → **OS 영역만** 이미지로 전송!  
> 데이터는 SFTP 등으로 따로 옮겨야 한다.

---

### 유사 서버 생성

**유사 서버 생성** = 선택한 서버와 동일한 스펙, 이미지, 스토리지, ACG를 **그대로 복사**해서 새 서버 생성.  
→ 동일한 설정의 서버를 빠르게 여러 대 만들 때 유용.

## 🔑 핵심 개념 3: 스토리지 Operation

### 스토리지 용량 규칙

| 항목 | **XEN (G1/G2)** | **KVM (G3)** |
|------|----------------|--------------|
| 최대 추가 스토리지 수 | **15개** | **20개** |
| 최대 스토리지 크기 | **2TB** | **16TB** |
| 스토리지 타입 | SSD, HDD | CB1/FB1/CB2/FB2 |

> KVM이 XEN보다 스토리지 측면에서 훨씬 유리하다.

### 용량 변경 규칙

| 방향 | 가능 여부 |
|------|----------|
| **증설 (늘리기)** | ✅ 가능 |
| **축소 (줄이기)** | ❌ **불가** |

> ⚠️ **중요**: 용량 증설 후 **OS 내부에서 추가 작업 필요!**  
> 물리 디스크가 커졌다고 자동으로 OS에 반영되지 않는다.

---

### 신규 스토리지 추가 후 Linux 설정 순서 (⭐ 빈출!)

```bash
1. lsblk          # 물리 디스크 인식 확인
2. parted         # 파티셔닝
3. mkfs           # 파일시스템 포맷 (예: mkfs.ext4 /dev/sdb1)
4. mount          # 마운트 (예: mount /dev/sdb1 /data)
```

> 💡 **순서 암기**: "lsblk → parted → mkfs → mount"  
> "**인식 → 파티션 → 포맷 → 마운트**"

## 🔑 핵심 개념 4: 스냅샷 (Snapshot)

"내 서버 이미지"와 "스냅샷"의 차이를 반드시 구분해야 한다!

| 항목 | **내 서버 이미지** | **스냅샷** |
|------|-----------------|-----------|
| **생성 단위** | **서버** 단위 | **스토리지(디스크)** 단위 |
| **복원 형태** | 서버로 바로 복원 | **추가 스토리지로만** 변환 가능 |
| **백업 방식** | - | 증분(Incremental) 지원 → 용량 효율적 |

> ⚠️ **시험 포인트 #3**  
> 스냅샷으로는 **서버를 바로 만들 수 없다!**  
> 오직 **추가 스토리지**로만 변환 가능.

---

### 스냅샷 장애 복구 시나리오

```
[OS 영역 손상으로 부팅 불가]
       ↓
OS 스토리지의 스냅샷 생성
       ↓
정상 서버에 "추가 스토리지"로 연결
       ↓
데이터 추출 후 복구
```

> 이 시나리오: 스냅샷 → 추가 스토리지 → 다른 서버에 붙이기 → 데이터 구출

## 🔑 핵심 개념 5: 네트워크 인터페이스 (NIC) 제한

서버에 네트워크 카드(NIC)를 여러 개 달 수 있다.  
하지만 개수 제한이 있다.

| 항목 | 제한 |
|------|------|
| **NIC 최대 수** | 서버 1대당 **최대 3개** |
| **NIC당 Secondary IP** | 최대 **5개** |
| **NIC당 총 IP** | Primary 1개 + Secondary 5개 = **총 6개** |

```
[IP 계산]
NIC 1개 → IP 6개 (1 + 5)
NIC 3개 (최대) → IP 최대 18개 (6 × 3)
```

### VIP (고가용성) 구성

**HA(High Availability)** 구성 시 VIP 사용:

```
Secondary IP + Pacemaker + corosync
→ 하나의 서버가 죽으면 VIP가 다른 서버로 자동 이동
```

## 🔑 핵심 개념 6: 서버 부가 운영

| 항목 | 내용 | 주의사항 |
|------|------|---------|
| **서버 이름 변경** | 콘솔 표시 이름만 바뀜 | **호스트명(Hostname)은 변경되지 않음** |
| **반납 보호 설정** | 실수로 서버 삭제 방지 | 보호 설정 시 삭제 요청 불가 |
| **물리 배치 그룹** | 서버를 다른 클러스터에 분산 | **Best-Effort 방식** (100% 보장 ❌) |
| **모니터링** | 상세 모니터링 / 네트워크 모니터링 | 더 디테일한 지표 확인 가능 |

> ⚠️ **시험 포인트 #4**  
> 서버 이름을 바꿔도 **OS 내부의 호스트명은 그대로!**  
> 실제 호스트명을 바꾸려면 OS 내부에서 직접 변경해야 한다.

## 🔑 핵심 개념 7: CLI & API 제어 도구 (⭐ 함정 포인트!)

| 서비스 | 사용 CLI/API | 이유 |
|--------|------------|------|
| **일반 서버** | **ncloud CLI** (`ncloud <command>`) | NCP 전용 |
| **Object Storage** | **AWS CLI / AWS API** | **S3 호환**이기 때문 |
| **Archive Storage** | **OpenStack keystone/swift CLI** | OpenStack 기반 |

```bash
# 서버 목록 조회
ncloud vserver getServerInstanceList

# Object Storage 파일 업로드 (AWS CLI)
aws s3 cp ./photo.jpg s3://my-bucket/ --endpoint-url=https://kr.object.ncloudstorage.com

# Archive Storage 파일 업로드 (swift CLI)
swift upload my-container ./archive.tar.gz
```

> ⚠️ **시험 포인트 #5 (함정!)**  
> "Object Storage를 CLI로 제어할 때 사용하는 도구는?"  
> 답: **AWS CLI** (NCP 자체 CLI가 아님! S3 호환이기 때문)

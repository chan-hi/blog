## 이야기: K8s를 직접 구축하다가 포기한 민준

민준은 직접 쿠버네티스를 설치해보려 했다.

```
쿠버네티스 직접 구축 시도:

1. Master Node 서버 3대 구성 (HA를 위해)
2. etcd 클러스터 설정
3. API Server, Scheduler, Controller-Manager 설치
4. Worker Node 설정
5. 네트워크 플러그인(CNI) 설정

```

실제로 Legacy Kubernetes 직접 구축 시에는 이런 단계들이 필요하다:

```
  - Subnet IP range 설정
  - Cluster CNI 선택/설정
  - Helm (k8s package managing tool) 설치

4단계 - 모니터링 도구 설치 (사용자 직접):

5단계 - 애플리케이션 배포 (사용자 직접):
  - Deployment 생성
  - Service expose

  - PV 생성 및 할당
  - Service Health check 설정/모니터링

```

그때 NCP 담당자가 전화했다.

## 🔑 핵심 개념 1: NKS의 핵심 이점 (⭐⭐ 최빈출!)

**NKS** = NCP가 제공하는 **완전 관리형(Managed) 쿠버네티스** 서비스.  
빠르고 간편하게 컨테이너를 배포하고 실행할 수 있는 관리형 서비스로,  
Kubernetes의 컨테이너 디플로이 자동화, 스케일링, 컨테이너화된 애플리케이션의 관리를 위한 오픈소스 Kubernetes를 이용한다.

### 가장 큰 이점

> **Master Node(Control Plane)를 NCP가 직접 관리!**

| 항목 | Legacy (직접 구축) | **NKS** |
|------|-----------------|---------|
| Master Node 설치 | 직접 (Kubespray/Ansible) | **NCP가 자동으로** |
| Master Node HA | 직접 구성 | **NCP가 보장 (3중화)** |
| K8s 버전 업그레이드 | 직접 | **NCP가 지원** |
| Worker Node 추가 | 동일 프로세스 반복 | **콘솔에서 클릭 몇 번** |
| 클러스터 생성 | 수십 분~수 시간 | **수 분 이내** |
| 서버 스펙 | 직접 준비 | **Standard, CPU Intensive 등 다양한 클러스터 스펙 제공** |
| 스토리지 | 추가 스토리지 구성 필요 | **PersistentVolumes (NAS, Block Storage) 바로 사용 가능** |
| 설치/설정 | 단계별 수동 작업 | **클러스터 생성/설정/설치를 한 번에 제공** |

---

### NKS 아키텍처: Control Plane 구조

NKS의 Master Node는 **Hidden 구성**으로 3중화되어 있어 고가용성을 보장한다. 관리자가 별도로 관리할 필요가 없다.

```
[NKS Master Node 구조 (사용자에게는 보이지 않음)]

Master-0          Master-1          Master-2
  etcd              etcd              etcd       ← HA 구성 (3중화)
  api-server        api-server        api-server
  scheduler         scheduler         scheduler
  controller-mgr    controller-mgr    controller-mgr

        ↓ NCP가 자동 관리, 사용자는 신경 X

Worker Node-0   Worker Node-1   Worker Node-2   Worker Node-3
  Docker          Docker          Docker          Docker
  kubelet         kubelet         kubelet         kubelet
  Kube-Proxy      Kube-Proxy      Kube-Proxy      Kube-Proxy

다양한 스펙의 워커 노드 구성 가능
라이브러리 자동 설치 제공
```

#### NKS Auto-Install Addon Services

NKS는 Worker Node 설정 및 설치와 함께 다음 Addon을 자동으로 설치한다:
- **CSI for Block Storage** — PersistentVolume 연동
- **Weave Scope** — 클러스터 시각화
- **Ingress Nginx** — Ingress Controller
- **Cluster AutoScaler** — 노드 자동 확장

---

### NKS 주요 기능 (K8s 핵심 기능 복습)

| 기능 | 설명 |
|------|------|
| **Self Healing** | 죽은 컨테이너 자동 재시작 |
| **Automatic Binpacking** | 노드 리소스 상황에 맞게 최적 배치 |
| **Horizontal Scaling** | CPU 등 지표에 따라 컨테이너 수 자동 확장 |
| **Service Discovery & LB** | Pod에 IP/DNS 자동 할당 및 트래픽 분산 |

## 🔑 핵심 개념 2: 책임 공유 모델 (⭐ 시험 포인트!)

NKS를 쓰면 NCP와 사용자가 각각 어디까지 책임지는지 명확히 구분된다.

```
[NKS 책임 공유 모델]

┌─────────────────────────────────────────────────────┐
│ NCP가 관리하는 영역                                   │
│  ✅ 마스터/워커 노드 인프라 설정                       │
│  ✅ K8s 설치 도구 설정                                │
│  ✅ K8s 클러스터 배포 (Deploy)                        │
└─────────────────────────────────────────────────────┘
         ↓ 이 위에 사용자가 올림
┌─────────────────────────────────────────────────────┐
│ 사용자가 관리하는 영역                                 │

```

> "NKS를 사용하면 사용자가 직접 관리하지 않아도 되는 것은?"  
> → **Master Node** (Control Plane)

## 🔑 핵심 개념 3: 쿠버네티스 핵심 오브젝트

NKS 시험에는 쿠버네티스 핵심 오브젝트 개념도 함께 출제된다.

### Pod (파드)

파드는 **쿠버네티스 애플리케이션의 기본 실행 단위**이자, 쿠버네티스 객체 모델 중 만들고 배포할 수 있는 **가장 작고 간단한 단위**이다.

```
[계층 구조]
Cluster > Node > Pod > Container
```

파드에는 두 가지 사용 패턴이 있다:

1. **단일 컨테이너 파드** — "단일 컨테이너 당 한 개의 파드" 모델. 가장 흔한 사용 사례. 한 개의 파드가 단일 컨테이너를 감싸고 있으며, 쿠버네티스는 컨테이너가 아닌 파드를 직접 관리한다.

2. **다중 컨테이너 파드** — 강하게 결합되어 있고 리소스 공유가 필요한 여러 컨테이너가 하나의 파드에 함께 배치된다. 예를 들어 메인 컨테이너가 공유 볼륨에 파일을 쓰고, 별도의 "사이드카" 컨테이너가 그 파일들을 업데이트하거나 복구하는 구조.

> 각 파드는 자체 IP, 호스트이름, 프로세스 등이 있는 별도의 논리적인 시스템이다.

---

### Controller: ReplicaSet, Deployment, DaemonSet

쿠버네티스에서 파드를 관리하는 세 가지 주요 컨트롤러:

#### ReplicaSet

- 파드의 복제본(레플리카)을 **지정한 수만큼 유지**하는 기능
- 같은 유형의 Pod만 포함하는 그룹
- **가용성(Availability) 보장**을 위해 사용
- ⚠️ Pod 업데이트 시 롤링 업데이트를 **지원하지 않음**

#### Deployment

- **ReplicaSet을 그룹화하고 관리**하는 상위 개념
- 배포에 대한 **이력 관리** → 새 버전에 문제가 생기면 **이전 버전으로 롤백** 가능
- **Deployment 배포 전략**: RollingUpdate, Blue/Green, Canary 등
- 애플리케이션을 **다운타임 없이 업데이트** 가능
- **라벨 셀렉터, 원하는 복제본 수, 포드 템플릿**으로 구성

```
[Deployment 동작 원리]

1. K8s 관리자가 매니페스트 파일을 API 서버에 제출
2. API 서버가 스케줄러에 알림 → 스케줄러가 파드 요구사항을 충족하는 노드 탐색
3~5. kubelet이 컨테이너 생성 후 상태 정보를 API 서버로 전달,
     이상 발생 시 API 서버에 알림
```

```
[Deployment 버전 관리]

Deployment
  ├── ReplicaSet v1 → [Pod][Pod]
  ├── ReplicaSet v2 → [Pod][Pod]
  └── ReplicaSet v3 → [Pod][Pod]  ← 현재 버전
         ↑ 문제 발생 시 v1/v2로 즉시 롤백
```

#### DaemonSet

- K8s 클러스터 내 **모든 노드에서 Pod의 복제본을 실행**하도록 함
- 클러스터에 노드가 추가되면 Pod도 추가되고, 노드가 삭제되면 해당 Pod도 삭제됨
- `nodeSelector`, `podAffinity` 같은 조건을 명시하면 **조건에 일치하는 노드에만** Pod를 생성

```bash
# 노드에 라벨 설정
kubectl label nodes $nodename type=prod
```

## 🔑 핵심 개념 4: Service (서비스)

**Service** = 클러스터 내 Pod들에게 접근하기 위한 방법.

- 여러 Pod를 묶어 **Healthy한 Pod로 트래픽을 라우팅**하는 로드밸런싱 기능 제공
- 서비스 이름은 클러스터 내 고유한 **DNS로 동작** → 컴포넌트 간 Service Discovery에 활용
- Pod는 랜덤 IP를 부여받지만 언제든 없어지고 생성될 수 있으므로 **직접 호출은 지양**

> 포드는 일시적으로 생성한 컨테이너의 집합으로, IP 주소의 지속적인 변동 문제를 해결하기 위해 Service가 등장했다.

### Service Type

| 타입 | 설명 | 사용 시나리오 |
|------|------|-------------|
| **ClusterIP** | 클러스터 내부에서만 사용 (기본 옵션) | 내부 마이크로서비스 통신 |
| **NodePort** | 노드 자체의 이름을 사용하여 포트 리다이렉션. 보통 **30000~32768 포트** 범위로 Pod에 트래픽 포트포워딩 | 가상머신에 할당된 공인/사설 IP로 서비스 접근 |
| **LoadBalancer** | ClusterIP + NodePort + CSP의 External LB. External LB가 Service에서 지정된 포트로 트래픽을 받아 NodePort로 할당된 포트를 통해 프록시. **장기간 유지되는 엔드포인트 및 워커노드 간 로드밸런싱** 필요 시 사용 | 외부에 서비스 노출 |

> **NCP 연동 포인트**: NKS에서 Service 타입을 **LoadBalancer**로 지정하면 **NCP 로드밸런서 인스턴스를 자동으로 생성**한다.

## 🔑 핵심 개념 5: Ingress 심화 (NCP 환경)

**Ingress** = 외부에서 오는 트래픽을 클러스터 내부의 Service로 라우팅하기 위한 **규칙 집합**.

- **Host 기반 및 URL Path 기반 라우팅** 규칙 사용
- 외부 트래픽 수신을 위해 Ingress에 대한 Service 타입으로 반드시 **NodePort 혹은 LoadBalancer 지정** 필요
- Ingress는 URL, 로드밸런스 트래픽, SSL/TLS 종료, 이름 기반의 **가상 호스팅**을 제공하도록 구성 가능

### 앱 노출 컴포넌트 흐름

```
[외부 → 내부 접근 흐름]

클라이언트
    │
    ▼
인그레스-매니지드 로드밸런서
    │
    ▼ (인그레스 라우팅 규칙 적용)
  Ingress
    │
    ├──→ /path-A → Service A → [Pod][Pod]
    └──→ /path-B → Service B → [Pod][Pod]
```

- **Service**: 클러스터 안 혹은 밖으로 Pod를 노출. DNS 등으로 Service Discovery 제공
- **Ingress**: 외부에서 내부 Service로 라우팅. HTTP 리퀘스트의 Host 혹은 Path 기반

### NCP Ingress의 핵심: VIP 하나로 모든 도메인 처리

NCP는 로드밸런서 및 공인 IP 사용을 **최소화**하기 위해, 하나의 VIP에 모든 도메인을 할당한다.

```
[공인 IP 비용 절감]
전통 방식:
  frontend.bakery.com → 공인 IP 1개
  api.bakery.com      → 공인 IP 1개
  admin.bakery.com    → 공인 IP 1개
  총 공인 IP: 3개 (= 비용 3배)

NCP Ingress 방식:
  VIP(공인 IP) 1개에 여러 도메인 매핑
  frontend.bakery.com ──┐
  api.bakery.com        ├─→ 공인 IP 1개 (= 비용 1개)
  admin.bakery.com      ──┘
```

### Ingress YAML 설정 예시 (NCP 실제 예시)

```yaml
# NCP Container Registry (NCR) Ingress 예시
apiVersion: extensions/v1beta1
kind: Ingress
metadata:
  name: ncr-ingress
  annotations:
    nginx.ingress.kubernetes.io/proxy-read-timeout: '900'
    nginx.ingress.kubernetes.io/configuration-snippet: |
      chunked_transfer_encoding on;
spec:
  rules:
  - host: ncr.ncloud.com
    http:
      paths:
      - path: /ncr/api
        backend:
          serviceName: ServiceA
          servicePort: 80
      - path: /ncr/
        backend:
          serviceName: ServiceB
          servicePort: 80
  - host: "*.kr.ncr.ntruss.com"
    # 와일드카드 호스트 기반 라우팅
```

> **해설**: `ncr.ncloud.com` 호스트명으로 접근 시 `/ncr/api` 경로는 ServiceA의 80포트로, `/ncr/` 경로는 ServiceB의 80포트로 라우팅.

### 일반적인 Ingress 설정 예시

```yaml
# Ingress 설정 예시
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: bakery-ingress
spec:
  rules:
  - host: bakery.com
    http:
      paths:
      - path: /
        backend:
          service:
            name: frontend-service
            port: 80
      - path: /api
        backend:
          service:
            name: api-service
            port: 8080
  - host: admin.bakery.com
    http:
      paths:
      - path: /
        backend:
          service:
            name: admin-service
            port: 3000
```

### On-Premise(Bare Metal) 환경에서의 Ingress

- On-Premise 환경에서는 CSP의 LB와 같은 이용이 어려움
- **L3 DSR을 이용**하는 경우, Service의 `externalIPs` 속성에 L4 Loopback Interface에 할당된 VIP를 명시하여 사용
- L4 Switch는 **Syn Packet을 통한 Health Check** 수행
- Pod가 Healthy인 경우 VIP를 통해 Pod 애플리케이션 접근 가능

## 🔑 핵심 개념 6: NCP 서비스와의 통합 연동

NKS의 강력한 장점 중 하나 = **NCP 다른 서비스와 매끄러운 연동**.

| 연동 서비스 | 용도 |
|-----------|------|
| **Block Storage / NAS** | Persistent Volume(PV)으로 직접 사용 가능 |
| **Container Registry** | 이미지 저장소로 바로 연동 |
| **Load Balancer** | Service Type LoadBalancer 자동 연동 |
| **VPC / Subnet** | 클러스터 네트워크 구성 |
| **Cloud Log Analytics** | 로그 수집 연동 |

```
[NKS + NCP 서비스 통합 예시]

클러스터 생성 시:
  └─ VPC, Subnet 선택 → 자동으로 클러스터 네트워크 구성

Pod 배포 시:
  └─ Container Registry → 이미지 자동 pull
  └─ NAS/Block Storage → PersistentVolumeClaim으로 데이터 저장

외부 서비스 노출 시:
  └─ Service(LoadBalancer) → NCP 로드밸런서 자동 생성
```

## 🔑 핵심 개념 7: 노드 풀(Node Pool) 관리

### 워커 노드 스펙

NKS 노드 정보 화면에서 확인할 수 있는 정보:

| 항목 | 내용 |
|------|------|
| 서버 이름 | `nks-pool-{클러스터ID}-w-{번호}` 형식 |
| 서버 타입 | Standard (예: 2vCPU, 4GB Mem, 50GB Disk) |
| 노드 상태 | **Ready** 또는 **NotReady** (2가지) |
| Docker Version | (예: 18.9.9) |
| Kernel Version | (예: 5.3.13-050313-generic) |
| 비공인 IP | 클러스터 내부 사설 IP |
| 공인 IP | 필요 시 부여 |

> **NotReady 상태**: 워커 노드의 **kubelet이 동작하지 않거나 정상이 아닌 경우** NotReady로 표시된다.

### 워커 노드 수 제한

> 워커 노드는 **최대 250대**까지 생성 가능하다.

## 🔑 핵심 개념 9: NKS vs Legacy K8s 비교

| 항목 | Legacy (직접 구축) | **NKS** |
|------|-----------------|---------|
| Master Node 관리 | 사용자 | **NCP** |
| 클러스터 HA | 사용자 구성 | **NCP 보장 (3중화)** |
| Worker 노드 추가 | 동일 프로세스 반복 | **콘솔 클릭** |
| 버전 업그레이드 | 수동 | **NCP 지원** |
| 클러스터 스펙 | 직접 구성 | **Standard, CPU Intensive 등 다양한 스펙** |
| Block Storage 연동 | 별도 설정 | **자동 (CSI for Block Storage)** |
| Container Registry 연동 | 별도 설정 | **자동** |
| Ingress | 별도 설치 | **Ingress Nginx 자동 설치** |
| 클러스터 자동 확장 | 별도 구성 | **Cluster AutoScaler 자동 설치** |
| 최대 워커 노드 | 제한 없음 (관리는 사용자) | **최대 250대** |

## 클라우드빵집 최종 아키텍처

```
[클라우드빵집 완성 인프라]

인터넷
  │
  ▼
[NACL] ← Subnet 방화벽 (Stateless)
  │
  ▼
[NCP Load Balancer] ← NKS Service(LoadBalancer) 자동 연동
  │
  ▼
[Ingress] ← URL 기반 라우팅 (VIP 1개로 모든 도메인 처리)
  ├──→ /      → frontend-service → [Pod][Pod][Pod]
  ├──→ /api   → api-service      → [Pod][Pod][Pod]
  └──→ /admin → admin-service    → [Pod]

각 Pod에 ACG 적용 (Stateful)

데이터 저장:
  └─ DB Pod → NAS PersistentVolume (여러 Pod 공유)
  └─ 이미지 → Object Storage (AWS CLI)
  └─ 로그 → Archive Storage (OpenStack swift)
  └─ 백업 → Backup Service (증분 백업 매일)

이미지 관리:
  └─ Container Registry (Object Storage 기반, CVE 분석)

NKS Auto-Install Addons:
  └─ CSI for Block Storage, Weave Scope, Ingress Nginx, Cluster AutoScaler
```

## 📝 시험 대비 체크리스트

- [ ] NKS = **완전 관리형** K8s 서비스 (빠르고 간편한 컨테이너 오케스트레이션)
- [ ] NKS 최대 이점: **Master Node를 NCP가 관리** (3중화 HA, Hidden 구성)
- [ ] 사용자가 해야 할 것: **앱 배포, 모니터링 도구 설치**
- [ ] 클러스터 스펙: **Standard, CPU Intensive** 등 다양한 스펙 제공
- [ ] 워커 노드 최대 **250대**
- [ ] 노드 상태: **Ready** / **NotReady** (kubelet 비정상 시 NotReady)
- [ ] Pod = K8s 최소 실행 단위, 1개 이상의 컨테이너로 구성
- [ ] ReplicaSet = 지정 수만큼 Pod 복제 유지, 롤링 업데이트 **미지원**
- [ ] Deployment = ReplicaSet 상위 개념, 롤백/RollingUpdate/Blue-Green/Canary 지원
- [ ] DaemonSet = **모든 노드**에 Pod 1개씩 실행 보장
- [ ] Service Type: ClusterIP (내부) / NodePort (30000~32768) / LoadBalancer (NCP LB 자동 생성)
- [ ] NKS + NAS/Block Storage → **Persistent Volume**으로 연동
- [ ] NKS + Container Registry → 이미지 자동 pull
- [ ] NKS + Service(LB) → NCP Load Balancer 자동 생성
- [ ] Ingress: 하나의 VIP로 여러 도메인 처리 (비용 절감)
- [ ] Ingress 뒤 Service: 반드시 **NodePort 또는 LoadBalancer**
- [ ] NKS 사용 3단계: **프로비저닝 → 클러스터 연결(kubeconfig) → 앱 배포**

## 핵심 요약

- RAG, Retrieval-Augmented Generation은 검색 증강 생성입니다.
- LLM이 학습된 지식에만 의존하지 않고 외부 데이터 소스를 검색하여 답변을 생성하게 만드는 구조입니다.
- 주요 요소는 문서 로드, 분할, 임베딩, 벡터 DB, 검색, 프롬프트, LLM, 출력입니다.
- RAG는 최신 정보 반영, 출처 제시, Hallucination 감소, 도메인 특화 답변에 유리합니다.
- Advanced RAG는 검색 전/검색/검색 후 단계에서 검색 품질을 높이는 방식입니다.

## RAG란?

RAG는 `Retrieval-Augmented Generation`의 약자이며, 한국어로는 `검색 증강 생성`입니다.

| 요소 | 의미 |
|---|---|
| Retrieval | 요청된 정보를 가져오는 검색 단계 |
| Augmentation | 원래 정보에 검색된 정보를 덧붙여 보강하는 단계 |
| Generation | 보강된 문맥을 바탕으로 응답을 텍스트로 생성하는 단계 |

RAG는 LLM의 한계를 보완하기 위해 고안된 자연어 처리 기술입니다. 인덱싱, 검색, 생성을 결합해 사용자가 원하는 답변을 만들 때 학습된 데이터에만 의존하지 않고 외부 데이터 소스를 활용합니다.

기본 작동 방식:

1. 사용자가 질문을 입력합니다.
2. RAG는 외부 데이터베이스, 예를 들어 웹 문서나 기업 내부 문서에서 질문과 관련된 정보를 검색합니다.
3. 검색된 정보를 기반으로 LLM이 답변을 생성합니다.

## RAG 도입 이유

RAG를 도입하는 이유는 다음과 같습니다.

- DB 최신 정보 반영
- Context 기반의 정확도와 연관성 높은 답변 품질 확보
- Hallucination 예방
- 개인 또는 기업 DB 검색과 활용
- 유사 Context 케이스를 대량으로 검색
- 복합 조건의 DB 검색
- 변동된 정보에 맞게 DB 비교 및 업데이트가 필요한 영역 지원

## 도메인별 활용 예시

| 도메인 | 활용 예시 |
|---|---|
| 고객 지원 | 고객센터, 고객 대응 답변 품질 제고, 복잡한 쿼리 대응 |
| 콘텐츠 제작 | 정확하고 최신인 데이터를 기반으로 콘텐츠 작업, 리서치 |
| 교육 | 커스텀 자료 제작, 최신 정보 기반 교육 자료 제작 |
| 의료 | 최신 연구, 희귀 케이스, 최신 의료 시술/수술 방법 검색, 환자용 교육 자료 |
| 법률 | 유사 케이스 검색, 법 개정에 따른 법 위반 케이스 단속 |

## RAG의 장점

| 장점 | 설명 |
|---|---|
| Fine tuning보다 시간과 비용이 적음 | 외부 DB를 활용하므로 별도 학습 데이터를 준비하지 않아도 됨 |
| 모델의 일반성 유지 | 특정 도메인에만 묶이지 않고 필요에 따라 다양한 분야 질문에 대응 |
| 답변 근거 제시 | 답변과 함께 정보 출처를 제공해 신뢰도 향상 |
| Hallucination 감소 | 외부 데이터 기반으로 답변해 모델 자체 편향이나 오류를 줄임 |

## RAG 구조

RAG는 다음 요소로 구성됩니다.

| 순서 | 요소 | 주요 내용 |
|---:|---|---|
| 1 | 문서 로드, Load | PDF, Word, Raw Data, 웹페이지 등 데이터 읽기 |
| 2 | 분할, Split | 불러온 문서를 chunk 단위로 분할 |
| 3 | 임베딩, Embedding | 문서를 벡터 표현으로 변환 |
| 4 | 벡터 DB, VectorStore | 변환된 벡터를 DB에 저장 |
| 5 | 검색, Retrieval | 유사도 검색. similarity, Multi-Query, Multi-Retriever 등 |
| 6 | 프롬프트, Prompt | 검색 결과를 바탕으로 원하는 결과를 도출하기 위한 지시 구성 |
| 7 | 모델, LLM | 언어 모델 선택 |
| 8 | 결과, Output | 텍스트, JSON, 마크다운 등으로 출력 |

## 일반 AI 챗봇 vs RAG 챗봇

| 구분 | 일반 AI 챗봇 | RAG 활용 AI 챗봇 |
|---|---|---|
| 흐름 | 사용자 질문 -> AI 모델 답변 | 사용자 질문 -> 관련 문서 검색 -> 관련 문서 포함 질문 -> AI 모델 답변 |
| 데이터 활용 | 모델 내부 지식 중심 | 외부 데이터 소스 활용 |
| 장점 | 구조가 단순함 | 최신 정보, 내부 문서, 출처 기반 답변 가능 |
| 한계 | 최신/전문/내부 정보 반영 어려움 | 문서 품질, 검색 품질, 파이프라인 구성 중요 |

## 파인튜닝 vs RAG

| 구분 | 파인튜닝 | RAG |
|---|---|---|
| 비유 | 언어 모델이 특정 지식을 학습하고 암기 | 도서관 사서와 협업해 관련 책을 찾고 참고해 답변 |
| 접근 방식 | 특정 업무에 특화된 LLM | 데이터베이스의 Context로 기존 LLM 보강 |
| 도메인 적용성 | 다른 도메인에 맞게 조정 가능 | 도메인에 구애받지 않고 필요에 따라 모델 전환 가능 |
| 환각 감소 | 세심한 튜닝 없이는 환각이 더 발생할 수 있음 | 환각을 효과적으로 감소 |
| 활용 사례 | 의료 문서 분석 등 전문 도메인 작업 | 고객 문의 대응, 개인화된 학습 콘텐츠 |

파인튜닝은 `Pre-trained based Model`, 즉 모델 중심 접근입니다. RAG는 `data driven`, 즉 데이터 중심 접근입니다.

## HTML/PDF 문서 기반 RAG 아키텍처

RAG는 다양한 Raw Data를 처리할 수 있습니다.

- PDF
- HTML
- CSV
- DOCX
- TXT
- JSON

흐름:

1. Raw Data
2. Connecting
3. Embedding
4. Vector DB
5. Retrieval
6. LLM + Prompt
7. Response

### Raw Data -> Connecting

데이터 소스를 가져오는 과정입니다. HTML이나 PDF 등 원본 데이터 형식을 컴퓨터가 처리 가능한 형태로 변환해야 합니다. PDF의 경우 OCR이 필요할 수 있습니다.

### Connecting -> Embedding

대용량 데이터를 모델이 한 번에 처리하기 어렵기 때문에, 모델이 처리 가능한 작은 단위로 나눕니다. 이 과정을 Chunking이라고 합니다. CLOVA Studio 익스플로러의 문단 나누기 API가 이 기능을 지원합니다.

### Chunking과 임베딩 파라미터 주의

- 분할된 Chunk 텍스트가 임베딩 API 1회 처리 Input 한도인 500토큰을 넘지 않아야 함
- 임베딩 v2는 8192토큰
- Vector DB Indexing과 Search 구현 시 임베딩 모델별 파라미터를 일치시켜야 함

모델별 거리 지표:

- clir-emb-dolphin: IP
- clir-sts-dolphin: Cosine

## LangChain

LangChain은 대규모 언어 모델 기반 애플리케이션을 구축하기 위한 오픈소스 프레임워크이자 SDK입니다. LLM과 애플리케이션의 통합을 간소화합니다.

## LangChain 구성 요소

| 구성 요소 | 설명 |
|---|---|
| Prompt Templates | 사용자 input을 언어 모델에 전달하는 포맷 정의 |
| Chat Models | 메시지를 입력으로 사용하고 메시지를 출력으로 반환하는 언어 모델 |
| Output Parsers | LLM 결과물을 구조화된 포맷으로 파싱 |
| Document Loaders | 웹 페이지, PDF, DB 등 다양한 소스의 문서를 로딩하고 처리 |
| Text Splitters | 문서 내 텍스트를 작은 Chunk로 구성 |
| Vector Stores | 임베딩 값을 효율적으로 저장하고 검색하는 DB |
| Tools | Agent, Chain, LLM이 외부 세계와 상호작용하기 위한 인터페이스 |

## Document Loaders

Document Loader는 외부 다양한 소스로부터 문서 데이터를 가져옵니다.

### URL Document Loaders

대표:

- WebBaseLoader
- UnstructuredURLLoader

### PDF Document Loaders

대표:

- PyPDFLoader: PDF 문서를 페이지별로 로드
- UnstructuredPDFLoader: 형식이 없는 PDF 문서 로드
- PyMuPDFLoader: PDF 문서 메타데이터를 상세 추출
- OnlinePDFLoader: 온라인 PDF 문서 로드
- PyPDFDirectoryLoader: 특정 폴더의 모든 PDF 문서 로드

### Word, TXT, CSV, Directory Loaders

| Loader | 설명 |
|---|---|
| Docx2txtLoader | docx 형식 로드 |
| TextLoader | txt 형식 로드 |
| CSVLoader | CSV 파일의 각 행을 서로 다른 Document 객체로 변환 |
| DirectoryLoader | 디렉토리 내 모든 문서 로드 |

## Text Splitters

### CharacterTextSplitter vs RecursiveCharacterTextSplitter

| 구분 | CharacterTextSplitter | RecursiveCharacterTextSplitter |
|---|---|---|
| 작동 방식 | Separator 기반 분할 | 재귀적 분할 |
| 장점 | 속도, 일관성 | 문맥 유지 |
| 단점 | 문맥 분리 가능 | 속도, chunk_size 의존 |
| 활용 사례 | 속도와 일관성이 중요한 경우 | 문맥 유지가 중요한 경우 |

### Text Splitter 매개변수

| 매개변수 | 설명 |
|---|---|
| separator | 분할 기준 문자열 |
| chunk_size | 각 청크의 최대 길이 |
| chunk_overlap | 인접 청크 사이에 중복으로 포함될 문자 수 |
| length_function | 청크 길이를 계산하는 함수. 예: len |
| is_separator_regex | separator를 정규식으로 사용할지 여부 |

## Advanced RAG

Advanced RAG는 기존 RAG 구성 방식에서 검색 품질을 높이기 위해 검색 결과를 최적화한 것입니다.

### Pre-Retrieval

Pre-Retrieval은 실제 문서를 검색하기 전에 질문을 최적화하거나 문서 저장을 준비하는 모든 과정입니다.

| 구성 요소 | 설명 |
|---|---|
| Document Chunking | 문서를 의미 단위로 분할 |
| Embedding | 문서 또는 질문을 벡터로 변환 |
| Metadata Enrichment | 문서에 날짜, 출처, 태그 등 부가 정보 추가 |
| Indexing | 벡터 DB에 저장해 검색 준비 |
| Query Preprocessing | 사용자 질문 정제, 변형, 확장 |
| Intent Routing | 질문 의도 분류 및 라우팅 |

### Retrieval

| 구성 요소 | 설명 |
|---|---|
| Query Embedding | 사용자 입력을 벡터로 변환 |
| Vector Similarity Search | 쿼리 벡터와 문서 벡터를 비교해 가장 유사한 문서 chunk를 Top-K로 검색 |
| Hybrid Retrieval | 벡터 검색과 전통적 키워드 검색, BM25를 조합해 정확도 향상 |

### Post-Retrieval

Post-Retrieval은 Retriever가 가져온 문서들을 LLM에 전달하기 전에 정제하고 정렬하는 고급 처리 과정입니다.

| 구성 요소 | 설명 |
|---|---|
| Reranking | 검색된 문서들을 의미적 중요도 기준으로 다시 정렬 |
| Filtering & Scoping | 쿼리나 메타데이터 기반으로 검색 결과 필터링 |

## 시험 암기 포인트

- RAG는 검색, 증강, 생성의 결합입니다.
- RAG는 최신 정보, 내부 문서, 출처 제시, 환각 감소에 강합니다.
- 파인튜닝은 모델 중심, RAG는 데이터 중심 접근입니다.
- RAG 기본 순서: Load -> Split -> Embedding -> Vector DB -> Retrieval -> Prompt -> LLM -> Output.
- Chunk 크기와 임베딩 토큰 한도를 맞춰야 합니다.
- LangChain 구성 요소: Prompt Template, Chat Model, Output Parser, Document Loader, Text Splitter, Vector Store, Tool.
- Advanced RAG는 Pre-Retrieval, Retrieval, Post-Retrieval을 구분합니다.
- Hybrid Retrieval은 벡터 검색과 BM25 같은 키워드 검색을 결합합니다.
